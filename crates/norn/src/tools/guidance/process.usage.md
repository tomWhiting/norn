Use this tool to work alongside long-running background processes — a dev server, a long test run, a watched build.

A process id (e.g. "p1") is returned when you start a command with bash `run_in_background: true`, or when a foreground bash command exceeds its timeout and is automatically moved to the background (the result carries `migrated: true` and the id).

- `output` (id): returns only the output produced since your last `output` call for that process, plus its current status. Call it repeatedly to follow a process incrementally. If there is no new output it says so — that is not an error. A very large new region is written to the process's spool file instead of being inlined; read or grep that file.
- `status` (id): the process's status (running / exited / killed), pid, start time, exit code when it has exited, and its spool path and size.
- `kill` (id): stop the process (and its process group, so a `server &` grandchild dies too). Killing an already-finished process just reports its final status — it is safe to call.
- `list`: every background process this session owns, with ids, commands, statuses, and each process's active watches (watch id + brief).
- `watch` (id, brief, filter): attach a deterministic watch to a running process. `filter` is a shell script run via `sh -c` over each new region of the process's output on stdin; exit 0 means match and its stdout becomes the excerpt that wakes you, any other exit means no match. Exits 126 and 127 are reserved by the shell (command not executable, and command not found) and are reported as watch errors, not no-matches. `brief` is your human-readable note of what you are watching for (e.g. "a compile error"). The filter runs with no model cost on every new chunk of output — a match wakes you with just the matching excerpt instead of the whole stream. There is no cap: attach as many watches to a process as you like. A match on the process's final output is delivered before its completion notice. Because that final region must be filtered first, a watched process's completion notice waits for its output pipes to close — if a backgrounded grandchild keeps them open, kill the process to force completion; unwatched processes are unaffected.
- `unwatch` (watch_id): detach a watch (e.g. `w1`). Any in-flight filter run finishes; nothing after that is examined.

Watches are how you react to a noisy long-running process without polling it: attach `{brief, filter}` and keep working; a match arrives as a message. If a filter cannot run (e.g. it names a binary that does not exist), you get a watch-error message and the watch stays attached — fix the filter or unwatch it; it is never silently disabled. Watches are in-session state and end when their process exits or is killed.

Background processes have no timeout and no turn limit — they live until they exit or you kill them. When one finishes you are notified automatically with its exit status. Kill processes you no longer need so nothing keeps running past its usefulness.
