Use to steer a running sub-agent that you spawned or forked. Address the recipient by hierarchical registry path (e.g. "/workers/analyzer") or UUID. The content field accepts any valid JSON value — use structured objects for machine-readable coordination, strings for human-readable notes.

The signal is delivered through the message router onto the child's inbound channel and drains at the child's next tool boundary, injected as a harness-framed <agent_message> turn whose attribution (sender path, id, kind, sequence) is resolved by the runtime — message content cannot impersonate another sender. trigger_turn=true sends a steer message, injected immediately after the child's current tool batch completes and before its next provider request — use this when the new direction must take effect right away. trigger_turn=false sends an update message, buffered until the child would otherwise stop and injected only then — use this to extend the child's work with additional context rather than redirecting it mid-stride.

Delivery requires a live handle to the recipient, which you hold only for agents you spawned or forked yourself. Signalling a sibling, a peer, or any agent you did not create returns a structured delivery failure ("delivered": false with a reason) — the message is not silently queued anywhere. Signalling an agent that already finished also fails with the recipient's recorded status and completion time — read its delivered result instead. Before retrying a failed delivery, check the recipient's status with the agents tool. To reach such an agent, route the request through its parent. Every accepted send is recorded as an agent_message.sent audit event in your session store, and the recipient records agent_message.delivered plus the framed message itself at injection — the audit trail persists with the session; the in-flight channel does not.
