## 2. Findings

### F-1 — Code defect (fix before final Gate C): the two-phase index-lock gate reports a residual, not the caller's deadline
`crates/norn/src/session/persistence/lock.rs:120-147` — `lock_index` now (correctly) admits through the process-local gate first, then computes `remaining = limit − elapsed` and hands **remaining** to `lock_with_deadline`, whose timeout arm populates the typed error with `waited: deadline` — i.e. the residual (≈49.98ms for a configured 50ms). The total wait *behavior* is right; the *reported* `waited` no longer names the configured deadline, which breaks the typed contract and its integration test (`assert_eq!(*waited, Duration::from_millis(50))` — fails 3/3, deterministic). The process-gate arm reports the original limit; the OS arm reports the residual — the two arms disagree about what `waited` means. Fix shape: thread the caller's original limit into `lock_with_deadline` for error construction (the poll budget stays the residual). One parameter, or construct the error at the `lock_index` level.

### F-2 — Nit: `JsonlSink.needs_newline` is dead state under reopen-heal
`crates/norn/src/session/jsonl_sink.rs:117,226` — `persist` resets the flag to `false` before every write, and the per-event reopen heals any torn tail before the handle is returned, so the flag can never be observed `true`. The shared `write_event_line` helper keeps its meaning for single-descriptor writers (and its unit test); the sink's own field is vestigial. Delete the field or document why it's retained.

### F-3 — Field report (external, needs a repro fixture): first resume after SIGKILL mid-tool-call returns provider 400 despite the open-time repair
Reported by Cally Ray from a live headless norn worker (session `adbcbaa6`, 2026-07-13): a worker SIGKILLed mid-tool-call left a dangling function call; the **first** resume attempt failed with a provider 400 and the **second** succeeded. `open_root_session` runs `repair_dangling_tool_calls` before the first request by design (`agent/session_open.rs:86`), so the observed 400 means some resume path isn't covered. Prime suspect: server-side thread state (`previous_response_id`) — a locally synthesized ToolResult can't heal what the server's conversation state already holds, and the retry presumably rebuilt without the stale anchor. Needs: a repro fixture (kill mid-call → resume → assert zero 400s), and if the server-thread theory holds, this feeds the open **D3** threading decision rather than a local patch.

### F-4 — Observation (no action): per-event reopen cost
The `JsonlSink` reopen-verify-heal-write-close cycle adds a full descriptor-relative traversal + metadata check + 1-byte tail read per persisted event. For session-event cadence this is noise, and it buys the zero-idle-descriptor property the 2026-07-13 ruling demands. Noted so the trade is on the record; if event-rate-sensitive embedders appear later, the D1E weighted-authority work is the right place to revisit, not a cache bolted onto the sink.
