# PW-002: Auth adapter — async auth_headers method on AuthProvider

> **Cluster:** norn-provider-ws
> **Checklist:**
> - C48 — AuthProvider trait gains async auth_headers method returning Vec<(String, String)> of header name-value pairs.
> - C49 — OAuthAuthProvider implements auth_headers via AuthManager::auth().await, returning bearer token and chatgpt-account-id.
> - C50 — ApiKeyAuthProvider implements auth_headers returning bearer token only.
> - C51 — MockAuthProvider implements auth_headers consuming from the same token sequence as apply_auth.

## Purpose

Extend the AuthProvider trait with an async auth_headers method that extracts bearer token and chatgpt-account-id as header name-value pairs. The existing apply_auth method couples auth to reqwest::RequestBuilder; WebSocket handshakes need raw headers for the HTTP upgrade request. This brief provides that surface without touching the WS transport itself.

## Task

Add an async auth_headers() method to the AuthProvider trait. Implement it for OAuthAuthProvider (via AuthManager::auth().await), ApiKeyAuthProvider (bearer token only), and MockAuthProvider (consuming from the shared token sequence). The method returns Vec<(String, String)> of header name-value pairs.

## Requirements

### R1: AuthProvider trait gains auth_headers

The AuthProvider trait SHALL define an async method auth_headers(&self) -> Result<Vec<(String, String)>, ProviderError>. The method SHALL return header name-value pairs suitable for an HTTP upgrade request.

**Acceptance:**
- auth_headers method defined on AuthProvider trait
- Method is async (via #[async_trait])
- Return type is Result<Vec<(String, String)>, ProviderError>

**Files:**
- modify: crates/norn/src/provider/auth.rs

**Checklist:**
- C48 — AuthProvider trait gains async auth_headers method returning Vec<(String, String)> of header name-value pairs.

### R2: OAuthAuthProvider implements auth_headers

WHEN auth_headers is called on OAuthAuthProvider THE SYSTEM SHALL call self.manager.auth().await to retrieve the current token AND SHALL return Authorization: Bearer <token> and chatgpt-account-id headers.

**Acceptance:**
- OAuthAuthProvider::auth_headers returns bearer token header
- chatgpt-account-id header included when available from AuthManager
- Returns ProviderError::AuthenticationFailed when no token is available

**Files:**
- modify: crates/norn/src/provider/auth.rs

**Checklist:**
- C49 — OAuthAuthProvider implements auth_headers via AuthManager::auth().await, returning bearer token and chatgpt-account-id.

### R3: ApiKeyAuthProvider implements auth_headers

WHEN auth_headers is called on ApiKeyAuthProvider THE SYSTEM SHALL return a single Authorization: Bearer <key> header AND SHALL NOT include chatgpt-account-id.

**Acceptance:**
- ApiKeyAuthProvider::auth_headers returns one-element vec with bearer header
- No chatgpt-account-id header in the result

**Files:**
- modify: crates/norn/src/provider/auth.rs

**Checklist:**
- C50 — ApiKeyAuthProvider implements auth_headers returning bearer token only.

### R4: MockAuthProvider implements auth_headers

WHEN auth_headers is called on MockAuthProvider THE SYSTEM SHALL consume the next token from the same sequence used by apply_auth AND SHALL increment the shared apply_count.

**Acceptance:**
- MockAuthProvider::auth_headers returns bearer header from token sequence
- Calling auth_headers consumes the same token that apply_auth would have consumed
- apply_call_count increments on auth_headers calls

**Files:**
- modify: crates/norn/src/provider/auth.rs

**Checklist:**
- C51 — MockAuthProvider implements auth_headers consuming from the same token sequence as apply_auth.

## Boundaries

- SHALL NOT modify the existing apply_auth method signature or behavior
- SHALL NOT modify the on_unauthorized method
- SHALL NOT add WebSocket-specific code — this is a generic auth surface extension
