# PW-008: Connection lifecycle — age tracking, reconnection, and warm floor

> **Cluster:** norn-provider-ws
> **Depends on:** PW-005
> **Checklist:**
> - C34 — Per-slot connection age tracked from establishment time via std::time::Instant.
> - C35 — Proactive reconnection triggered when connection age exceeds reconnect_margin before the 60-minute limit.
> - C36 — websocket_connection_limit_reached error triggers fresh connection and single request retry.
> - C37 — Connection dropped and recreated on any unexpected WS Close frame from server.
> - C38 — Warm floor established via parallel tokio::spawn on first pool access. First caller waits only for its own connection, remaining establish in background.
> **Stories:**
> - S2 (AI Agent, Making Provider Requests) — As an AI agent, I want the WebSocket connection to reconnect seamlessly when the 60-minute server limit is reached so that long sessions complete without transport errors.
> - S7 (Workflow Engine, Running Agent Steps) — As the workflow engine, I want long-running workflow sessions to maintain provider connectivity across many agent turns so that multi-hour workflows complete reliably.

## Purpose

Implement connection lifecycle management so that WS connections survive long sessions. Each slot tracks its connection age and proactively reconnects before the 60-minute server limit. Unexpected server Close frames trigger reconnection. The warm floor is established lazily on first pool access, not eagerly at provider construction.

## Task

Extend the pool's take path to check connection age before handing a connection to a caller. If age exceeds (60 min - reconnect_margin), close the connection, establish fresh, and hand the new one to the caller. Handle websocket_connection_limit_reached errors during a request by closing, re-establishing, and retrying once. Handle unexpected WS Close frames by marking the slot Disconnected. Establish warm floor connections on first pool access.

## Requirements

### R1: Per-slot age tracking

Each WsSlot SHALL track established_at: Option<Instant> set when a WS connection is successfully established. The age SHALL be computed as Instant::now() - established_at.

**Acceptance:**
- established_at set on successful WsConnection establishment
- established_at is None for slots in Disconnected state
- Age computable for Available slots

**Files:**
- modify: crates/norn/src/provider/openai/websocket.rs

**Checklist:**
- C34 — Per-slot connection age tracked from establishment time via std::time::Instant.

### R2: Proactive reconnection before server limit

WHEN a caller takes an Available connection AND the connection age exceeds (60 min - reconnect_margin) THE SYSTEM SHALL close the old connection, establish a fresh one in the same slot, and hand the fresh connection to the caller.

**Acceptance:**
- Connection older than (60 min - reconnect_margin) is closed before use
- Fresh connection established in the same slot
- Caller receives the fresh connection, not the stale one
- established_at updated to reflect the new connection

**Files:**
- modify: crates/norn/src/provider/openai/websocket.rs

### R3: 60-minute limit error handling

IF the server sends websocket_connection_limit_reached during a request THE SYSTEM SHALL mark the connection broken, establish a fresh connection in the slot, and retry the request once on the new connection.

**Acceptance:**
- websocket_connection_limit_reached error detected from WS error envelope
- Old connection closed, fresh one established
- Request retried once on the new connection
- If retry also fails, error propagated to caller

### R4: Unexpected Close frame handling

IF the server sends an unexpected WS Close frame THE SYSTEM SHALL mark the slot as Disconnected. The next caller to encounter the Disconnected slot SHALL establish a fresh connection.

**Acceptance:**
- WS Close frame sets slot state to Disconnected
- Connection resources are released
- Next request on this slot triggers fresh establishment

**Files:**
- modify: crates/norn/src/provider/openai/websocket.rs

**Checklist:**
- C37 — Connection dropped and recreated on any unexpected WS Close frame from server.

### R5: Lazy warm floor establishment

WHEN the pool is accessed for the first time THE SYSTEM SHALL establish the warm floor via parallel tokio::spawn tasks — one per warm slot. The first caller waits only for its own connection; the remaining warm floor connections establish in the background. The pool SHALL NOT establish connections eagerly at provider construction.

**Acceptance:**
- No WS connections established during OpenAiProvider::new()
- First call to the pool triggers warm floor establishment
- First caller's connection established synchronously; remaining slots spawn as background tasks
- Background warm floor tasks complete and set their slots to Available without blocking any caller
