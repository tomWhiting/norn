# PW-010: Transport selection, dispatch, and HTTP fallback

> **Cluster:** norn-provider-ws
> **Depends on:** PW-001, PW-005, PW-006, PW-007, PW-008
> **Checklist:**
> - C44 — WS transport attempted first for api.openai.com and chatgpt.com endpoints.
> - C45 — HTTP transport used for custom base_url endpoints without WS attempt.
> - C46 — HTTP fallback always available — pool-full, cooldown, Disabled, and establishment failure all degrade to HTTP.
> - C47 — Transport fallback logged at warn level with failure reason.
> - C56 — Pool state changes (scale up, scale down, cooldown enter/exit, Disabled) logged at debug level.
> **Stories:**
> - S3 (AI Agent, Making Provider Requests) — As an AI agent, I want provider requests to fall back to HTTP automatically if WebSocket is unavailable so that no request is lost due to transport issues.
> - S6 (Workflow Engine, Running Agent Steps) — As the workflow engine, I want the provider to handle transport selection automatically so that workflow steps don't need transport-level configuration.
> - S11 (Human Operator, Troubleshooting Provider Connectivity) — As an operator, I want transport fallback events logged with reasons so that I can diagnose why the provider switched from WebSocket to HTTP.
> - S12 (Human Operator, Troubleshooting Provider Connectivity) — As an operator, I want connection lifecycle events logged at debug level so that I can trace WebSocket establishment, reconnection, and closure.

## Purpose

Wire the WS transport into OpenAiProvider's Provider::stream() implementation. This is the final integration brief that connects the pool (PW-005), framing (PW-007), lifecycle (PW-008), and HTTP transport (PW-001) into a unified dispatch. WS is attempted first for known endpoints; HTTP is the fallback for custom endpoints and when WS is unavailable.

## Task

Modify OpenAiProvider to hold a WsPool. Update Provider::stream() to first try acquiring a WS connection from the pool. If successful, use the WS framing (PW-007) to send the request and stream events. If the pool returns a fallback signal (pool full, cooldown, custom endpoint), dispatch to the HTTP transport (PW-001). Log transport selection and fallback events.

## Requirements

### R1: WsPool on OpenAiProvider

OpenAiProvider SHALL hold an Option<WsPool> initialized for known OpenAI endpoints (api.openai.com, chatgpt.com) and None for custom base_url endpoints.

**Acceptance:**
- WsPool created during OpenAiProvider construction for known endpoints
- Option<WsPool> is None for custom base_url
- Pool config (min/max connections, idle timeout) sourced from ProviderConfig

**Files:**
- modify: crates/norn/src/provider/openai/mod.rs

**Checklist:**
- C44 — WS transport attempted first for api.openai.com and chatgpt.com endpoints.
- C45 — HTTP transport used for custom base_url endpoints without WS attempt.

**Stories:**
- S6 (Workflow Engine, Running Agent Steps) — As the workflow engine, I want the provider to handle transport selection automatically so that workflow steps don't need transport-level configuration.

### R2: WS-first dispatch in Provider::stream()

WHEN Provider::stream() is called AND the provider has a WsPool THE SYSTEM SHALL first attempt to acquire a connection from the pool. If a connection is acquired, the spawned task SHALL use WS framing to send the request and stream events.

**Acceptance:**
- stream() tries WS pool before HTTP for known endpoints
- WS path uses PW-007 send/receive functions
- Spawned task structure matches the existing HTTP pattern (mpsc channel + ReceiverStream)

**Files:**
- modify: crates/norn/src/provider/openai/mod.rs

**Checklist:**
- C44 — WS transport attempted first for api.openai.com and chatgpt.com endpoints.

**Stories:**
- S6 (Workflow Engine, Running Agent Steps) — As the workflow engine, I want the provider to handle transport selection automatically so that workflow steps don't need transport-level configuration.

### R3: HTTP fallback on pool unavailability

WHEN the WS pool signals fallback (pool full, cooldown, establishment failure) OR when the provider has no WsPool (custom endpoint) THE SYSTEM SHALL dispatch to the HTTP transport (http.rs SenderProvider).

**Acceptance:**
- Pool-full signal triggers HTTP dispatch
- Cooldown state triggers HTTP dispatch
- Custom base_url (no WsPool) always uses HTTP
- HTTP path is the exact extracted code from PW-001 — no reimplementation

**Files:**
- modify: crates/norn/src/provider/openai/mod.rs

**Checklist:**

**Stories:**

### R4: Fallback logging

WHEN the transport falls back from WS to HTTP THE SYSTEM SHALL log the event at warn level with the reason (pool full, cooldown, establishment failure, custom endpoint).

**Acceptance:**
- tracing::warn emitted on every WS-to-HTTP fallback
- Log message includes the specific reason for fallback
- No warn log when HTTP is the expected transport (custom endpoint)

### R5: Pool state change logging

Pool state changes (scale up, scale down, cooldown enter/exit) SHALL be logged at debug level.

**Acceptance:**
- tracing::debug on pool scale-up (new slot created)
- tracing::debug on pool scale-down (idle slot removed)
- tracing::debug on cooldown enter and exit

- SHALL NOT reimplement the HTTP transport — use PW-001's http.rs directly
