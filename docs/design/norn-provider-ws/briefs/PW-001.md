# PW-001: Extract HTTP transport from openai/mod.rs to openai/http.rs

> **Cluster:** norn-provider-ws
> **Checklist:**
> - C1 — SenderProvider struct and execute() method extracted from openai/mod.rs to openai/http.rs.
> - C2 — HTTP transport in http.rs is structurally symmetric with websocket.rs — each transport is a self-contained module.
> - C3 — HTTP transport produces identical ProviderEvent stream as before extraction (no behavioral change).
> **Stories:**
> - S9 (Human Developer, Maintaining the Provider) — As a developer, I want the HTTP transport extracted to its own file so that HTTP and WS transports are structurally symmetric self-contained modules.

## Purpose

Create structural symmetry for the dual-transport architecture by extracting the HTTP+SSE transport code into its own file. This is a pure refactor with zero behavioral change — it moves the SenderProvider struct and its execute() method from openai/mod.rs to openai/http.rs, establishing the pattern where each transport is a self-contained module alongside the shared OpenAI API code.

## Task

Move the SenderProvider struct and its execute() method (~220 lines) from openai/mod.rs to a new openai/http.rs file. Update mod.rs to declare the new module and use its public interface. Verify that all existing tests pass unchanged — the extraction must be a pure refactor with identical behavior.

## Requirements

### R1: Create openai/http.rs with SenderProvider

The SenderProvider struct and its execute() method SHALL be defined in openai/http.rs. The struct SHALL be pub(super) so that mod.rs can construct it. The execute method signature SHALL remain unchanged.

**Acceptance:**
- SenderProvider struct defined in openai/http.rs
- execute() method has identical signature to current mod.rs version
- SenderProvider is pub(super), not pub

**Files:**
- create: crates/norn/src/provider/openai/http.rs

**Checklist:**
- C1 — SenderProvider struct and execute() method extracted from openai/mod.rs to openai/http.rs.

**Stories:**
- S9 (Human Developer, Maintaining the Provider) — As a developer, I want the HTTP transport extracted to its own file so that HTTP and WS transports are structurally symmetric self-contained modules.

### R2: Update mod.rs module declarations

openai/mod.rs SHALL declare mod http and use SenderProvider from it in the Provider::stream() spawned task. The imports previously used by SenderProvider SHALL move to http.rs.

**Acceptance:**
- mod.rs contains pub(super) mod http or mod http declaration
- Provider::stream() constructs SenderProvider using http::SenderProvider
- No unused imports remain in mod.rs after extraction

**Files:**
- modify: crates/norn/src/provider/openai/mod.rs

**Checklist:**
- C2 — HTTP transport in http.rs is structurally symmetric with websocket.rs — each transport is a self-contained module.

### R3: Behavioral parity verification

All existing tests in openai/mod.rs test modules SHALL pass unchanged after extraction. The ProviderEvent stream produced by the HTTP transport SHALL be identical to the pre-extraction behavior.

**Acceptance:**
- All tests in openai/mod.rs pass without modification
- cargo test -p norn passes clean
- No behavioral difference observable via DebugDumper output

**Checklist:**
- C3 — HTTP transport produces identical ProviderEvent stream as before extraction (no behavioral change).

## Boundaries

- SHALL NOT change any behavior — pure refactor only
- SHALL NOT modify the Provider trait or any public API
- SHALL NOT move test modules — tests stay in mod.rs
- SHALL NOT add WebSocket code — that is PW-004+ scope

## Verification

- cargo check --workspace passes
- cargo test -p norn passes clean
- SenderProvider no longer defined in openai/mod.rs (grep confirms)
- openai/http.rs exists and contains SenderProvider + execute()
