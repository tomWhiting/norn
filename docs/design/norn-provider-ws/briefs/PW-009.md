# PW-009: Protocol acknowledgment (response.processed) and internal prewarm

> **Cluster:** norn-provider-ws
> **Depends on:** PW-007
> **Checklist:**
> - C39 — response.processed message sent on WS after receiving ProviderEvent::Done with response_id.
> - C40 — response.processed send failure logged at warn level and does not fail the request.
> - C41 — WS transport supports internal generate: false prewarm on fresh connections as a transport-level optimization.
> - C42 — Prewarm is best-effort — wasted without harm if another caller uses the connection before the real request.
> - C43 — No prewarm method on the Provider trait. Provider trait remains single-method (stream only).
> **Stories:**
> - S4 (AI Agent, Optimizing Latency) — As an AI agent, I want the WebSocket transport to automatically prime the server's prompt cache on fresh connections so that time-to-first-token is reduced without any action on my part.
> - S5 (AI Agent, Optimizing Latency) — As an AI agent, I want the provider to acknowledge completed responses on the WebSocket so that the server can manage connection state correctly.

## Purpose

Complete the WS protocol contract by sending response.processed after each completed response, and add the internal prewarm optimization that sends generate: false on fresh connections. Both are WS-specific protocol behaviors that sit on top of the request/response framing (PW-007).

## Task

After the WS receive loop produces ProviderEvent::Done, send {"type": "response.processed", "response_id": "..."} back on the WebSocket. Log failures at warn level but do not fail the request. Add an internal prewarm capability: the WS transport may send a generate: false request on fresh connections to prime the server's prompt cache. Prewarm is best-effort and internal — no method on the Provider trait.

## Requirements

### R1: Send response.processed after Done

WHEN the WS transport receives a ProviderEvent::Done with a response_id THE SYSTEM SHALL send {"type": "response.processed", "response_id": "<id>"} on the WebSocket.

**Acceptance:**
- response.processed message sent as WS Text after Done event
- response_id matches the value from ProviderEvent::Done
- Message sent before returning the connection to the pool

**Files:**
- modify: crates/norn/src/provider/openai/websocket.rs

**Checklist:**
- C39 — response.processed message sent on WS after receiving ProviderEvent::Done with response_id.

**Stories:**
- S5 (AI Agent, Optimizing Latency) — As an AI agent, I want the provider to acknowledge completed responses on the WebSocket so that the server can manage connection state correctly.

### R2: response.processed failure is non-fatal

IF sending response.processed fails (connection dropped, write error) THE SYSTEM SHALL log the failure at warn level AND SHALL NOT propagate the error to the caller. The response was already successfully received.

**Acceptance:**
- Send failure logged at tracing::warn with error details
- Caller receives the complete ProviderEvent stream regardless of ack failure
- Connection marked broken for the pool if the send failed due to connection loss

**Files:**
- modify: crates/norn/src/provider/openai/websocket.rs

**Checklist:**
- C40 — response.processed send failure logged at warn level and does not fail the request.

### R3: Internal prewarm on fresh connections

The WS transport MAY send a request with generate: false internally on a fresh connection to prime the server's prompt cache. This is a transport-level optimization, not a caller-facing API.

**Acceptance:**
- Prewarm sends the same payload structure as a real request with generate: false appended
- Server response to prewarm is consumed and discarded
- Prewarm is triggered by transport internals, not by callers

**Files:**
- modify: crates/norn/src/provider/openai/websocket.rs

**Checklist:**
- C41 — WS transport supports internal generate: false prewarm on fresh connections as a transport-level optimization.

**Stories:**
- S4 (AI Agent, Optimizing Latency) — As an AI agent, I want the WebSocket transport to automatically prime the server's prompt cache on fresh connections so that time-to-first-token is reduced without any action on my part.

### R4: Prewarm is best-effort

IF a prewarm fails or another caller takes the connection before the real request THE SYSTEM SHALL proceed with the real request normally. Prewarm failure SHALL NOT cause request failure.

**Acceptance:**
- Failed prewarm does not prevent the real request from being sent
- Connection stolen between prewarm and real request: caller falls back to another slot or HTTP
- No error propagated to caller from prewarm failure

**Files:**
- modify: crates/norn/src/provider/openai/websocket.rs

**Checklist:**
- C42 — Prewarm is best-effort — wasted without harm if another caller uses the connection before the real request.

### R5: Provider trait unchanged

The Provider trait SHALL NOT gain a prewarm method. The trait SHALL remain single-method (stream only).

**Acceptance:**
- Provider trait in traits.rs has only fn stream()
- No prewarm method on Provider, MockProvider, or any trait implementation

**Checklist:**
- C43 — No prewarm method on the Provider trait. Provider trait remains single-method (stream only).

## Boundaries

- SHALL NOT add prewarm to the Provider trait
- SHALL NOT block the real request waiting for prewarm to complete if the connection is contested
- SHALL NOT modify the ProviderEvent stream — response.processed is sent after the stream is complete
