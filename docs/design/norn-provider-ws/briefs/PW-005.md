## Purpose

Build the connection pool that manages multiple WS connections for concurrent callers. The pool provides demand-driven scaling with a configurable warm floor (min_connections) and ceiling (max_connections), per-slot state management via the WsState enum, take-use-return locking that holds the Mutex for microseconds, and idle connection cleanup. This is the central coordination layer between the raw WS connection (PW-004) and the transport dispatch (PW-010).

## Task

Create the WsPool, WsSlot, and WsState types in openai/websocket.rs. WsPool holds Vec<WsSlot> protected by tokio::sync::Mutex. Each WsSlot has a WsState (Available/InUse/Disconnected/Disabled), establishment timestamp, and last-used timestamp. Implement the take-use-return pattern: callers scan for Available slots, swap to InUse, use the connection, then return it. When all slots are InUse and pool is below max, create a new slot and establish. When pool is at max, caller falls back to HTTP. Idle connections above min_connections are closed after idle_timeout.

WsState SHALL have four variants: Available(WsConnection) for idle connections ready for use, InUse for connections claimed by a caller, Disconnected for slots awaiting fresh establishment, and Disabled for slots that have been permanently retired by the circuit breaker.

**Acceptance:**

- Enum is not pub — internal to the pool module

### R4: Take-use-return pattern

WHEN a caller requests a connection from the pool THE SYSTEM SHALL lock the Mutex, scan for the first Available slot, swap it to InUse, and unlock. The Mutex lock SHALL be held only for the state scan and swap — never for the streaming duration.

**Acceptance:**
- Mutex lock duration is bounded to the scan+swap operation
- Caller receives a WsConnection taken from an Available slot
- Slot state is InUse while the caller holds the connection

### R5: InUse guard against double establishment

WHEN a caller finds a Disconnected slot THE SYSTEM SHALL swap it to InUse before unlocking, then attempt establishment outside the lock. A concurrent caller finding the same slot as InUse SHALL move to the next slot.

**Acceptance:**
- Disconnected slot swapped to InUse before Mutex unlock
- Concurrent caller seeing InUse skips to next available slot or scales up
- No two callers attempt establishment on the same slot

**Acceptance:**

## Boundaries
