# PW-006: Pool recovery with exponential-backoff cooldown

> **Cluster:** norn-provider-ws
> **Depends on:** PW-005
> **Checklist:**
> - C28 — When all recent establishment attempts fail, pool enters cooldown with exponential backoff.
> - C29 — During cooldown, all callers use HTTP. After cooldown expires, pool tries WS again on next request.
> - C30 — Successful establishment resets cooldown backoff to zero.
> - C31 — No caller is permanently stuck on HTTP unless WS establishment fails through full cooldown escalation to Disabled.
> - C32 — Callers that used HTTP due to pool-full rotate back to WS when a slot frees up on subsequent requests.
> - C33 — Circuit breaker: when backoff reaches max_backoff and final attempt fails, pool transitions to Disabled (irreversible within session).
> **Stories:**
> - S14 (AI Agent, Making Provider Requests) — As an AI agent, I want the provider to recover WebSocket connectivity after transient failures so that I'm not stuck on HTTP for the rest of the session.

## Purpose

Add recovery semantics to the connection pool so that WS connectivity is restored after transient failures. When all establishment attempts fail, the pool enters a cooldown with exponential backoff. After the cooldown expires, the pool tries WS again. WS is permanently disabled only after the full cooldown escalation reaches max_backoff and the final attempt fails (circuit breaker).

## Task

Extend WsPool with cooldown_until (Option<Instant>), consecutive_all_failures counter, and current backoff duration. When establishment fails and all slots are Disconnected, enter cooldown. During cooldown, all callers use HTTP. After cooldown expires, reset and try WS on next request. Double the backoff on each consecutive all-fail cycle. Reset backoff on successful establishment.

## Requirements

### R1: Cooldown state on WsPool

WsPool SHALL track cooldown_until: Option<Instant>, consecutive_all_failures: u32, and current_backoff: Duration for managing recovery timing.

**Acceptance:**
- cooldown_until, consecutive_all_failures, and current_backoff fields on WsPool
- Initial state: cooldown_until is None, consecutive_all_failures is 0

**Files:**
- modify: crates/norn/src/provider/openai/websocket.rs

**Checklist:**
- C28 — When all recent establishment attempts fail, pool enters cooldown with exponential backoff.

### R2: Enter cooldown on all-fail

WHEN a WS establishment attempt fails AND all slots in the pool are Disconnected THE SYSTEM SHALL increment consecutive_all_failures, compute the next backoff (exponential doubling), and set cooldown_until to now + backoff.

**Acceptance:**
- Cooldown entered only when all slots are Disconnected (not when some are Available or InUse)
- Backoff doubles on each consecutive all-fail: base, 2x, 4x, 8x, etc.
- cooldown_until set to Instant::now() + current_backoff

**Files:**
- modify: crates/norn/src/provider/openai/websocket.rs

**Checklist:**
- C28 — When all recent establishment attempts fail, pool enters cooldown with exponential backoff.

**Stories:**
- S14 (AI Agent, Making Provider Requests) — As an AI agent, I want the provider to recover WebSocket connectivity after transient failures so that I'm not stuck on HTTP for the rest of the session.

### R3: HTTP during cooldown

WHILE the pool is in cooldown (Instant::now() < cooldown_until) WHEN a caller requests a connection THE SYSTEM SHALL signal HTTP fallback without attempting WS establishment.

**Acceptance:**
- Callers during cooldown get HTTP fallback signal immediately
- No WS establishment attempted during cooldown
- Cooldown check happens before slot scan

**Files:**
- modify: crates/norn/src/provider/openai/websocket.rs

**Checklist:**
- C29 — During cooldown, all callers use HTTP. After cooldown expires, pool tries WS again on next request.

### R4: Recovery after cooldown

WHEN cooldown expires (Instant::now() >= cooldown_until) THE SYSTEM SHALL clear cooldown_until and allow the next caller to attempt WS establishment normally.

**Acceptance:**
- After cooldown expiry, next caller attempts WS establishment
- cooldown_until set to None on expiry
- Pool returns to normal take-use-return behavior

**Files:**
- modify: crates/norn/src/provider/openai/websocket.rs

**Checklist:**
- C31 — No caller is permanently stuck on HTTP unless WS establishment fails through full cooldown escalation to Disabled.

**Stories:**
- S14 (AI Agent, Making Provider Requests) — As an AI agent, I want the provider to recover WebSocket connectivity after transient failures so that I'm not stuck on HTTP for the rest of the session.

### R5: Reset on success

WHEN a WS connection is successfully established THE SYSTEM SHALL reset consecutive_all_failures to zero and current_backoff to the base value.

**Acceptance:**
- Successful establishment resets failure counter and backoff
- Next failure cycle starts from base backoff, not from the previous escalated value

**Files:**
- modify: crates/norn/src/provider/openai/websocket.rs

**Checklist:**
- C30 — Successful establishment resets cooldown backoff to zero.

### R6: Pool-full callers rotate back

WHEN a caller used HTTP because the pool was full (all InUse, at max) THE SYSTEM SHALL allow that caller to acquire a freed slot on subsequent requests.

**Acceptance:**
- No sticky per-caller HTTP state — every request checks the pool fresh
- Caller that got HTTP on request N can get WS on request N+1 if a slot freed

**Files:**
- modify: crates/norn/src/provider/openai/websocket.rs

**Checklist:**
- C32 — Callers that used HTTP due to pool-full rotate back to WS when a slot frees up on subsequent requests.

### R7: Circuit breaker: escalation to Disabled

WHEN backoff reaches max_backoff AND the final attempt fails THE SYSTEM SHALL transition the pool to a Disabled state. In the Disabled state, all callers receive HTTP fallback without any further WS establishment attempts for the remainder of the session.

**Acceptance:**
- Pool tracks whether current_backoff has reached max_backoff
- Final failed attempt when at max_backoff sets pool state to Disabled
- Disabled pool returns HTTP fallback signal immediately on every subsequent request
- No WS establishment attempted while pool is Disabled
- Disabled state is not reversible within a session

**Files:**
- modify: crates/norn/src/provider/openai/websocket.rs

**Checklist:**
- C33 — Circuit breaker: when backoff reaches max_backoff and final attempt fails, pool transitions to Disabled (irreversible within session).

## Boundaries

- SHALL NOT hardcode the base backoff duration — configurable via ProviderConfig as base_backoff
- SHALL NOT hardcode the maximum backoff — configurable via ProviderConfig as max_backoff
- SHALL NOT make cooldown permanent before max_backoff is reached — Disabled is only entered after final failure at max_backoff
