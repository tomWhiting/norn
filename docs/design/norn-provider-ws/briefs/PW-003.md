# PW-003: DebugDumper WS entry types — ws_send and ws_lifecycle

> **Cluster:** norn-provider-ws
> **Checklist:**
> - C52 — DebugDumper gains write_ws_send method producing ws_send JSONL entry type for outgoing WS frames.
> - C53 — DebugDumper gains write_ws_lifecycle method producing ws_lifecycle JSONL entry type for connection events.
> - C54 — Incoming WS events logged via existing write_sse_event after SseEvent construction (same entry type as HTTP).
> **Stories:**
> - S10 (Human Developer, Maintaining the Provider) — As a developer, I want WS transport debug output in the same JSONL format as HTTP so that debug-api dumps are transport-agnostic.

## Purpose

Extend the DebugDumper with entry types for WebSocket traffic so that WS transport debug output uses the same JSONL structure as HTTP. Outgoing WS frames get a ws_send entry type, connection lifecycle events get a ws_lifecycle entry type, and incoming WS events reuse the existing sse_event type via SseEvent mapping (D3).

## Task

Add two new DebugPayload variants (WsSend and WsLifecycle) and corresponding write methods (write_ws_send and write_ws_lifecycle) to the DebugDumper. Incoming WS events need no new code — they are mapped to SseEvent and logged via the existing write_sse_event method.

## Requirements

### R1: WsSend debug payload variant

The DebugPayload enum SHALL include a WsSend variant with message_type (String) and body (serde_json::Value) fields. The entry type discriminator SHALL be ws_send.

**Acceptance:**
- DebugPayload::WsSend variant exists with message_type and body fields
- Serializes to JSONL with type: ws_send

**Files:**
- modify: crates/norn/src/provider/debug.rs

**Checklist:**
- C52 — DebugDumper gains write_ws_send method producing ws_send JSONL entry type for outgoing WS frames.

**Stories:**
- S10 (Human Developer, Maintaining the Provider) — As a developer, I want WS transport debug output in the same JSONL format as HTTP so that debug-api dumps are transport-agnostic.

### R2: WsLifecycle debug payload variant

The DebugPayload enum SHALL include a WsLifecycle variant with event (String — connect/reconnect/close), url (String), and reason (Option<String>) fields. The entry type discriminator SHALL be ws_lifecycle.

**Acceptance:**
- DebugPayload::WsLifecycle variant exists with event, url, and reason fields
- Serializes to JSONL with type: ws_lifecycle
- reason is omitted from JSON when None

**Files:**
- modify: crates/norn/src/provider/debug.rs

**Checklist:**
- C53 — DebugDumper gains write_ws_lifecycle method producing ws_lifecycle JSONL entry type for connection events.

**Stories:**
- S10 (Human Developer, Maintaining the Provider) — As a developer, I want WS transport debug output in the same JSONL format as HTTP so that debug-api dumps are transport-agnostic.

### R3: write_ws_send and write_ws_lifecycle methods

DebugDumper SHALL expose pub fn write_ws_send(&self, message_type: &str, body: &serde_json::Value) and pub fn write_ws_lifecycle(&self, event: &str, url: &str, reason: Option<&str>) methods that append the corresponding JSONL entries.

**Acceptance:**
- write_ws_send appends a ws_send entry with timestamp, message_type, and body
- write_ws_lifecycle appends a ws_lifecycle entry with timestamp, event, url, and optional reason
- Both methods use the same append_entry infrastructure as existing methods

**Files:**
- modify: crates/norn/src/provider/debug.rs

**Checklist:**
- C52 — DebugDumper gains write_ws_send method producing ws_send JSONL entry type for outgoing WS frames.
- C53 — DebugDumper gains write_ws_lifecycle method producing ws_lifecycle JSONL entry type for connection events.

### R4: Incoming WS events reuse write_sse_event

Incoming WS messages SHALL be logged via the existing write_sse_event method after being mapped to SseEvent. No new DebugDumper method is needed for incoming WS events.

**Acceptance:**
- No new method added for incoming WS events
- Design doc (D3) confirms SseEvent mapping makes write_sse_event applicable to WS

**Checklist:**
- C54 — Incoming WS events logged via existing write_sse_event after SseEvent construction (same entry type as HTTP).

## Boundaries

- SHALL NOT modify existing DebugPayload variants (Request, ResponseMeta, SseEvent)
- SHALL NOT modify the write_request, write_response_meta, or write_sse_event methods
- SHALL NOT add WebSocket transport code — this brief only extends the debug infrastructure
