# PW-007: WS request/response framing and event mapping

> **Depends on:** PW-003, PW-004

## Purpose

Implement the WebSocket wire protocol for sending requests and receiving streaming events. Outgoing requests are wrapped in a response.create JSON envelope. Incoming WS text messages are parsed as JSON, mapped to SseEvent, and fed through the existing map_sse_event() dispatcher — zero duplication of event mapping logic (D3). WS-specific error envelopes are classified through the existing classify_failed_error().

## Task

Add send and receive functions to openai/websocket.rs. The send function wraps a ResponsesApiPayload (from build_payload) in a {"type": "response.create", ...} envelope and sends it as a WS Text message. The receive loop reads WS Text messages, parses each as JSON, extracts the type field as event_type, constructs an SseEvent, and feeds it through map_sse_event(). WS error envelopes ({"type": "error"}) are mapped to ProviderError. Debug logging via DebugDumper write_ws_send (PW-003) and write_sse_event.

## Requirements

### R1: response.create envelope

WHEN sending a request over WS THE SYSTEM SHALL wrap the ResponsesApiPayload fields in a JSON object with "type": "response.create" and send it as a WS Text message.

**Acceptance:**
- Outgoing WS message is a JSON object with type field set to response.create
- All ResponsesApiPayload fields (model, instructions, input, tools, etc.) present in the envelope
- Sent as tungstenite Message::Text

**Files:**
- modify: crates/norn/src/provider/openai/websocket.rs

**Checklist:**
- C10 — Request payload wrapped in {"type": "response.create", ...} envelope before WS send.

### R2: WS message to SseEvent mapping

WHEN a WS Text message is received THE SYSTEM SHALL parse it as JSON, extract the type field as event_type, use the full JSON object as data, construct an SseEvent, and feed it through map_sse_event().

**Acceptance:**
- WS JSON message {"type": "response.output_text.delta", "delta": "hello"} produces SseEvent with event_type response.output_text.delta
- data field is the complete JSON object (not just the delta)
- map_sse_event produces the same ProviderEvent as it would for an equivalent SSE event

### R3: Event stream parity

The ProviderEvent stream produced by the WS transport SHALL match the HTTP transport for identical request payloads. All event types (TextDelta, ThinkingDelta, ToolCallDelta, TextComplete, ThinkingComplete, ToolCallComplete, Done) SHALL be produced identically.

**Acceptance:**
- WS transport produces the same sequence of ProviderEvent variants as HTTP for the same conversation
- Done event includes stop_reason, usage, and response_id
- No event types are silently dropped by the WS path

### R4: WS error envelope mapping

WHEN a WS message has type "error" THE SYSTEM SHALL extract the error object and map it to ProviderError. Rate limit errors (status 429) SHALL include retry_after. Authentication errors (status 401) SHALL use ProviderError::AuthenticationFailed.

**Acceptance:**
- WS error {"type": "error", "error": {"code": "rate_limit_exceeded", ...}} maps to ProviderError::RateLimited
- WS error with status 401 maps to ProviderError::AuthenticationFailed
- Other error codes mapped via classify_failed_error where applicable

**Files:**
- modify: crates/norn/src/provider/openai/websocket.rs

**Checklist:**

### R5: Debug logging for WS traffic

WHEN sending a WS message THE SYSTEM SHALL call DebugDumper::write_ws_send with the message type and body. WHEN receiving WS events THE SYSTEM SHALL call DebugDumper::write_sse_event after SseEvent construction. The WS transport SHALL log request start, event count, and completion at debug level.

**Acceptance:**
- Outgoing response.create envelope logged via write_ws_send
- Incoming events logged via write_sse_event (same entry type as HTTP)
- tracing::debug spans for request start, event count, completion match HTTP transport pattern
