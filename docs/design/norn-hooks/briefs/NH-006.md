## Purpose

This brief wires the hook system into the norn-cli runtime. At startup, the CLI reads hook configuration from settings files (using NH-003's loader), builds ShellCommandHook instances (using NH-005), and registers them on the HookRegistry that the agent loop consumes. After this brief, users can configure hooks in settings.json and have them enforced at runtime.

## Task

Update norn-cli runtime builder to call load_hooks_from_settings during build_runtime, construct ShellCommandHook instances from the loaded HookEntry values, and register them on the HookRegistry that gets installed on LoopContext. Handle the case where no settings files exist (no hooks registered, no error). Use NC-001's path resolution for locating ~/.norn/settings.json.

## Requirements

### R1: Load hook settings during build_runtime

WHEN build_runtime executes, it SHALL call load_hooks_from_settings with the working directory to obtain the merged HookSettings. IF load_hooks_from_settings returns an error, the error SHALL be propagated. IF no settings files exist, an empty HookSettings SHALL be used (no hooks, no error).

**Acceptance:**
- build_runtime or a new wiring function calls load_hooks_from_settings(cwd)
- Uses norn::config::paths for ~/.norn/settings.json resolution
- Error from load_hooks_from_settings propagates as BuildError
- No settings files → empty HookSettings → no hooks registered → no error
- Test: builder with no settings files produces a HookRegistry with 0 hooks

### R2: Build ShellCommandHook instances from HookEntry values

For each HookEntry in the loaded HookSettings, the builder SHALL construct a ShellCommandHook with the entry's command, compiled matcher, validated timeout, and the appropriate HookEventType. The hook context (session_id, cwd, agent_id, profile_name) SHALL be populated from the runtime bundle.

**Acceptance:**
- Each HookEntry produces one ShellCommandHook instance
- HookEntry.matcher compiled via HookMatcher::new — invalid regex is a startup error
- HookEntry.timeout (validated as required by NH-003 loader) converted to Duration
- HookEventType determined by which field of HookSettings the entry came from
- ShellCommandHook context populated with session ID, working directory, root agent ID, and profile name from the runtime bundle
- Test: HookSettings with 2 pre_tool entries produces 2 ShellCommandHook instances registered as PreTool

### R3: Register hooks on HookRegistry

Built ShellCommandHook instances SHALL be registered on the HookRegistry via Hook enum variants matching their HookEventType. The registration order SHALL reflect the 3-tier merge order: user-global hooks first, then project hooks, then local hooks.

**Acceptance:**
- Each ShellCommandHook boxed and wrapped in the appropriate Hook variant
- HookEventType::PreTool → Hook::PreTool(Box::new(shell_hook))
- HookEventType::PostTool → Hook::PostTool(Box::new(shell_hook))
- All 13 event types mapped to their Hook variants
- Registration order matches concatenation order from load_hooks_from_settings
- HookRegistry installed on LoopContext.hooks (may already be Some if programmatic hooks were registered earlier — shell hooks append, not replace)
- Test: HookRegistry pre_tool_len() returns correct count after registration

### R4: Settings snapshot at startup

Hook settings SHALL be read once during build_runtime and not re-read during the session. The HookRegistry populated at startup SHALL be the definitive set of hooks for the session's lifetime.
