## Purpose

This is the bridge between config-driven hooks and the trait-based dispatch system. ShellCommandHook is a single struct that implements all 11 hook traits, spawning an external process for each hook invocation. It handles the stdin/stdout JSON protocol, exit code interpretation, timeout enforcement, matcher filtering, and the fire-and-forget semantics for session event shell hooks. After this brief, config entries can be translated into registered hooks.

## Task

Create crates/norn/src/integration/hooks/shell.rs with ShellCommandHook struct. Implement all 11 hook traits (5 existing + 6 new). Each trait method: check matcher, build HookInput, spawn sh -c, pipe stdin, enforce timeout, interpret exit code, parse HookOutput, return HookOutcome. Add fire-and-forget dispatch for session event shell hooks via tokio::spawn. Add non-reentrant guard on HookRegistry session event dispatch. Update hooks/mod.rs.

### R4: Exit code protocol

WHEN exit code is 0, stdout SHALL be parsed as HookOutput JSON and mapped to HookOutcome. WHEN exit code is 2, HookOutcome::Block SHALL be returned with stderr as reason. WHEN exit code is any other value, HookOutcome::Proceed SHALL be returned with stderr logged at warn level.

**Acceptance:**
- Exit 0: parse stdout as HookOutput, call to_hook_outcome(self.event_type)
- Exit 0 with empty stdout: HookOutcome::Proceed (no JSON to parse)
- Exit 0 with invalid JSON stdout: HookOutcome::Proceed, warning logged
- Exit 2: HookOutcome::Block { reason: stderr_string }
- Exit 2 with empty stderr: Block with empty reason string
- Other exit codes: HookOutcome::Proceed, stderr logged at warn
- Test: script exits 0 with '{"decision":"block","reason":"test"}' → Block
- Test: script exits 2 with 'denied' on stderr → Block { reason: 'denied' }
- Test: script exits 1 → Proceed with warning
- Test: script exits 0 with empty stdout → Proceed

### R5: Matcher check before spawn

WHEN a ShellCommandHook fires for an event type that supports matchers, it SHALL check the matcher against the event's matcher input BEFORE spawning the child process. IF the matcher does not match, the hook SHALL return HookOutcome::Proceed immediately without spawning.

**Acceptance:**
- Pre-tool hook checks matcher against envelope.tool_name
- Pre-LLM hook checks matcher against request.model
- Session event hook checks matcher against event variant name
- No-matcher events (user_prompt, stop, etc.) skip the matcher check
- Non-matching events return Proceed without spawning a process
- Test: ShellCommandHook with matcher 'Write' does not fire for tool name 'Read'

### R8: Non-reentrant session event dispatch guard

HookRegistry SHALL have a non-reentrant guard on run_on_event that prevents recursive dispatch. IF run_on_event is called while already inside a run_on_event dispatch, the inner call SHALL skip hook dispatch silently.

**Acceptance:**
- HookRegistry gains a dispatching_session_events: AtomicBool field (or equivalent)
- run_on_event checks and sets the flag before dispatching, clears after
- Nested calls while flag is set skip dispatch and return immediately
- Flag is always cleared even if a hook panics (guard pattern)
- Test: SessionEventHook that calls append_and_notify (triggering run_on_event) does not recurse
