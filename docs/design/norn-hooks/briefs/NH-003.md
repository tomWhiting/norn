## Purpose

Config-driven hooks require typed config structures, regex-based matchers, and settings file loading. This brief defines the HookEventType taxonomy that maps config names to traits, extends NC-002's HookSettings with the 8 additional event types from the norn-hooks design, implements regex-compiled matchers, and builds the 3-tier settings loader. NH-005 (ShellCommandHook) and NH-006 (CLI wiring) consume these types.

## Task

Create config.rs with HookEventType enum (13 variants) and any bridge types needed beyond NC-002's HookSettings/HookEntry. Extend HookSettings in norn::config::types with the 8 new event type fields (user_prompt, stop, subagent_start, subagent_stop, session_start, session_end, pre_compaction, post_tool_failure). Create matchers.rs with HookMatcher: compile regex at load time, match against per-event-type inputs. Create loader.rs with load_hooks_from_settings: read 3 settings files, merge hook arrays, validate required timeout, return parsed config. All in crates/norn/src/integration/hooks/.

### R4: HookMatcher regex compilation

HookMatcher SHALL compile a regex pattern at construction time. WHEN the pattern is None, empty, or "*", the matcher SHALL match all inputs. IF the pattern is not valid regex, THEN construction SHALL return a config error.

**Acceptance:**
- HookMatcher::new(pattern: Option<&str>) -> Result<Self, ConfigError>
- None, empty string, and "*" all produce a match-all matcher
- Valid regex compiled via regex::Regex
- Invalid regex returns ConfigError with the regex error message
- matches(&self, input: &str) -> bool uses regex full-match semantics
- Test: exact match ("Write" matches "Write", not "WriteFile")
- Test: regex match ("Edit|Write" matches both)
- Test: wildcard match ("*" matches anything)
- Test: invalid regex rejected at construction

### R5: Per-event-type matcher inputs

Each HookEventType SHALL define what string the matcher is tested against. Tool events match tool name. LLM events match model name. Subagent events match profile name or agent type. Session events match event variant name. Events without matcher support always fire.

**Acceptance:**
- Tool events (pre_tool, post_tool, post_tool_failure): matcher tested against tool_name
- LLM events (pre_llm, post_llm): matcher tested against model name
- Subagent events (subagent_start, subagent_stop): matcher tested against agent type string
- Session event: matcher tested against event variant name (e.g. "UserMessage", "ToolResult")
- No-matcher events (user_prompt, stop, session_start, session_end, pre_compaction): always fire regardless of matcher value
- A method or function maps HookEventType to whether matchers are supported
- Test: verify each event type category returns the correct matcher input behavior

### R6: Settings file loading and 3-tier merge

load_hooks_from_settings SHALL read hook configuration from up to 3 files: ~/.norn/settings.json (user-global), .norn/settings.json (project), .norn/settings.local.json (local project). WHEN multiple files exist, hook arrays SHALL be concatenated per event type (user-global first, local-project last). Missing files SHALL be silently skipped.

**Acceptance:**
- load_hooks_from_settings(cwd: &Path) -> Result<HookSettings, ConfigError>
- Reads ~/.norn/settings.json via norn::config::paths::norn_dir()
- Reads {cwd}/.norn/settings.json
- Reads {cwd}/.norn/settings.local.json
- Missing files silently skipped (not errors)
- Per event type: arrays concatenated in order (user-global, project, local)
- Invalid JSON in any file produces ConfigError with file path in message
- Settings captured as a snapshot — the function reads once and returns
- Test: 3-tier merge with overlapping event types produces concatenated arrays
- Test: missing files produce empty HookSettings (not error)
