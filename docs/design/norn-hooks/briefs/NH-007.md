# NH-007: Hook dispatch wiring for new lifecycle events

> **Depends on:** NH-002, NH-006
> **Checklist:**

> - C55 — StopHook dispatched after the final iteration before AgentStepResult is returned.
> - C56 — SubagentHook on_subagent_start dispatched in spawn.rs launch function.
> - C57 — SubagentHook on_subagent_stop dispatched in spawn.rs finish function.
> - C58 — CompactionHook dispatched in compaction.rs before compaction event is appended.
> - C59 — PostToolFailureHook dispatched in tool_dispatch.rs when tool output contains an error.
> - C60 — SessionLifecycleHook on_session_start dispatched in norn-cli runtime at session construction.
> - C61 — SessionLifecycleHook on_session_end dispatched in norn-cli runtime at session teardown.

> - S19 (AI Agent, Operating Under Hooks) — As an AI agent, I want stop hooks to be able to force me to continue working so that external logic can override my stopping decision when tasks remain.
> - S20 (AI Agent, Operating Under Hooks) — As an AI agent, I want hook blocks to surface as clear error messages so that I understand why an operation was prevented and can try an alternative.

## Purpose

NH-002 defined the 6 new hook traits and NH-006 wired config-driven hooks into the registry. But the new traits have no dispatch points — the agent loop never calls them. This brief wires each new trait into its dispatch point: UserPromptHook at loop entry, StopHook at loop exit, SubagentHook in spawn.rs, CompactionHook in compaction.rs, SessionLifecycleHook in norn-cli, and PostToolFailureHook in tool_dispatch.rs.

## Task

Wire 6 new hook dispatch points into the agent loop and CLI runtime. Each dispatch point follows the pattern established by the existing pre/post tool and pre/post LLM hooks: check if hooks is Some on LoopContext, call the appropriate dispatch method, handle Block outcomes where applicable. For blocking hooks, propagate as NornError::HookBlocked with the appropriate HookType variant.

## Requirements

### R1: UserPromptHook dispatch at loop entry

WHEN run_agent_step begins, WHILE LoopContext.hooks is Some, the runner SHALL call run_user_prompt with the user prompt and session ID. IF the result is Block, the runner SHALL return NornError::HookBlocked with HookType::UserPrompt.

**Acceptance:**
- run_user_prompt called in run_agent_step_inner before build_initial_messages
- Block returns NornError::HookBlocked { hook_type: HookType::UserPrompt, reason }
- Proceed continues normal execution
- When hooks is None, dispatch is skipped
- Test: register UserPromptHook that blocks, verify HookBlocked error returned

**Files:**
- modify: crates/norn/src/loop/runner.rs

**Checklist:**
- C54 — UserPromptHook dispatched at the entry of run_agent_step before initial message construction.

**Stories:**

### R2: StopHook dispatch at loop exit

WHEN the agent loop determines the model wants to stop (no tool calls, final text produced), WHILE LoopContext.hooks is Some, the runner SHALL call run_stop with the final text. IF the result is Block, the runner SHALL inject the block reason as a user message and continue the loop instead of returning.

**Acceptance:**
- run_stop called after classifying the response as terminal (no tool calls)
- Block causes the loop to inject the block reason as a follow-up user message
- Block prevents AgentStepResult from being returned on this iteration
- Proceed allows normal stop (AgentStepResult returned)
- When hooks is None, dispatch is skipped
- Test: register StopHook that blocks, verify the loop continues with injected reason

### R3: SubagentHook dispatch in spawn.rs

WHEN a sub-agent is spawned via SpawnAgentTool, the runner SHALL call run_subagent_start with the child's agent ID and type. WHEN a sub-agent completes (in the finish function), the runner SHALL call run_subagent_stop. IF run_subagent_stop returns Block, the child SHALL NOT be marked as completed.

**Acceptance:**
- run_subagent_start called in launch_child after the child agent is spawned
- run_subagent_stop called in finish_child before marking terminal status
- Block from run_subagent_stop prevents status transition to Completed/Failed
- Agent ID and profile name (or agent type) passed to both hooks
- Hooks retrieved from parent's LoopContext (if available) or from ToolContext extension
- Test: register SubagentHook, verify start fires on spawn, stop fires on completion

**Files:**
- modify: crates/norn/src/tools/agent/spawn.rs

### R4: CompactionHook dispatch in compaction.rs

WHEN auto-compaction triggers, WHILE LoopContext.hooks is Some, the compaction logic SHALL call run_pre_compaction with the current event count. IF the result is Block, compaction SHALL be skipped for this trigger (not an error).

**Acceptance:**
- run_pre_compaction called in maybe_auto_compact before the compaction event is created
- Block causes maybe_auto_compact to return without compacting (Ok, no compaction)
- Proceed allows compaction to proceed normally
- Event count passed to the hook is the number of events currently in the store
- When hooks is None, dispatch is skipped
- Test: register CompactionHook that blocks, verify compaction is skipped

**Files:**
- modify: crates/norn/src/loop/compaction.rs

**Checklist:**
- C58 — CompactionHook dispatched in compaction.rs before compaction event is appended.

### R5: PostToolFailureHook dispatch in tool_dispatch.rs

WHEN a tool execution produces an error output (output contains 'error' key or is_error is true), WHILE LoopContext.hooks is Some, tool_dispatch SHALL call run_post_tool_failure with the envelope and output. This is observational — no return value affects control flow.

**Acceptance:**
- run_post_tool_failure called in execute_tool_call when the tool output indicates an error
- Fires in addition to (not instead of) the existing PostToolHook
- PostToolHook fires for all tool completions; PostToolFailureHook fires only for errors
- Receives the same ToolEnvelope and ToolOutput as PostToolHook
- When hooks is None, dispatch is skipped
- Test: register PostToolFailureHook with counter, trigger tool error, verify counter incremented

**Files:**

### R6: SessionLifecycleHook dispatch in norn-cli runtime

WHEN the norn-cli runtime creates a session, it SHALL call run_session_start with the session ID. WHEN the session ends (normal exit or error), it SHALL call run_session_end with the session ID. Both are observational.

**Acceptance:**
- run_session_start called after HookRegistry is built and installed on LoopContext, before the first run_agent_step
- run_session_end called during session teardown (in a drop guard or explicit cleanup path)
- Session ID passed to both hooks
- Both hooks fire even if no shell hooks are configured (trait hooks may be registered programmatically)
- When hooks is None, dispatch is skipped
- Test: register SessionLifecycleHook, run a session, verify start and end both fire
