# NS-005: Runtime integration: effort override, deferred fields, and slash commands

> **Cluster:** norn-skills
> **Depends on:** NS-001, NS-002, NS-004
> **Checklist:**

> **Stories:**

> - S16 (Profile Author, Configuring Skills for Agents) — As a profile author, I want to create background-knowledge skills (user-invocable: false) so that the model can load them automatically but the user doesn't see them in the slash command menu.

## Purpose

Skills can override the agent's reasoning effort for the activation turn, store deferred fields (allowed-tools, fork mode) as metadata, and register as slash commands for user invocation. This brief combines the runtime integration features: SkillEffort-to-ReasoningEffort mapping, effort override lifecycle on LoopContext, allowed-tools as a stored hint, fork mode documentation, slash command registration from the catalog, and the invocation control matrix.

## Task

Add SkillEffort-to-ReasoningEffort mapping (max maps to Max). Add override/restore methods on LoopContext for effort. Store allowed-tools as hint with diagnostic. Document fork mode semantics (skill body = task input). Register user_invocable skills as slash commands via SlashCommandRegistry. Implement the invocation control matrix: user_invocable controls slash command visibility, disable_model_invocation controls catalog listing visibility.

## Requirements

### R1: SkillEffort to ReasoningEffort mapping

skill/types.rs SHALL provide a method or From impl that maps SkillEffort to ReasoningEffort: Low->Low, Medium->Medium, High->High, XHigh->XHigh, Max->Max.

**Acceptance:**
- SkillEffort::Low maps to ReasoningEffort::Low
- SkillEffort::Medium maps to ReasoningEffort::Medium
- SkillEffort::High maps to ReasoningEffort::High
- SkillEffort::XHigh maps to ReasoningEffort::XHigh
- SkillEffort::Max maps to ReasoningEffort::Max

**Files:**
- modify: crates/norn/src/skill/types.rs

**Checklist:**
- C51 — SkillEffort maps to ReasoningEffort: low->Low, medium->Medium, high->High, xhigh->XHigh, max->Max.

**Stories:**
- S14 (Profile Author, Configuring Skills for Agents) — As a profile author, I want to set an effort level in a skill's frontmatter so that the agent uses appropriate reasoning effort for that skill's domain.

### R2: Effort override on LoopContext

WHEN a skill with an effort field is invoked, LoopContext reasoning_effort SHALL be overridden with the mapped value. The previous value SHALL be captured. After the activation turn's provider call completes, the previous value SHALL be restored.

**Acceptance:**
- After applying effort override, LoopContext.reasoning_effort is Some(mapped_value)
- The previous reasoning_effort value is returned for restoration
- Restoring returns LoopContext.reasoning_effort to its original state
- A skill without effort field does not modify LoopContext.reasoning_effort

**Files:**
- modify: crates/norn/src/loop/loop_context.rs

**Checklist:**
- C52 — Skill effort overrides LoopContext.reasoning_effort for the activation turn, then restores the previous value.

**Stories:**
- S20 (Workflow Author, Orchestrating Agents with Skills) — As a workflow author, I want skill effort overrides to apply to the child LoopContext in fork mode so that sub-agents use the appropriate reasoning level.

### R3: allowed-tools stored as hint

The allowed_tools field SHALL be stored in SkillMetadata (done in NS-001) but NOT enforced at runtime. A diagnostic SHALL be recorded noting that allowed-tools is stored but not enforced.

**Acceptance:**
- A skill with allowed-tools: 'Bash(git *) Read' has the field stored in metadata
- No tool filtering or permission changes occur when the skill is activated
- A diagnostic is recorded stating allowed-tools is not enforced in current version

**Files:**
- modify: crates/norn/src/skill/types.rs

**Checklist:**
- C53 — The allowed_tools field is parsed and stored in SkillMetadata but not enforced.

### R4: Fork mode documentation and metadata

Fork mode metadata (context: fork, agent field) SHALL be stored in SkillMetadata. When fork execution is wired (future), the agent field SHALL select the subagent configuration (system prompt, tools); the expanded skill body SHALL be the task input, NOT the system prompt.

**Acceptance:**
- A skill with context: fork and agent: Explore has both fields stored correctly
- No fork execution occurs in this brief (deferred to agent wiring)
- The types.rs documentation clearly states skill body = task, not system prompt

**Files:**
- modify: crates/norn/src/skill/types.rs

**Checklist:**
- C56 — Fork mode: the agent field selects the subagent configuration; the expanded skill body becomes the task input, not the system prompt.

### R5: Slash command registration from catalog

WHEN building the SlashCommandRegistry, each skill in the SkillCatalog where user_invocable is true SHALL be registered as a slash command named after the skill.

**Acceptance:**
- A skill 'deploy' with user_invocable: true is registered as /deploy
- A skill 'background-knowledge' with user_invocable: false is NOT registered
- Slash command names match skill names exactly

**Files:**
- modify: crates/norn/src/skill/catalog.rs

**Checklist:**
- C28 — Each skill where user_invocable is true is registered in the SlashCommandRegistry.

**Stories:**
- S8 (Human Operator, Invoking Skills Interactively) — As a human operator, I want to invoke skills via slash commands (/skill-name) so that I can trigger domain workflows without remembering tool syntax.

### R6: Slash command handler invokes SkillTool

WHEN a skill slash command is invoked with text (e.g. /deploy prod), the handler SHALL call the SkillTool with name set to the skill name and arguments set to the trailing text.

**Acceptance:**
- /deploy prod calls SkillTool with name='deploy' and arguments='prod'
- /deploy (no args) calls SkillTool with name='deploy' and arguments absent
- The handler returns the SkillTool's output to the model

**Files:**
- modify: crates/norn/src/skill/catalog.rs

**Stories:**
- S8 (Human Operator, Invoking Skills Interactively) — As a human operator, I want to invoke skills via slash commands (/skill-name) so that I can trigger domain workflows without remembering tool syntax.

### R7: Invocation control matrix

The invocation control SHALL follow three modes: (1) default — slash command AND in catalog listing. (2) disable_model_invocation: true — slash command, NOT in listing. (3) user_invocable: false — no slash command, in listing.

**Acceptance:**
- Default skill: slash command registered AND in system_prompt_listing()
- disable_model_invocation: true: slash command registered, NOT in listing
- user_invocable: false: no slash command, appears in listing
