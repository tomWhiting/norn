## Purpose

Establish the skill module's type system and file loading infrastructure. All other norn-skills briefs depend on SkillMetadata, StringOrList, and the loader. NC-001 provides the shared frontmatter splitter; this brief builds the skill-specific deserialization on top of it, including the custom StringOrList deserializer that makes Claude Code SKILL.md files parseable.

## Task

Create the skill/ folder module with types.rs (SkillMetadata with serde rename_all = kebab-case, StringOrList custom deserializer, supporting enums) and loader.rs (frontmatter parsing into SkillMetadata, skill file discovery, lenient validation with diagnostics). The loader uses util::frontmatter::split_frontmatter() from NC-001. StringOrList handles both 'arguments: issue branch' (string) and 'arguments: [issue, branch]' (list) YAML shapes.

### R2: StringOrList custom deserializer

skill/types.rs SHALL define a StringOrList type that deserializes from either a space-separated YAML string or a YAML list of strings. Serialization SHALL produce a list.

**Acceptance:**
- serde_yaml::from_str('"issue branch"') deserializes to StringOrList(["issue", "branch"])
- serde_yaml::from_str('["issue", "branch"]') deserializes to StringOrList(["issue", "branch"])
- serde_yaml::from_str('"single"') deserializes to StringOrList(["single"])
- serde_yaml::from_str('[]') deserializes to StringOrList([])
- StringOrList provides as_slice() and into_vec() accessors

**Files:**
- create: crates/norn/src/skill/types.rs

### R5: Frontmatter parsing with lenient validation

WHEN a SKILL.md file is loaded, skill/loader.rs SHALL split the content using util::frontmatter::split_frontmatter(), deserialize the YAML into SkillMetadata with deny_unknown_fields NOT set, and apply lenient validation: warn on name mismatch or length, skip on missing description.

**Acceptance:**
- loader.rs imports and calls util::frontmatter::split_frontmatter()
- YAML with all fields deserializes correctly
- YAML with unknown fields does not error
- A skill with missing description is skipped and a diagnostic is recorded
- A skill with name exceeding 64 chars produces a diagnostic warning but is still loaded
- A skill with name not matching directory name produces a diagnostic warning but is still loaded

### R6: Name defaulting from directory

WHEN the name field is absent from the SKILL.md frontmatter, the loader SHALL default the name to the directory name (for <name>/SKILL.md) or the file stem (for <name>.md).

**Acceptance:**
- A SKILL.md in my-skill/ with no name field produces SkillMetadata.name == Some("my-skill")
- A flat file deploy.md with no name field produces SkillMetadata.name == Some("deploy")
- A SKILL.md with name: custom produces SkillMetadata.name == Some("custom")

### R7: Skill file discovery

WHEN given an ordered list of search directories, the loader SHALL discover skills by checking <name>/SKILL.md and <name>.md patterns in each directory.

**Acceptance:**
- A directory containing my-skill/SKILL.md is discovered as skill 'my-skill'
- A directory containing alias.md is discovered as skill 'alias'
- If both my-skill/SKILL.md and my-skill.md exist, the directory-based skill wins
- Discovery returns a list of (name, path, metadata, body) tuples
