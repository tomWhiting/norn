## Purpose

Skill templates contain dynamic placeholders that must be resolved before the model sees the content. This brief implements the three-stage expansion pipeline with full shell execution safety: global disable policy, cwd specification, stdout cap, stderr handling. This is what makes Claude Code SKILL.md files work in Norn without modification.

## Task

Create skill/template.rs with an expand() function that processes a skill body through three sequential stages. Stage 1 executes backtick-bang commands with safety controls (disable policy, cwd, 32KB stdout cap, 1KB stderr in failures, 5s timeout). Stage 2 resolves dollar-sign arguments and built-ins. Stage 3 resolves mustache variables. Each stage is single-pass; replacement text is visible to subsequent stages.

## Requirements

### R1: Stage 1: Inline backtick-bang execution

WHEN the skill body contains !`command` patterns, stage 1 SHALL execute each command via sh -c (or the configured shell), replace the pattern with trimmed stdout, and enforce a 5-second timeout.

**Acceptance:**
- !`echo hello` is replaced with 'hello'
- Multiple !`...` patterns in the same body are each independently executed
- Text between backtick-bang patterns is preserved verbatim

**Files:**
- create: crates/norn/src/skill/template.rs
- modify: crates/norn/src/skill/mod.rs

**Checklist:**
- C30 — Stage 1 finds and executes inline !`command` patterns, replacing with trimmed stdout.
- C35 — Stage 1 enforces a 5-second timeout per command.

**Stories:**

### R2: Stage 1: Fenced block execution

WHEN the skill body contains ```! fenced blocks, stage 1 SHALL execute the block content as a single shell invocation and replace the entire fenced block with trimmed stdout. Standard ``` code blocks (without !) SHALL NOT be executed.

**Acceptance:**
- A ```! block containing 'echo one\necho two' produces 'one\ntwo'
- Standard ``` code blocks without ! are left unchanged
- The opening ```! and closing ``` are both removed from output

**Files:**

**Checklist:**

### R3: Stage 1: Failure markers with stderr

IF a command fails (non-zero exit, timeout, spawn error), stage 1 SHALL replace the pattern with '[skill shell command failed: {error}]' including the first 1KB of stderr on failure.

**Acceptance:**
- !`exit 1` produces a failure marker containing the exit code
- !`sleep 10` with 5s timeout produces a timeout failure marker
- Stderr content (up to 1KB) is included in the failure marker for non-zero exit

**Files:**

### R4: Stage 1: Shell field, cwd, disable policy, stdout cap

Stage 1 SHALL respect the shell parameter (Bash vs PowerShell), run commands from the agent's cwd (not skill dir), respect disableSkillShellExecution by replacing with policy marker, and truncate stdout at 32KB.

**Acceptance:**
- With shell=PowerShell, commands execute via pwsh -c
- Commands run from the agent's working directory, not the skill directory
- With disableSkillShellExecution=true, all commands produce '[shell command execution disabled by policy]'
- Output exceeding 32KB is truncated with '[truncated — output exceeded 32KB]' appended

### R5: Stage 2: Dollar-sign expansion

Stage 2 SHALL resolve $ARGUMENTS, $N/$ARGUMENTS[N], named $name, ${CLAUDE_SESSION_ID}, ${CLAUDE_EFFORT}, ${CLAUDE_SKILL_DIR}. $$ SHALL escape to literal $. Unrecognized $name references SHALL be left as-is.

**Acceptance:**
- $ARGUMENTS resolves to the full arguments string
- $0 resolves to the first positional argument
- Named $issue with arguments=['issue'] resolves correctly
- ${CLAUDE_SESSION_ID} resolves to a non-empty session ID
- $$ produces a literal $
- $unknown_ref passes through as-is

### R7: Three-stage ordering and rescanning

The three stages SHALL run in order: backtick-bang, dollar-sign, mustache. Each stage is single-pass. Replacement text produced by stage N is visible to stage N+1.

**Acceptance:**
- Shell output containing $ARGUMENTS is dollar-expanded in stage 2
- Dollar expansion output containing {{name}} is mustache-expanded in stage 3
- The function signature accepts the shell setting, arguments, variable store, and built-in values
