# NTA-004: Follow-up action storage in action log + discovery query

> **Cluster:** norn-tools/action-log
> **Depends on:** NTA-001, NTA-002
> **Blocked by:** NTF-001
> **Checklist:**
> - C14 — follow_ups query returns follow-up actions matching the filter with expired actions filtered out at query time
> - C17 — Follow-up actions stored on action log entries with action_id, description, tool, args, expires, and confidence fields
> - C18 — Follow-up actions indexed by tool_call_id + action name for O(1) lookup
> - C19 — follow_ups query returns only unexpired follow-up actions — expired ones filtered out at query time
> - C20 — Expiry checking at query time compares file modification timestamp or content hash against stored value
> - C34 — Follow-up action expiry checked at query time, not maintained in real-time (CO2)
> **Stories:**

> - S6 (AI Agent, Discovering Follow-Up Actions) — As an AI agent, I want expired follow-up actions filtered out automatically so that I do not attempt actions whose preconditions have changed.
> - S7 (AI Agent, Discovering Follow-Up Actions) — As an AI agent, I want follow-up actions to include the original tool_call_id and action name so that I can pass them directly to the follow-up tool without re-specifying arguments.

## Purpose

Add follow-up action storage and discovery to the action log. When tools register follow-up actions (via NTF-002's lifecycle phase), the action log indexes them by tool_call_id + action name for O(1) lookup. The follow_ups query on the action_log tool returns unexpired follow-up actions, with expiry checked at query time. This enables agents to discover deferred options even after context compaction removes the original tool result.

## Task

Extend ActionLog to store Vec<FollowUpAction> per tool_call_id (already partially wired in NTA-001). Add a follow_ups index (HashMap<(String, String), usize> mapping (tool_call_id, action_name) to index into the Vec for O(1) lookup). Add the follow_ups query to the action_log tool that filters by tool name and outcome, checks expiry at query time, and returns only valid actions. Expiry checking delegates to the same hash-based logic used in the follow-up tool's expiry module.

## Requirements

### R1: Follow-up action indexing by tool_call_id + action name

ActionLog SHALL maintain an index that maps (tool_call_id, action_name) pairs to the corresponding FollowUpAction for O(1) lookup. When follow-ups are recorded via record_completion, each action's (tool_call_id, action_name) pair is inserted into the index. Duplicate action names for the same tool_call_id are rejected with a warning (should not happen in practice — tools register unique action names).

**Acceptance:**
- ActionLog.follow_up_index maps (tool_call_id, action_name) to FollowUpAction
- Lookup by (tool_call_id, action_name) returns Some(FollowUpAction) or None
- Duplicate (tool_call_id, action_name) logged as warning, first value kept
- Index is populated during record_completion

### R2: Follow-up action expiry checking at query time

When follow-up actions are queried (via the follow_ups query or via direct lookup for the follow_up tool), expiry SHALL be checked at query time. The expiry check reads the file, computes its content hash, and compares against the stored content_hash on the ExpiryCondition. Expired actions are filtered out of query results. The expiry checking logic SHALL reuse the same hash computation used in NTF-003's expiry module (or implement a compatible function that NTF-003 can later import).

**Acceptance:**
- follow_ups query returns only unexpired actions
- FileModified expiry checked by hashing file and comparing to stored hash
- AnyFileModified expiry checked by hashing all files
- TurnScoped expiry checked against current turn_id
- Never expiry always passes
- Expired actions excluded silently (no error — just filtered)

### R3: Add follow_ups query to the action_log tool

The action_log tool (NTA-002) SHALL accept query="follow_ups". The follow_ups query returns all unexpired follow-up actions matching the filter. The filter object's tool field filters by the tool that registered the follow-ups (not the target tool). The outcome field filters by the original tool call's outcome. Output structure: {"query": "follow_ups", "count": N, "actions": [{tool_call_id, action, description, tool, expires}, ...]}. Each action includes the tool_call_id so the agent can pass it directly to the follow_up tool.

**Acceptance:**
- action_log tool accepts query="follow_ups"
- follow_ups query returns unexpired actions with tool_call_id included
- filter.tool filters by the registering tool's name
- filter.outcome filters by the original call's outcome
- Expired actions excluded from results
- Output has query, count, actions fields
- Each action has tool_call_id, action, description, tool, expires

### R4: Direct lookup for follow_up tool consumption

ActionLog SHALL expose a get_follow_up(tool_call_id, action_name) method that returns Option<&FollowUpAction> using the O(1) index. This method does NOT check expiry — the caller (follow_up tool, NTF-003) handles expiry checking because it needs the full ExpiredError for its error response. The method also returns the original tool call arguments from original_args.

**Acceptance:**
- get_follow_up returns Some for known (tool_call_id, action_name)
- get_follow_up returns None for unknown combinations
- get_follow_up does NOT check expiry
- Original args accessible alongside the follow-up action
