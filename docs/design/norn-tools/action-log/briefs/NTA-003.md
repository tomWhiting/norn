## Purpose

Add a mutation ledger to the action log — a derived view that tracks every file modification during the session. The ledger answers "what files did I change?" without relying on git status, which shows pre-existing dirty files and changes from other agents in a shared working tree. External revert detection is lazy: when queried, each file is hashed and compared against the post-mutation hash. No filesystem watching, no polling.

## Task

Create crates/norn/src/session/mutation_ledger.rs with MutationLedgerEntry (file_path, operation, first/last tool_call_id, revert_status, diff_stats). The MutationLedger is built from ActionLog entries — each time a mutation tool completes, the ledger updates. Add the mutations query to the action_log tool (NTA-002) that returns ledger entries with lazy revert detection. The ledger is a derived view, not a separate store — it reads from ActionLog and the filesystem.

### R3: External revert detection via hash comparison at query time

When the mutation ledger is queried (not at mutation time), each entry's file SHALL be read and hashed. The hash is compared against the post-mutation hash stored when the last tool call touching that file completed. If the hash differs AND no subsequent tool call in the ActionLog touched that file, the entry's revert_status is set to ExternallyReverted. If a subsequent tool call DID touch the file, revert_status is SubsequentlyModified. If the hash matches, revert_status is Active. Hash computation uses the same algorithm as ExpiryCondition (SHA-256 of file content).

**Acceptance:**
- File unchanged since mutation: revert_status = Active
- File changed externally (no subsequent tool call): revert_status = ExternallyReverted
- File changed by a later tool call: revert_status = SubsequentlyModified
- File deleted since mutation: revert_status = ExternallyReverted
- Hash comparison only happens at query time, not at mutation time
- No filesystem watching or polling

### R4: Store post-mutation content hash on record_mutation

When record_mutation is called, the MutationLedger SHALL compute and store the content hash of the file as it exists after the mutation. For deletions, store a sentinel hash. This stored hash is the baseline for external revert detection at query time.

**Acceptance:**
- record_mutation computes SHA-256 hash of file content after mutation
- Hash stored on the MutationLedgerEntry
- Deleted files store a sentinel hash value
- Hash computation does not block the tool dispatch path (file is already on disk)
