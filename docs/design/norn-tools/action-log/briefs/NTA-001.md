> - S12 (Human Developer, Auditing Agent Activity) — As a developer, I want the action log stored in the existing EventStore with no new file formats so that existing tooling for inspecting session events works without modification.

## Purpose

Establish the action log as a query layer over the existing EventStore. Define the ActionLogEntry and Outcome types, implement Level 1 summary computation at tool-call completion, and store structured metadata on existing events. The action log retains full data regardless of context compaction — this is the key value proposition. No new storage backend; everything builds on EventStore and PersistenceSink.

## Task

Create crates/norn/src/session/action_log.rs with ActionLogEntry (Level 1 summary fields), Outcome enum, and an ActionLog struct that indexes tool-call events from the EventStore. The ActionLog maintains an in-memory index of ActionLogEntry values keyed by tool_call_id. Level 1 summaries are computed at tool-call completion and stored. Level 2 is the existing ToolResult event data (no duplication). Level 3 is populated lazily on first query (non-mutation tools) or eagerly via register_follow_ups (mutation tools — the FollowUpAction already carries before-content). Wire ActionLog into the tool dispatch path to capture entries as tools complete.

### R4: Level 1 summary computation at tool-call completion

ActionLog SHALL expose a record_completion method called after each tool dispatch: record_completion(tool_name, tool_call_id, tool_use_description, outcome, output, args, follow_ups). This method SHALL: (1) compute a one-line summary_line from the output (e.g., for edit: "edit committed: {path} +{added}/-{removed}", for error: "error: {first_line}"), (2) create an ActionLogEntry, (3) store it in the entries index, (4) store follow_ups if non-empty, (5) store original args. Summary generation SHALL be a match on tool_name with tool-specific formatters and a generic fallback.

**Acceptance:**
- record_completion creates an ActionLogEntry and stores it keyed by tool_call_id
- Summary line for edit shows path and diff stats
- Summary line for read shows path and line count
- Summary line for bash shows command (truncated to 80 chars) and exit code
- Generic fallback shows tool_name and outcome
- Follow-ups stored in follow_ups index
- Original args stored in original_args index

The tool output is retrieved from the EventStore by finding the ToolResult event matching the tool_call_id. Input arguments come from the original_args index. Follow-ups come from the follow_ups index.

**Acceptance:**
- get_detail returns Level 1 entry plus full output JSON, args, duration, follow-ups
- Output JSON comes from EventStore ToolResult event matching tool_call_id

### R6: Level 3 context retrieval with lazy population

ActionLog SHALL expose a get_context(tool_call_id) method that returns Level 3 data: Level 2 plus raw tool output before compaction, pre/post validate outcomes, and before/after file content. For mutation tools, before-content is sourced from the follow-up action's BeforeContentSource::StoredContent (captured eagerly at register_follow_ups time). For non-mutation tools, Level 3 data is the same as Level 2 (no file content to capture).

**Acceptance:**
- get_context returns Level 3 data for mutation tools including before-content from follow-up's StoredContent
- get_context for non-mutation tools returns same as get_detail (no file snapshots)
- Before-content is not stored separately — sourced from FollowUpAction.before_content field
- Returns None for unknown tool_call_id

### R8: Token-efficient Level 1 serialization

ActionLogEntry SHALL serialize to a compact JSON format suitable for embedding in tool results. A helper method compact_json(&self) -> serde_json::Value SHALL return a minimal representation: {tool, id, desc, ts, outcome, summary}. Short field names keep token count low. Scanning 100 entries via compact_json SHALL produce under 1000 tokens.

**Acceptance:**
- compact_json uses short field names: tool, id, desc, ts, outcome, summary
- 100 entries with typical values (tool=edit, desc=20 chars, summary=40 chars) produce under 1000 tokens estimated at 4 chars/token
- compact_json output is valid JSON parseable by serde_json
