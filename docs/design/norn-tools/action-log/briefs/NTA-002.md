# NTA-002: Action log query tool: list, detail, and context queries

> **Cluster:** norn-tools/action-log
> **Depends on:** NTA-001

> - C10 — filter object supports tool (tool name), outcome (success/error/blocked), file (affected file path), since (tool_call_id), and last (count) fields
> - C11 — list query returns Level 1 summaries matching the filter
> - C12 — detail query requires call_id parameter and returns Level 2 for that specific tool call
> - C13 — context query requires call_id parameter and returns Level 3 for that specific tool call
> - C16 — Invalid query values or missing required parameters (call_id for detail/context) return structured errors
> - C36 — action_log tool queries logged at Level 1 only — no Level 2/3 self-referential entries that create recursive noise (CO4)
> **Stories:**
> - S1 (AI Agent, Reviewing Prior Actions) — As an AI agent, I want to list all tool calls in the session with one-line outcome summaries so that I can quickly see what I have done without re-reading the full conversation.
> - S2 (AI Agent, Reviewing Prior Actions) — As an AI agent, I want to filter the action list by tool name and outcome so that I can find specific failures or edits without scanning every entry.
> - S3 (AI Agent, Reviewing Prior Actions) — As an AI agent, I want to drill into a specific tool call by ID to see the full structured result so that I can re-examine details that context compaction removed.
> - S4 (AI Agent, Reviewing Prior Actions) — As an AI agent, I want to retrieve Level 3 full context for a tool call including before/after file content so that I can debug a mutation that produced unexpected results.

## Purpose

Implement the action_log tool that gives the agent a queryable memory of its own actions. The tool supports three query modes: list (Level 1 summaries with filtering), detail (Level 2 for a specific call), and context (Level 3 for a specific call). The mutations and follow_ups query modes are added by NTA-003 and NTA-004 respectively. The tool output is compact by design — Level 1 list queries are cheap to call frequently.

## Task

Create crates/norn/src/tools/action_log.rs implementing the Tool trait. The tool accepts a query parameter (list, detail, context), an optional filter object, and an optional call_id. List queries return Level 1 compact summaries via ActionLog. Detail and context queries delegate to ActionLog::get_detail and get_context. The tool itself is logged at Level 1 only in the action log (no recursive noise). The tool is registered in the tool module.

### R2: Filter object for query scoping

The filter object SHALL support: tool (String — filter by tool name), outcome (String — "success", "error", or "blocked"), file (String — filter by affected file path, matched against summary_line), since (String — tool_call_id, return entries after this call), and last (u32 — return only the last N entries). All filter fields are optional. Multiple filters combine with AND logic. An empty or absent filter returns all entries.

**Acceptance:**
- filter.tool filters to entries matching the tool name
- filter.outcome filters to entries matching the outcome variant
- filter.file filters to entries whose summary_line contains the file path
- filter.since filters to entries recorded after the given tool_call_id
- filter.last limits results to the most recent N entries
- Multiple filters combine with AND (all must match)
- Absent filter returns all entries

The list query SHALL return an array of Level 1 compact summaries (via ActionLogEntry::compact_json) matching the filter. Results are ordered by insertion (chronological). The output JSON structure: {"query": "list", "count": N, "entries": [...]}.

- Each entry uses compact_json format (short field names)
- Results ordered chronologically
- Output has query, count, and entries fields

### R4: Detail query returning Level 2 data

The detail query SHALL require call_id and return Level 2 data from ActionLog::get_detail. If call_id is missing, return is_error with {"error": "call_id required for detail query"}. If call_id is not found, return is_error with {"error": "tool_call_id not found", "call_id": "..."}. Output structure: {"query": "detail", "entry": {Level 1 fields}, "output": {full tool output}, "args": {input arguments}, "duration_ms": N, "follow_ups": [...]}.

**Acceptance:**
- detail query requires call_id
- Missing call_id returns structured error
- Unknown call_id returns structured error
- Found entry returns Level 1 + output + args + duration + follow-ups
- Output is the full tool output JSON from EventStore

### R5: Context query returning Level 3 data

The context query SHALL require call_id and return Level 3 data from ActionLog::get_context. Same error handling as detail. Output structure extends detail with: before_content (HashMap of file paths to pre-mutation content, if available from follow-up's StoredContent) and post_validate_outcome (if recorded).

**Acceptance:**
- context query requires call_id
- Error handling identical to detail query
- Mutation tool entries include before_content from follow-up's StoredContent
- Non-mutation tool entries have before_content as null
- Output extends detail format with before_content and post_validate_outcome

### R6: Self-referential query constraint (Level 1 only)

When the action_log tool is itself called, its execution SHALL be recorded in the action log at Level 1 only. The tool dispatch path SHALL detect tool_name == "action_log" and set a flag that prevents Level 2/3 storage for this call. This prevents recursive noise — querying the log should not bloat the log with full query results.

**Acceptance:**
- action_log tool call appears in the action log with Level 1 summary
- Level 2 query for an action_log entry returns only the Level 1 fields (no full output stored)
- 100 action_log queries do not grow the action log's storage disproportionately
