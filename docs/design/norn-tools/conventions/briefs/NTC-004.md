## Purpose

Without an easy way to create a CONVENTIONS.toml, projects start with no post-mutation checks and users must write the TOML from scratch. The init command scans the project for file types and available tools, then generates a starter configuration with sensible per-language defaults. This makes adoption frictionless — run one command, get a working config, customize from there.

## Task

Add an Init { subcommand: InitCmd } variant to the CLI Command enum with an InitCmd::Conventions subcommand. Implement the conventions init handler in a new crates/norn-cli/src/commands/init/ module. The handler walks the project file tree to collect file extensions and counts, checks for compiled adapter availability per language, probes for external tools (mypy, ruff, eslint, shellcheck, biome, etc.) via which/where, generates convention groups with ecosystem-appropriate LOC limits and advise_on populated with available adapters (block_on empty by default), and writes a commented CONVENTIONS.toml. The command never overwrites an existing file.

### R2: Project file tree scanning

The init handler SHALL walk the project file tree from the current working directory, collecting file extensions and their counts. It SHALL skip hidden directories (starting with .), target/ directories, node_modules/ directories, and .git/ directories. The scan SHALL produce a map of extension to file count.

**Acceptance:**
- File tree walk starts from current working directory
- Hidden directories skipped
- target/ directory skipped
- node_modules/ directory skipped
- .git/ directory skipped
- Result is a map of extension (e.g. rs, ts, py) to file count
- Test: temp directory with 3 .rs files and 2 .ts files — map has rs:3, ts:2

### R3: Compiled adapter detection

For each detected file type, the init handler SHALL check which compiled adapters are available. The mapping SHALL be: rs → clippy, nextest; ts/tsx → biome; gleam → gleam_check, gleam_test. The file_length adapter is always available for all file types that tokei supports.

**Acceptance:**
- rs files map to clippy adapter
- ts/tsx files map to biome adapter
- gleam files map to gleam_check adapter
- file_length is always available
- Detection does not invoke external commands — it checks the compiled adapter list

### R4: External tool probing

The init handler SHALL probe for external tools using std::process::Command to check if they exist on PATH. Tools to probe: mypy, ruff (Python), eslint (TypeScript/JavaScript), shellcheck (shell), biome (TypeScript/JavaScript). WHEN a tool is found, it SHALL be included in the advise_on list for the matching language group. WHEN a tool is not found, it SHALL be omitted.

**Acceptance:**
- External tools probed via which or where equivalent (Command::new(tool).arg(--version).output())
- Found tools included in advise_on for matching language group
- Missing tools omitted silently — no warning or error
- Test: mock scenario where mypy is available — python group advise_on includes mypy

### R5: Convention group generation with ecosystem defaults

The init handler SHALL generate convention groups with language-appropriate LOC limits: Rust general 500, Rust entrypoints (mod.rs/lib.rs/main.rs) 200, TypeScript 400, Python 300, shell no LOC limit. All handling SHALL be advise. advise_on SHALL be populated with available adapters. block_on SHALL be empty. bypass_detection SHALL not be specified (uses the runtime default from NTC-002 R7).

**Acceptance:**
- Rust general group has LOC limit 500, handling advise
- Rust entrypoints group has LOC limit 200, handling advise, priority 10
- TypeScript group has LOC limit 400, handling advise
- Python group has LOC limit 300, handling advise
- Shell group has no LOC check
- All groups have advise_on populated, block_on empty
- Groups only generated for detected file types — no empty groups

### R6: CONVENTIONS.toml output with comments

The init handler SHALL write the generated CONVENTIONS.toml file. The output SHALL include TOML comments (# lines) explaining each field: tools, paths, loc.limit, loc.handling, advise_on, block_on, bypass_detection, adapters_dir, and priority. The file SHALL have a header comment explaining what CONVENTIONS.toml does.

**Acceptance:**
- Output file is valid TOML that parses with ConventionsConfig::load
- Header comment present explaining CONVENTIONS.toml purpose
- Each convention group has comments explaining its fields
- Comments use # prefix (TOML comment syntax)
- Test: generate a file, parse it back with ConventionsConfig::load — round-trips successfully

### R7: Never overwrite existing file

The init handler SHALL check if CONVENTIONS.toml already exists before writing. IF it exists, the handler SHALL NOT overwrite it. It SHALL print a message suggesting a different output path and exit without error. An --output flag SHALL allow writing to an alternate path.

**Acceptance:**
- Existing CONVENTIONS.toml is not overwritten
- Message printed: CONVENTIONS.toml already exists, use --output to write to a different path
- Exit code is success (not an error)
- --output <path> writes to the specified path instead of CONVENTIONS.toml
- Test: CONVENTIONS.toml exists — handler prints message and does not overwrite
- Test: --output alt.toml — writes to alt.toml even when CONVENTIONS.toml exists
