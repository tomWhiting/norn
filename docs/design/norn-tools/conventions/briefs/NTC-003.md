## Purpose

The advisory/blocking distinction is what separates CONVENTIONS.toml from the old all-or-nothing diagnostic system. Advisory checks provide quality feedback without rolling back edits — the model sees the feedback and can choose to act on it. Blocking checks trigger PostValidateOutcome::Fail and cause Gate rollback for edit/apply_patch. This brief changes the RuntimePostValidateCheck trait to return PostCheckResult (carrying both an outcome and advisories), updates the ToolRegistry execution loop to handle it, and wires the convention-driven adapter results into the correct pathway based on advise_on vs block_on.

## Task

Define PostCheckResult { outcome: PostValidateOutcome, advisories: Vec<Advisory> } and Advisory { message: String, source: String } in crates/norn/src/tool/lifecycle.rs. Change RuntimePostValidateCheck::check() to return PostCheckResult. Update ToolRegistry::execute_tool to collect advisories from PostCheckResult and inject them into the tool output under an advisories key. Advisories do not set is_error. Update DiagnosticsPostCheck to return advisory items for advise_on adapters and LOC checks with handling=advise, and errors for block_on adapters and LOC checks with handling=block. Add committed_output to ToolError::PostValidationFailed. Update the two existing test stubs (FailingPostCheck and PushingCheck) to return PostCheckResult.

## Requirements

PostCheckResult SHALL be a struct with fields: outcome (PostValidateOutcome), advisories (Vec<Advisory>). Advisory SHALL be a struct with fields: message (String), source (String). Both types SHALL be defined in crates/norn/src/tool/lifecycle.rs. PostCheckResult SHALL derive Clone and Debug. Advisory SHALL derive Clone, Debug, Serialize, and Deserialize.

**Acceptance:**
- PostCheckResult struct exists in lifecycle.rs with outcome and advisories fields
- Advisory struct exists in lifecycle.rs with message and source fields
- PostCheckResult derives Clone, Debug
- Advisory derives Clone, Debug, Serialize, Deserialize
- Both types are re-exported from crates/norn/src/tool/mod.rs

### R3: ToolRegistry advisory collection and injection

ToolRegistry::execute_tool SHALL collect advisories from all PostCheckResult values returned by runtime post_checks. Advisories SHALL be injected into the tool output JSON under an advisories key as an array of objects with message and source fields. Advisories SHALL NOT set is_error on the tool output. Advisories SHALL NOT cause PostValidateOutcome::Fail. The outcome field of PostCheckResult SHALL be processed exactly as the previous bare PostValidateOutcome was — errors collected, mode-dependent gate/report handling preserved.

**Acceptance:**
- Advisories from all post_checks are collected into a single Vec
- Non-empty advisories are injected into tool output under advisories key
- Advisory injection does not set is_error to true
- PostCheckResult.outcome is handled identically to the old PostValidateOutcome (errors collected, gate/report logic unchanged)
- Empty advisories Vec results in no advisories key on the output
- Test: post_check returns Pass with 2 advisories — output contains advisories array with 2 items, is_error is false
- Test: post_check returns Fail with 1 advisory — errors trigger gate/report handling, advisory still present in output

### R4: Convention-driven advisory vs blocking dispatch

DiagnosticsPostCheck SHALL route adapter results based on whether the adapter name appears in advise_on or block_on. Adapter results from advise_on SHALL produce Advisory items (source set to the adapter name). Adapter results from block_on SHALL produce error strings that trigger PostValidateOutcome::Fail. LOC checks with handling=Advise SHALL produce Advisory items. LOC checks with handling=Block SHALL produce error strings.

**Acceptance:**
- advise_on adapter results become Advisory { message, source: adapter_name }
- block_on adapter results become error strings in PostValidateOutcome::Fail
- LOC check with handling=Advise produces Advisory
- LOC check with handling=Block produces error string
- An adapter appearing in neither advise_on nor block_on is not dispatched
- Test: group with advise_on=[clippy], LOC advise — all results are advisories, outcome is Pass
- Test: group with block_on=[clippy], LOC block — results are errors, outcome is Fail

### R5: AST validation unconditional

The tool's own AST validation (tree-sitter parse check in the tool's post_validate method) SHALL remain unconditional and unaffected by CONVENTIONS.toml. AST validation is structural validity, not a quality convention. CONVENTIONS.toml controls quality checks only.

**Acceptance:**
- Tool's post_validate method is not modified by this brief
- A broken-AST edit still triggers PostValidateOutcome::Fail from the tool's own post_validate
- CONVENTIONS.toml with no groups matching the file does not suppress the tool's AST check
- Test: edit with broken syntax and no matching convention — AST check still fires

**Checklist:**
- C27 — Tool's own AST validation (tree-sitter parse check) remains unconditional and always blocks, unaffected by conventions

**Stories:**
- S8 (AI Agent, Experiencing Advisory vs Blocking Checks) — As an AI agent, I want the tool's own AST validation to remain unconditional so that structurally invalid edits are always caught regardless of convention configuration.

### R6: ToolError::PostValidationFailed committed_output

ToolError::PostValidationFailed SHALL gain an optional committed_output: Option<serde_json::Value> field. WHEN a blocking convention check fires after a successful tool execution (edit committed, then convention check fails), the committed output SHALL be preserved in this field so the model knows the edit landed even though diagnostics failed. The registry SHALL populate this field when raising PostValidationFailed in Gate mode.

**Acceptance:**
- ToolError::PostValidationFailed has committed_output: Option<serde_json::Value> field
- Gate-mode PostValidationFailed carries Some(output_value) when the tool executed successfully before post-checks failed
- Pre-validation failures (tool never executed) carry None
- All match arms on PostValidationFailed updated for the new field
- Test: gate-mode tool with passing execution but failing post-check — committed_output is Some
