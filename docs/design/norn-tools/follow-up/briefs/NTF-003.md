## Purpose

Implement the follow_up tool that executes deferred actions by referencing a prior tool_call_id and action name. This is the core execution path: look up the original call in the action log, find the named follow-up action, check expiry, merge argument overrides with the original arguments, and dispatch the target tool through the full lifecycle. The tool eliminates re-generation of large arguments (patch text, file content) by reading them from the action log.

## Task

Create crates/norn/src/tools/follow_up/ as a folder module with mod.rs (tool registration), lookup.rs (action log query), expiry.rs (expiry checking against current file state), merge.rs (argument override merging), and dispatch.rs (target tool invocation). The follow_up tool takes tool_call_id and action parameters. It validates the reference, checks expiry, merges arguments, and invokes the target tool through the registry's full lifecycle (pre-validate, execute, post-validate, on-success, register-follow-ups).

### R4: Expiry checking against current file state

The expiry module (crates/norn/src/tools/follow_up/expiry.rs) SHALL implement is_expired(condition: &ExpiryCondition, turn_id: &str) -> Result<(), ExpiredError>. FileModified: read the file, compute hash, compare against stored content_hash. AnyFileModified: check each file's hash. TurnScoped: compare against the current turn_id. Never: always Ok. ExpiredError SHALL carry the reason (which file changed or turn ended), the tool_call_id, the action name, and a suggestion string.

**Acceptance:**
- FileModified returns Err when file hash differs from stored hash
- FileModified returns Ok when file hash matches
- AnyFileModified returns Err naming the first file that changed
- TurnScoped returns Err when turn_id differs from stored turn_id
- Never always returns Ok
- Expired error includes reason, tool_call_id, action, and suggestion
- File read errors are treated as expired (file may have been deleted)

### R5: Argument override merging

The merge module (crates/norn/src/tools/follow_up/merge.rs) SHALL implement merge_args(original: &serde_json::Value, overrides: &serde_json::Value) -> serde_json::Value. Both inputs SHALL be JSON objects. The merge is shallow: each key in overrides replaces the corresponding key in original. Keys in original not present in overrides are preserved. Keys in overrides not present in original are added. The merge is deterministic — same inputs always produce same output.

**Acceptance:**
- Original {"a": 1, "b": 2} with overrides {"b": 3} produces {"a": 1, "b": 3}
- Original {"a": 1} with overrides {"b": 2} produces {"a": 1, "b": 2}
- Original {"a": 1, "b": 2} with overrides {} produces {"a": 1, "b": 2}
- Empty original with overrides {"a": 1} produces {"a": 1}
- Non-object inputs return an error (both must be JSON objects)

### R6: Target tool dispatch through full lifecycle

The dispatch module (crates/norn/src/tools/follow_up/dispatch.rs) SHALL invoke the target tool (named in the FollowUpAction's tool field) through the ToolRegistry's full lifecycle: pre-validate, execute, post-validate, on-success, register-follow-ups. The merged arguments are wrapped in a ToolEnvelope. The result is returned as if the target tool had been called directly. The follow_up tool itself does NOT produce follow-up actions (no recursive follow-ups). The target tool's result MAY produce new follow-ups attributed to the target tool.

**Acceptance:**

- Missing target tool returns is_error with structured message
- Target tool's full lifecycle runs (pre-validate through register-follow-ups)
- Result format matches direct target tool invocation
