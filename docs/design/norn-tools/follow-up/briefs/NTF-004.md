## Purpose

Wire follow-up registration into the mutation tools (edit, write, apply_patch) and the bash tool. Each mutation tool gains a register_follow_ups override that registers undo actions backed by stored before-content or Yggdrasil operation tracking. apply_patch also registers apply_structural, apply_auto, and apply_dry_run follow-ups on appropriate outcomes. edit registers apply_at_occurrence_N on ambiguous matches. bash registers rerun follow-ups. Yggdrasil operation tracking records each mutation as an operation for atomic multi-file undo.

## Task

Override register_follow_ups on EditTool, WriteTool, ApplyPatchTool, and BashTool. Each override inspects the ToolOutput to determine which follow-ups to offer. Mutation tools capture before-content eagerly during execute() and store it via BeforeContentSource::StoredContent. When Yggdrasil tracking is available (detected via libyggd on ToolContext), mutation tools additionally record operations and use BeforeContentSource::YggdrasilOp. Implement the undo execution path: follow_up tool dispatches undo, which reads BeforeContentSource and writes the stored content back (or calls libyggd::ops::revert_operation for YggdrasilOp).

## Requirements

### R1: Edit tool: undo follow-up registration

EditTool SHALL override register_follow_ups. On successful commit (output.content["committed"] == true), it SHALL register an undo FollowUpAction with: action="undo", description="Revert {path} to pre-edit content", tool="edit", expires=ExpiryCondition::FileModified with the edited file's path and post-edit content hash, before_content=BeforeContentSource::StoredContent with the pre-edit file content captured during execute(). The edit tool's execute() SHALL capture the file content before applying the edit and make it available for register_follow_ups.

**Acceptance:**
- Successful edit produces a follow-up with action="undo"
- Undo follow-up description includes the file path
- Undo follow-up expires when the file is modified after the edit
- before_content contains the pre-edit file content
- Uncommitted edits (output.content["committed"] != true) produce no undo follow-up

**Checklist:**

### R2: Edit tool: apply_at_occurrence follow-ups on ambiguous match

When an edit matches at multiple locations (output.content["match_count"] > 1), EditTool's register_follow_ups SHALL register apply_at_occurrence_N FollowUpActions for each match location. Each follow-up has: action="apply_at_occurrence_{N}" (1-indexed), description="Apply edit at occurrence {N} (line {line}, in {context})", tool="edit", args overriding occurrence to N, expires=ExpiryCondition::FileModified.

**Acceptance:**
- Multi-match edit produces N follow-ups (one per occurrence)
- Each follow-up has action name apply_at_occurrence_1, apply_at_occurrence_2, etc.
- Each follow-up's args override sets occurrence to the correct index
- Single-match edit produces no apply_at_occurrence follow-ups

### R3: Write tool: undo follow-up registration

WriteTool SHALL override register_follow_ups. On successful write, it SHALL register an undo FollowUpAction. If the file existed before the write, before_content=BeforeContentSource::StoredContent with the original content. If the file was newly created, before_content stores empty content and undo deletes the file. Expires with FileModified on the written file.

**Acceptance:**
- Successful write produces a follow-up with action="undo"
- Overwrite of existing file stores the original content in before_content
- Creation of new file stores empty content (undo will delete)
- Undo follow-up expires when the file is modified after the write

### R4: ApplyPatch tool: undo follow-ups (all-or-nothing and per-file)

ApplyPatchTool SHALL override register_follow_ups. On successful patch application, it SHALL register: (1) an all-or-nothing undo with action="undo", expires=AnyFileModified over all patched files; (2) per-file undo follow-ups with action="undo_file:{path}" for each file in the patch, each expiring with FileModified on that file. The all-or-nothing undo uses YggdrasilOp when available, otherwise StoredContent. Per-file undo always uses StoredContent (not operation revert, which is all-or-nothing).

**Acceptance:**
- Successful multi-file patch produces 1 all-or-nothing undo + N per-file undos
- All-or-nothing undo expires when any patched file is modified
- Per-file undo expires only when its specific file is modified
- All-or-nothing undo uses YggdrasilOp when available, StoredContent otherwise
- Per-file undo always uses StoredContent

### R5: ApplyPatch tool: apply_structural, apply_auto, apply_dry_run follow-ups

On strict-mode patch failure (output indicates strict matching failed but structural/auto would work), register_follow_ups SHALL register: apply_structural (overrides mode to "structural") and apply_auto (overrides mode to "auto"). On successful dry-run (output.content["dry_run"] == true), register apply_dry_run with overrides {"dry_run": false}. All expire with FileModified on the target file(s).

**Acceptance:**
- Strict failure produces apply_structural and apply_auto follow-ups
- apply_structural overrides {"mode": "structural"}
- apply_auto overrides {"mode": "auto"}
- Successful dry-run produces apply_dry_run with overrides {"dry_run": false}
- Non-strict failure (already auto mode) does not produce apply_structural

### R6: Bash tool: rerun and rerun_with_timeout follow-ups

BashTool SHALL override register_follow_ups. On any completed execution, it SHALL register: rerun (re-execute the same command) and rerun_with_timeout (re-execute with a longer timeout, overriding timeout_ms to 2x the original). Both expire with ExpiryCondition::Never (commands can always be re-run).

**Acceptance:**
- Completed bash execution produces rerun and rerun_with_timeout follow-ups
- rerun has empty overrides (same args as original)
- rerun_with_timeout overrides timeout_ms to 2x the original value
- Both use ExpiryCondition::Never

### R7: Yggdrasil operation tracking for mutation tools

Mutation tools (edit, write, apply_patch) SHALL record their changes as Yggdrasil operations when libyggd is available. Each tool call that modifies files creates an operation entry via the libyggd operation log with: operation_id mapping to tool_call_id, files modified with before/after content hashes, timestamp, and agent identity from ToolContext. The operation_id is stored in BeforeContentSource::YggdrasilOp for the undo follow-up. When libyggd is not available (detected at ToolContext construction), skip operation recording and use StoredContent only.

**Acceptance:**
- When libyggd is available, successful mutation creates a Yggdrasil operation
- Operation ID maps to the tool_call_id
- Operation records before/after content hashes for modified files
- Undo follow-up uses YggdrasilOp when operation was recorded
- When libyggd is unavailable, undo follow-up uses StoredContent
- No panic or error when libyggd is unavailable — graceful fallback

### R8: Follow-ups visible in tool results

All registered follow-ups SHALL appear in the tool result's follow_ups array via the model_facing_json serialization (wired by NTF-002's registry integration). The model sees action, description, and expires for each follow-up. The follow_ups key is present only when follow-ups were registered.

**Acceptance:**
- Successful edit result contains follow_ups array with undo entry
- Strict-failed patch result contains follow_ups with apply_structural and apply_auto
- Bash result contains follow_ups with rerun and rerun_with_timeout
- Each follow_up entry has exactly action, description, expires keys

- SHALL NOT implement the tool_follow_ups! macro — that is NTM-004
- SHALL NOT modify the Tool trait or lifecycle dispatch — that is NTF-002
- SHALL NOT implement search or read tool follow-ups — D7 lists them as possibilities but they are not in scope for this brief
