# NTF-001: Follow-up action types: FollowUpAction, ExpiryCondition, BeforeContentSource

> **Cluster:** norn-tools/follow-up
> **Checklist:**
> - C1 — FollowUpAction struct in norn::tool::follow_up with action (name), description, tool (target tool name), args (pre-populated argument overrides), expires (ExpiryCondition), confidence, and before_content (BeforeContentSource) fields
> - C2 — ExpiryCondition enum in norn::tool::follow_up with FileModified(PathBuf), AnyFileModified(Vec<PathBuf>), TurnScoped, and Never variants
> - C3 — BeforeContentSource enum in norn::tool::follow_up with YggdrasilOp(OperationId), StoredContent(HashMap<PathBuf, String>), and Unavailable variants
> - C4 — FollowUpAction serializes to JSON matching the follow_ups array format in tool results (action, description, expires fields visible to model)
> **Stories:**
> - S10 (Tool Author, Registering Follow-Up Actions) — As a tool author, I want to register follow-up actions in a register_follow_ups lifecycle phase so that my tool can offer deferred options without mixing follow-up logic into execute().
> - S11 (Tool Author, Registering Follow-Up Actions) — As a tool author, I want the register_follow_ups phase to be optional with a default empty implementation so that existing tools require no changes.

## Purpose

Define the foundational types for the follow-up system in norn::tool::follow_up. These types are the vocabulary that every subsequent brief in both the follow-up and action-log clusters depends on. FollowUpAction describes a deferred action the agent can take. ExpiryCondition describes when an action becomes invalid. BeforeContentSource describes where undo content is stored. The macros cluster (NTM-004) generates code that constructs these types — it depends on this brief.

## Task

Create crates/norn/src/tool/follow_up.rs with three types: FollowUpAction (the serializable follow-up action descriptor), ExpiryCondition (an enum of expiry conditions), and BeforeContentSource (an enum of undo content sources). Add the follow_up module to crates/norn/src/tool/mod.rs with appropriate re-exports. All types must derive Clone, Debug, Serialize, Deserialize. FollowUpAction must serialize to the JSON shape that tool results expose to the model.

## Requirements

### R1: Define FollowUpAction struct

FollowUpAction SHALL be defined in crates/norn/src/tool/follow_up.rs with fields: action (String — the short action name like "undo" or "apply_structural"), description (String — human-readable description for the model), tool (String — name of the target tool to invoke), args (serde_json::Value — pre-populated argument overrides to merge with the original call's args), expires (ExpiryCondition — when this action becomes invalid), confidence (String — "high", "medium", or "low"), and before_content (BeforeContentSource — where undo content is sourced from). FollowUpAction SHALL derive Clone, Debug, Serialize, Deserialize.

**Acceptance:**
- crates/norn/src/tool/follow_up.rs exists
- FollowUpAction has fields: action: String, description: String, tool: String, args: serde_json::Value, expires: ExpiryCondition, confidence: String, before_content: BeforeContentSource
- FollowUpAction derives Clone, Debug, Serialize, Deserialize
- FollowUpAction is pub and accessible via norn::tool::follow_up::FollowUpAction
- serde round-trip test passes for FollowUpAction with all fields populated

### R2: Define ExpiryCondition enum

ExpiryCondition SHALL be defined in crates/norn/src/tool/follow_up.rs with variants: FileModified { path: PathBuf, content_hash: String } (expires when the file's content hash no longer matches), AnyFileModified { paths: Vec<PathBuf>, content_hash_map: HashMap<PathBuf, String> } (expires when any file's hash no longer matches), TurnScoped { turn_id: String } (expires at end of the named turn), and Never (does not expire). ExpiryCondition SHALL derive Clone, Debug, Serialize, Deserialize. The content_hash fields store the hash recorded at registration time — expiry checking compares against the file's current hash.

**Acceptance:**
- ExpiryCondition has exactly 4 variants: FileModified, AnyFileModified, TurnScoped, Never
- FileModified carries path: PathBuf and content_hash: String
- AnyFileModified carries paths: Vec<PathBuf> and content_hash_map: HashMap<PathBuf, String>
- TurnScoped carries turn_id: String
- Never has no fields
- ExpiryCondition derives Clone, Debug, Serialize, Deserialize
- serde round-trip test passes for all 4 variants

**Files:**
- modify: crates/norn/src/tool/follow_up.rs

**Checklist:**
- C2 — ExpiryCondition enum in norn::tool::follow_up with FileModified(PathBuf), AnyFileModified(Vec<PathBuf>), TurnScoped, and Never variants

### R3: Define BeforeContentSource enum

BeforeContentSource SHALL be defined in crates/norn/src/tool/follow_up.rs with variants: YggdrasilOp { operation_id: String } (undo via libyggd operation revert), StoredContent { files: HashMap<PathBuf, String> } (undo by writing stored content directly), and Unavailable (no undo source — returns clear error on attempt). BeforeContentSource SHALL derive Clone, Debug, Serialize, Deserialize. Default SHALL be Unavailable.

**Acceptance:**
- BeforeContentSource has exactly 3 variants: YggdrasilOp, StoredContent, Unavailable
- YggdrasilOp carries operation_id: String
- StoredContent carries files: HashMap<PathBuf, String>
- Unavailable has no fields
- BeforeContentSource derives Clone, Debug, Serialize, Deserialize
- BeforeContentSource implements Default returning Unavailable
- serde round-trip test passes for all 3 variants

### R4: Model-facing serialization shape for tool results

FollowUpAction SHALL serialize to JSON matching the follow_ups array format visible to the model. The model-facing fields are: action, description, and expires. The args, confidence, before_content, and tool fields are internal — they SHALL be serialized for storage but the tool result builder (future briefs) is responsible for filtering to the model-facing subset. A helper method model_facing_json(&self) -> serde_json::Value SHALL return only {action, description, expires} for embedding in tool results.

**Acceptance:**
- FollowUpAction has a pub fn model_facing_json(&self) -> serde_json::Value method
- model_facing_json returns a JSON object with exactly 3 keys: action, description, expires
- Full serde serialization of FollowUpAction includes all fields (action, description, tool, args, expires, confidence, before_content)
- model_facing_json output for a FileModified expiry includes the path but not the content_hash (the model does not need implementation details)
