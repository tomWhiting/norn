# NTF-002: register_follow_ups lifecycle phase on the Tool trait

> **Cluster:** norn-tools/follow-up
> **Depends on:** NTF-001
> **Checklist:**
> - C35 — Tool trait gains a register_follow_ups method as a new lifecycle phase after on-success
> - C36 — register_follow_ups also runs after error for failure-recovery follow-ups (e.g., apply_structural on strict failure)
> - C37 — register_follow_ups returns Vec<FollowUpAction>
> - C38 — Registry attaches returned follow-ups to the tool result under a follow_ups key
> - C39 — Phase is optional — default implementation returns empty Vec, tools without follow-ups are unchanged
> **Stories:**
> - S10 (Tool Author, Registering Follow-Up Actions) — As a tool author, I want to register follow-up actions in a register_follow_ups lifecycle phase so that my tool can offer deferred options without mixing follow-up logic into execute().
> - S11 (Tool Author, Registering Follow-Up Actions) — As a tool author, I want the register_follow_ups phase to be optional with a default empty implementation so that existing tools require no changes.
> - S12 (Tool Author, Registering Follow-Up Actions) — As a tool author, I want to register follow-ups on both success and error outcomes so that failure-recovery actions (like switching from strict to structural mode) are available.

## Purpose

Extend the tool lifecycle with a register_follow_ups phase that runs after on-success (and after error for failure-recovery follow-ups). This is the mechanism by which tools declare what follow-up actions are available on their result. The phase is optional — tools that do not override it return an empty Vec. The registry attaches returned follow-ups to the tool result under a follow_ups key.

## Task

Add a register_follow_ups method to the Tool trait in traits.rs with a default implementation returning empty Vec<FollowUpAction>. Modify the lifecycle dispatch in registry.rs to call register_follow_ups after on_success (on success path) and after execute error handling (on error path for failure-recovery follow-ups). The registry SHALL attach the returned Vec<FollowUpAction> to the tool output by injecting a follow_ups array (using model_facing_json) into the output content. When the follow-ups Vec is empty, no follow_ups key is added.

## Requirements

### R1: Add register_follow_ups to the Tool trait

The Tool trait in crates/norn/src/tool/traits.rs SHALL gain a new async method: async fn register_follow_ups(&self, output: &ToolOutput, ctx: &ToolContext) -> Vec<FollowUpAction>. The default implementation SHALL return Vec::new(). The method receives the tool output and context so it can inspect the result to decide which follow-ups to offer.

**Acceptance:**
- Tool trait in traits.rs declares register_follow_ups with the specified signature
- Default implementation returns Vec::new()
- Existing tools that do not override register_follow_ups continue to compile without changes
- The method is async and takes &self, &ToolOutput, &ToolContext
- cargo check -p norn succeeds

### R2: Call register_follow_ups on the success path in registry dispatch

The ToolExecutor impl in registry.rs SHALL call tool.register_follow_ups(&output, &ctx) after the on_success phase completes. The returned Vec<FollowUpAction> SHALL be attached to the tool output: if non-empty, inject a follow_ups array into output.content (which is a serde_json::Value) containing the model_facing_json() of each FollowUpAction. If the Vec is empty, output.content is unchanged.

**Acceptance:**
- register_follow_ups is called after on_success in the lifecycle dispatch
- A tool returning 2 FollowUpActions has output.content["follow_ups"] with 2 entries
- Each entry in follow_ups has exactly action, description, expires keys (from model_facing_json)
- A tool returning empty Vec has no follow_ups key in output.content
- The full Vec<FollowUpAction> is retained internally for action log storage (not discarded after serialization)

### R3: Call register_follow_ups on the error path for failure-recovery follow-ups

When a tool's execute() returns an error, or when post-validate fails, the registry SHALL still call register_follow_ups if the error is a domain error (not a system/infrastructure error). Specifically: after PostValidateOutcome::Fail in Gate mode, before returning ToolError::PostValidationFailed, the registry SHALL call register_follow_ups with the output that was produced (the output exists — it was committed before post-validate ran). The returned follow-ups SHALL be stored for action log indexing. For execute-phase errors (ToolError), register_follow_ups is NOT called — there is no output to inspect.

**Acceptance:**
- Gate-mode post-validate failure still calls register_follow_ups with the committed output
- Follow-ups from failure path are stored for action log indexing (retained on the internal result, not discarded)
- Execute-phase errors do NOT trigger register_follow_ups
- Pre-validate blocks do NOT trigger register_follow_ups

### R4: Expose follow-ups alongside dispatch result for action log consumption

The ToolExecutor dispatch method SHALL return the Vec<FollowUpAction> alongside the ToolOutput. The return type changes from Result<ToolOutput, ToolError> to Result<(ToolOutput, Vec<FollowUpAction>), ToolError>. Callers that do not need follow-ups can destructure and discard. For PostValidationFailed errors, the follow-ups SHALL be accessible via a new committed_follow_ups field on ToolError::PostValidationFailed (or a separate side-channel). The action log (future brief NTA-001) indexes these.

**Acceptance:**
- ToolExecutor::execute returns Result<(ToolOutput, Vec<FollowUpAction>), ToolError>
- All callers of ToolExecutor::execute updated to handle the new return type
- Follow-ups from the failure path are not silently dropped
- cargo check -p norn succeeds with all callers updated
