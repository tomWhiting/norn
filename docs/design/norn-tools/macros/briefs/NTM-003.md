## Purpose

Complete the ToolArgs derive macro by adding full serde attribute integration (rename, rename_all, flatten), the #[tool_args(...)] escape-hatch attributes for manual schema overrides, and compile-time validation that enforces doc comments on all schema-visible fields. After this brief, the derive macro handles every pattern needed for the existing tool codebase.

## Task

Extend tool_args/parse.rs to extract serde rename, rename_all, skip_serializing, and flatten attributes. Extend tool_args/schema.rs to apply rename transformations and flatten nested struct properties. Create tool_args/validate.rs for compile-time checks. Implement #[tool_args(schema = {...})], #[tool_args(description = "...")], #[tool_args(skip)], #[tool_args(required)], and #[tool_args(additional_properties)] override attributes. Add compile_error! for fields missing doc comments.

## Requirements

### R1: Parse serde rename attributes

The parser SHALL extract #[serde(rename = "x")] on fields and #[serde(rename_all = "...")] on structs. The rename_all value SHALL be parsed as a case-transformation rule supporting: lowercase, UPPERCASE, camelCase, snake_case, PascalCase, SCREAMING_SNAKE_CASE, kebab-case, SCREAMING-KEBAB-CASE.

**Acceptance:**
- #[serde(rename = "custom_name")] on a field records "custom_name" as the schema property name
- #[serde(rename_all = "camelCase")] on a struct records camelCase as the container-level rename rule
- All 8 rename_all case transformations are supported
- An unrecognized rename_all value produces compile_error!

### R2: Apply rename transformations in schema generation

WHEN a field has #[serde(rename = "x")], the schema property name SHALL be "x" instead of the Rust field name. WHEN the struct has #[serde(rename_all = "...")], all field names SHALL be transformed according to the case rule. Field-level rename SHALL take precedence over struct-level rename_all.

**Acceptance:**
- A struct with #[serde(rename_all = "camelCase")] and field my_field produces property "myField"
- A field with #[serde(rename = "custom")] produces property "custom" regardless of rename_all
- The required array uses the renamed names, not the Rust field names
- A struct with rename_all = "snake_case" and field MyField produces property "my_field"

### R3: Handle skip_serializing and skip_deserializing distinction

#[serde(skip_serializing)] on a field SHALL keep the field in the schema because it still deserializes. #[serde(skip)] and #[serde(skip_deserializing)] SHALL omit the field from the schema entirely. This distinction SHALL be correct: skip_serializing fields accept model input, skip_deserializing fields do not.

**Acceptance:**
- A field with #[serde(skip_serializing)] appears in the schema properties and may appear in required
- A field with #[serde(skip)] does not appear in properties
- A field with #[serde(skip_deserializing)] does not appear in properties
- A field with both #[serde(skip_serializing)] and #[serde(skip_deserializing)] does not appear in properties

### R4: Implement serde flatten support

WHEN a field has #[serde(flatten)], the macro SHALL merge the flattened struct's properties into the parent schema instead of nesting them. The flattened struct's required fields SHALL be added to the parent's required array. IF #[serde(flatten)] is applied to a non-struct type, the macro SHALL emit compile_error!.

**Acceptance:**
- A struct with a #[serde(flatten)] field of type Inner merges Inner's properties into the parent properties
- Inner's required fields appear in the parent's required array
- The flattened field itself does not appear as a named property
- #[serde(flatten)] on a field of type String produces compile_error! mentioning that flatten requires a struct type
- The error span points to the flatten attribute

### R5: Implement #[tool_args(...)] override attributes

The parser SHALL recognize #[tool_args(schema = {...})], #[tool_args(description = "...")], #[tool_args(skip)], #[tool_args(required)], and #[tool_args(additional_properties)] attributes on fields. schema replaces the generated schema entirely. description overrides the doc-comment description. skip excludes from schema without affecting deserialization. required forces inclusion in required even with #[serde(default)]. additional_properties sets additionalProperties: true on the field's object schema.

**Acceptance:**
- #[tool_args(schema = {"type": "string", "enum": ["strict", "auto"]})] produces the literal JSON as the field schema
- #[tool_args(description = "Custom desc")] overrides the doc comment
- #[tool_args(skip)] excludes the field from properties without adding #[serde(skip)]
- #[tool_args(required)] on a #[serde(default)] field puts the field back in required
- #[tool_args(additional_properties)] on a struct field adds "additionalProperties": true to that field's schema

### R6: Compile-time doc comment enforcement

IF a field has no doc comment AND no #[tool_args(skip)] AND no #[serde(skip)] AND no #[serde(skip_deserializing)], THEN the macro SHALL emit compile_error!("ToolArgs: field `{name}` has no doc comment. Add a /// comment to provide the schema description."). The error SHALL point to the field via syn::Error::new_spanned.

**Acceptance:**
- A field without a doc comment and without skip produces a compile error naming the field
- A field with #[tool_args(skip)] but no doc comment does not produce an error
- A field with #[serde(skip)] but no doc comment does not produce an error
- A field with #[tool_args(description = "...")] but no doc comment does not produce an error
- The error message includes the field name
- The error span highlights the field, not the struct or derive

### R7: Verify derive does not interfere with Deserialize

The ToolArgs derive SHALL NOT modify the struct definition, add attributes, or conflict with #[derive(Deserialize)]. Both derives SHALL be applicable to the same struct independently. The ToolArgs derive reads serde attributes but does not write them.

**Acceptance:**
- A struct with #[derive(Deserialize, ToolArgs)] compiles without errors
- Deserializing a JSON object into the struct works identically with and without ToolArgs
- A struct with #[derive(ToolArgs)] only (no Deserialize) also compiles
- The generated json_schema() method does not conflict with any serde-generated methods

**Checklist:**

## Boundaries

- SHALL NOT implement the tool_follow_ups! macro — that is NTM-004
- SHALL NOT modify any existing tool implementations — migration is NTM-005
- SHALL NOT implement custom deserialize_with schema inference — fields with custom deserializers use their declared Rust type for schema, or #[tool_args(schema = {...})] for override
- SHALL NOT support serde(with = "...") for schema generation — the macro uses the field's declared type
