## Purpose

Establish the norn-macros proc-macro crate as a workspace member and implement the core ToolArgs derive macro that generates json_schema() from struct definitions with String fields. This is the foundation every subsequent macro brief depends on — the crate must compile, the derive must produce valid JSON Schema objects, and the norn crate must re-export the macro for tool authors to use.

## Task

Create the norn-macros crate with proc-macro = true, add it as a dependency of norn, implement the #[derive(ToolArgs)] proc macro that generates a json_schema() -> serde_json::Value inherent method. This brief covers only String and bool field types — complex types (Option, Vec, enums) are NTM-002. The derive must: extract doc comments as descriptions, build the properties/required/additionalProperties structure, respect field declaration order, and handle #[serde(default)] to omit fields from required. The norn crate re-exports ToolArgs and tool_follow_ups (as a no-op stub re-export until NTM-004 fills it in).

### R4: Parse struct fields, doc comments, and basic serde attributes

The tool_args/parse.rs module SHALL parse a DeriveInput into a structured representation: struct name, fields with name/type/doc-comment/serde-attributes. For each field, it SHALL extract: the field ident, the type path, all /// doc comment lines joined with spaces, and the presence of #[serde(default)] or #[serde(default = "fn")]. Fields with #[serde(skip)] or #[serde(skip_deserializing)] SHALL be marked as skipped.

**Acceptance:**
- crates/norn-macros/src/tool_args/parse.rs exists with a parse function that accepts syn::DeriveInput
- Doc comments (syn::Attribute with path doc) are extracted and joined with spaces
- Multiple /// lines produce a single space-joined description string
- #[serde(default)] is detected and flags the field as having a default
- #[serde(default = "some_fn")] is detected identically to #[serde(default)]
- #[serde(skip)] and #[serde(skip_deserializing)] flag the field as skipped

### R5: Generate JSON Schema for struct with String and bool fields

The tool_args/schema.rs module SHALL generate a json_schema() inherent method that returns a serde_json::Value. The generated schema SHALL have type: object, a properties object with one entry per non-skipped field, a required array containing fields that are not #[serde(default)] and not Option<T>, and additionalProperties: false. String fields SHALL generate {"type": "string"}. bool fields SHALL generate {"type": "boolean"}. Property order in the generated JSON SHALL match field declaration order.

**Acceptance:**
- A struct with fields path: String, content: String generates a schema with properties.path.type == "string" and properties.content.type == "string"
- Both fields appear in the required array
- The schema includes "additionalProperties": false
- A field with #[serde(default)] is absent from the required array
- A field with #[serde(skip)] does not appear in properties at all
- A bool field generates {"type": "boolean"}
- Properties appear in the same order as struct field declarations
- Doc comments appear as description values on the corresponding properties

## Boundaries

- SHALL NOT implement type mappings beyond String and bool — complex types (integers, floats, Option, Vec, HashMap, Value, nested structs, enums) are NTM-002
- SHALL NOT implement serde rename, rename_all, or flatten attribute handling — those are NTM-003
- SHALL NOT implement #[tool_args(...)] override attributes — those are NTM-003
- SHALL NOT implement compile-time doc comment enforcement (compile_error on missing docs) — that is NTM-003
- SHALL NOT implement the real tool_follow_ups macro — only the no-op placeholder; NTM-004 replaces it
- SHALL NOT modify any existing tool implementations — migration is NTM-005
