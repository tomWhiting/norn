## Purpose

Extend the ToolArgs derive macro to handle all Rust types that appear in tool args structs: integers, floats, Option<T>, Vec<T>, HashMap<String, T>, serde_json::Value, nested structs, and enums (string, tagged, untagged, adjacently tagged). After this brief, the macro can generate correct JSON Schema for every type used across the existing 17+ tools.

## Task

Add type-to-JSON-Schema mapping logic in tool_args/schema.rs and create tool_args/enum_schema.rs for enum handling. The type mapping is closed — unsupported types produce compile_error! with a message suggesting #[tool_args(schema = {...})]. Enum support covers four serde representations: unit-variant-only (string enum), internally tagged, externally tagged/untagged (oneOf), and adjacently tagged. Nested structs with #[derive(ToolArgs)] generate inline object schemas via recursive json_schema() calls.

## Requirements

### R1: Integer type mapping

Signed integer types (i8, i16, i32, i64, isize) SHALL generate {"type": "integer"}. Unsigned integer types (u8, u16, u32, u64, usize) SHALL generate {"type": "integer", "minimum": 0}.

**Acceptance:**
- A field of type i64 generates {"type": "integer"} with no minimum
- A field of type u64 generates {"type": "integer", "minimum": 0}
- A field of type u32 generates {"type": "integer", "minimum": 0}
- All signed types (i8, i16, i32, i64, isize) produce identical schema
- All unsigned types (u8, u16, u32, u64, usize) produce identical schema with minimum: 0

**Files:**

**Checklist:**

### R3: Option<T> handling

Fields of type Option<T> SHALL generate the inner type T's schema and SHALL omit the field from the required array. Option<T> SHALL NOT produce a nullable schema — absent means None.

**Acceptance:**
- A field offset: Option<u64> generates {"type": "integer", "minimum": 0} and is absent from required
- A field name: Option<String> generates {"type": "string"} and is absent from required
- Option<Option<T>> produces the same schema as Option<T> (double-unwrap)

### R4: Vec<T> and HashMap<String, T> mapping

Vec<T> fields SHALL generate {"type": "array", "items": <schema of T>}. HashMap<String, T> fields SHALL generate {"type": "object", "additionalProperties": <schema of T>}. HashMap keys that are not String SHALL produce compile_error!.

**Acceptance:**
- A field tags: Vec<String> generates {"type": "array", "items": {"type": "string"}}
- A field counts: Vec<u64> generates {"type": "array", "items": {"type": "integer", "minimum": 0}}
- A field metadata: HashMap<String, String> generates {"type": "object", "additionalProperties": {"type": "string"}}
- A field with HashMap<i32, String> produces a compile error mentioning String key requirement

### R5: serde_json::Value escape hatch

### R6: Nested struct support

Fields whose type is a struct with #[derive(ToolArgs)] SHALL generate an inline object schema by calling T::json_schema() in the generated code. The generated code SHALL emit T::json_schema() as a function call, not attempt to introspect the nested struct at macro expansion time.

**Acceptance:**
- A struct Inner with #[derive(ToolArgs)] used as a field type in Outer generates Inner::json_schema() call in the expanded code
- The resulting schema nests Inner's properties inside Outer's properties
- Nested structs respect their own required/additionalProperties rules
- cargo check succeeds when both Outer and Inner have #[derive(ToolArgs)]

### R7: Unsupported type compile error

IF a field's type does not match any entry in the type mapping table AND the field has no #[tool_args(schema = {...})] override, THEN the macro SHALL emit compile_error! with a message naming the unsupported type and suggesting the override syntax.

**Acceptance:**
- A field of type PathBuf produces compile_error!("ToolArgs: unsupported type `PathBuf`. Use #[tool_args(schema = {...})] to provide a manual schema.")
- The error span points to the field, not the derive invocation
- The error message includes the exact type name from the source

### R8: String enum schema generation

WHEN ToolArgs is derived on an enum with only unit variants, the macro SHALL generate {"type": "string", "enum": [<variant names>]}. Variant names SHALL respect #[serde(rename_all)] on the enum and #[serde(rename)] on individual variants. Doc comments on variants SHALL be collected into a top-level description listing each option.

**Acceptance:**
- An enum with variants A, B, C generates {"type": "string", "enum": ["A", "B", "C"]}
- An enum with #[serde(rename_all = "snake_case")] and variants MyVariant generates enum: ["my_variant"]
- An enum with #[serde(rename = "custom")] on a variant uses "custom" in the enum array
- Doc comments on variants produce a description like "A: docs for A. B: docs for B."

**Files:**
- create: crates/norn-macros/src/tool_args/enum_schema.rs

**Checklist:**
- C27 — Unit-variant-only enums generate {"type": "string", "enum": [...]} with variant names

- C29 — #[serde(rename_all)] on enums transforms variant names in the enum array
- C30 — #[serde(rename)] on individual variants overrides the variant name in the enum array

### R9: Tagged and untagged enum schema generation

WHEN ToolArgs is derived on an enum with #[serde(tag = "...")], the macro SHALL generate a oneOf schema where each variant is an object with the tag field as a const value. WHEN #[serde(untagged)] is used, the macro SHALL generate oneOf without a discriminator. WHEN #[serde(tag = "t", content = "c")] is used (adjacent tagging), the macro SHALL generate a two-field object per variant with the tag as const and the content wrapping the variant data.

**Acceptance:**
- A tagged enum with #[serde(tag = "type")] and variants Create{title: String} and Update{id: String} generates oneOf with two objects, each having a const type field
- An untagged enum generates oneOf without any discriminator const fields
- An adjacently tagged enum with tag = "t", content = "c" generates objects with t as const and c wrapping the variant data
- Enum variants with named fields generate object schemas with properties matching the field names

## Boundaries

- SHALL NOT implement #[serde(rename)] or #[serde(rename_all)] on struct fields — field-level rename is NTM-003 (enum variant rename IS in scope for this brief since it's part of enum schema generation)
- SHALL NOT implement #[serde(flatten)] — that is NTM-003
- SHALL NOT implement #[tool_args(...)] override attributes — those are NTM-003
- SHALL NOT implement compile-time doc comment enforcement — that is NTM-003
- SHALL NOT modify any existing tool implementations — migration is NTM-005
