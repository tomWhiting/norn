# NTI-002: PostCheckResult type with advisory pathway

> **Cluster:** norn-tools
> **Depends on:** NTI-001

## Purpose

RuntimePostValidateCheck currently returns bare PostValidateOutcome (Pass or Fail). This means every diagnostic finding is either a hard failure (Gate blocks, discards output) or silent (Report appends to output but nothing blocks). There is no middle ground — no way for a check to say 'this is worth noting but should not block the tool.' This brief introduces PostCheckResult which separates the pass/fail outcome from advisory-level findings, enabling the conventions system (NTC-003) to report advisory findings without blocking tool execution.

## Task

Create a PostCheckResult struct that wraps PostValidateOutcome plus a Vec<Advisory>. Create the Advisory struct with severity, message, and source fields. Change RuntimePostValidateCheck::check() to return PostCheckResult. Update both execution pipelines (registry.rs and infra.rs) to collect advisories from PostCheckResult and inject them into tool output regardless of pass/fail outcome. Update all three implementors: DiagnosticsPostCheck (production), FailingPostCheck (test in registry.rs), PushingCheck (test in integration.rs).

## Requirements

### R1: Define Advisory and PostCheckResult types

Advisory SHALL be a struct in norn::tool::lifecycle with fields: severity (AdvisorySeverity enum: Info, Warning), message (String), and source (String identifying the check that produced it). PostCheckResult SHALL be a struct with fields: outcome (PostValidateOutcome) and advisories (Vec<Advisory>). PostCheckResult SHALL implement a convenience constructor pass() that returns Pass with empty advisories, and fail(errors) that returns Fail with empty advisories, for use by simple checks that don't produce advisories.

**Acceptance:**
- Advisory struct exists in crates/norn/src/tool/lifecycle.rs with severity, message, source fields
- AdvisorySeverity enum exists with Info and Warning variants
- PostCheckResult struct exists with outcome and advisories fields
- PostCheckResult::pass() returns PostCheckResult { outcome: Pass, advisories: vec![] }
- PostCheckResult::fail(errors) returns PostCheckResult { outcome: Fail { errors }, advisories: vec![] }
- All types derive Clone, Debug, Serialize, Deserialize

**Files:**
- modify: crates/norn/src/tool/lifecycle.rs

### R2: Change RuntimePostValidateCheck::check() return type

RuntimePostValidateCheck::check() SHALL return PostCheckResult instead of PostValidateOutcome. This is a breaking change to the trait — all implementors must be updated in the same brief.

**Acceptance:**
- RuntimePostValidateCheck::check() signature returns PostCheckResult
- The trait remains object-safe (no generic parameters, no Self return)
- cargo check -p norn produces only expected errors from unupdated implementors

**Files:**
- modify: crates/norn/src/tool/lifecycle.rs

### R3: Update execution pipelines to collect and inject advisories

WHEN ToolRegistry::execute (registry.rs) and SubAgentExecutor::execute (infra.rs) call post-validate checks, they SHALL destructure PostCheckResult into outcome and advisories. Advisories SHALL be collected from all checks (both the tool's own post_validate and runtime checks). WHEN advisories are non-empty, they SHALL be injected into output.content as an "advisories" array regardless of whether the outcome is Pass or Fail. Advisory injection SHALL happen before the Gate-mode error check so that NTI-001's committed_output includes the advisories.

**Acceptance:**
- registry.rs collects advisories from tool post_validate and all runtime checks
- infra.rs collects advisories from tool post_validate and all runtime checks
- Advisories are serialized into output.content["advisories"] as a JSON array
- Advisory injection occurs before the PostValidationFailed error construction
- An empty advisories vec does not add an advisories key to output.content

**Files:**
- modify: crates/norn/src/tool/registry.rs
- modify: crates/norn/src/tools/agent/infra.rs

### R4: Update DiagnosticsPostCheck implementor

DiagnosticsPostCheck::check() SHALL return PostCheckResult. For this brief, the implementation SHALL use PostCheckResult::pass() and PostCheckResult::fail(errors) — no advisories are produced yet. Advisory production is the conventions system's job (NTC-003). This change is mechanical: wrap the existing return values.

**Acceptance:**
- DiagnosticsPostCheck::check() returns PostCheckResult
- Existing Pass returns PostCheckResult::pass()
- Existing Fail returns PostCheckResult::fail(errors)
- No advisories produced (empty vec)
- Behavior is identical to pre-change

**Files:**
- modify: crates/norn/src/tools/diagnostics_check.rs

### R5: Update test implementors and assertions

FailingPostCheck (registry.rs tests) and PushingCheck (integration.rs tests) SHALL be updated to return PostCheckResult. All test assertions that inspect post-validate outcomes SHALL be updated for the new return type. Tests SHALL verify that advisories produced by checks appear in the tool output's advisories array.

**Acceptance:**
- FailingPostCheck::check() returns PostCheckResult::fail(errors)
- PushingCheck::check() returns PostCheckResult::pass()
- At least one test verifies advisories appear in output.content when produced
- All existing tests pass with the new return type
- cargo test -p norn passes all tests

**Files:**
- modify: crates/norn/src/tool/registry.rs
- modify: crates/norn/src/tests/integration.rs

## Boundaries

- SHALL NOT produce advisories from DiagnosticsPostCheck — advisory production is NTC-003 scope
- SHALL NOT change the Tool trait's post_validate method signature — only RuntimePostValidateCheck changes
- SHALL NOT add advisory severity levels beyond Info and Warning — Error severity is what PostValidateOutcome::Fail already represents
- SHALL NOT modify how PostValidateOutcome::Fail errors are collected — only add advisory collection alongside
