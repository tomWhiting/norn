# NTP-002: Reorder resolution tiers to entity-first

> **Cluster:** norn-tools/apply-patch
> **Checklist:**
> - C1 — Hunk resolution tries entity-guided placement first (tier 1), then context-anchored search (tier 2), then header correction + diffy (tier 3)
> - C2 — Tier 1 reads the @@ semantic anchor text (after the second @@) and uses it to locate the target entity
> - C3 — Tier 1 scopes context search to the matched entity's byte range when entity is found
> - C4 — Tier 2 full-file context-anchored search runs exact match, then whitespace-insensitive, then trim-insensitive
> - C5 — Tier 3 header correction + diffy is the last resort, not the first attempt
> - C6 — Resolution tier used is reported per-hunk in the tool result
> **Stories:**
> - S1 (AI Agent, Applying Code Changes) — As an AI agent applying a patch, I want the tool to resolve hunks using structural understanding of the code so that my patches apply correctly even when my line numbers are stale.
> - S4 (AI Agent, Applying Code Changes) — As an AI agent, I want the tool result to tell me which resolution tier was used for each hunk so that I can calibrate my patch accuracy over time.
> - S14 (Human Developer, Reviewing Agent Patches) — As a developer reviewing agent output, I want to see per-hunk resolution details so that I can understand how each change was placed and whether structural matching was needed.

## Purpose

The current three-tier resolution applies hunks in the weakest-signal-first order: header correction + diffy (line numbers), then context-anchored search, then entity-guided placement. This brief inverts the order so the strongest signal (structural entity matching) is tried first, context search is the fallback, and line-number-based diffy is the last resort. This improves patch reliability for model-generated diffs where line numbers are almost always stale but @@ semantic anchors are almost always correct.

## Task

Restructure apply_unified_tiered in patch_apply.rs to try entity-guided placement first when a semantic anchor exists, then full-file context search, then header correction + diffy. Add per-hunk resolution metadata to the return type so callers can report which tier was used, what entity was matched, and how much line drift occurred. Keep the existing entity-finding code (SupportedLanguage, find_entity_range, collect_named_containers) — NTP-003 replaces it with libyggd. This brief reorders the tiers; NTP-003 replaces the tier 1 implementation.

## Requirements

### R1: Define HunkResolution metadata type

A HunkResolution struct SHALL capture per-hunk resolution details: tier_used (1, 2, or 3), entity_matched (Option<String> — the entity name from the @@ anchor), matched_at_line (Option<usize>), stated_line (usize — from the @@ header), drift (i64 — difference between matched_at_line and stated_line), and confidence (High, Medium, Low based on tier and match quality). The struct SHALL be in patch_apply.rs.

**Acceptance:**
- HunkResolution struct exists with tier_used, entity_matched, matched_at_line, stated_line, drift, confidence fields
- Confidence enum has High (entity match), Medium (context match), Low (diffy fallback) variants
- HunkResolution derives Debug, Clone, Serialize

### R2: Extract semantic anchor from @@ header

A parse_semantic_anchor function SHALL extract the text after the second @@ in a unified diff hunk header. For example, from '@@ -10,5 +10,6 @@ fn process_event' it SHALL return Some("fn process_event"). If there is no text after the second @@, it SHALL return None. Leading/trailing whitespace SHALL be trimmed.

**Acceptance:**
- parse_semantic_anchor('@@ -10,5 +10,6 @@ fn process_event') returns Some('fn process_event')
- parse_semantic_anchor('@@ -1,3 +1,4 @@') returns None
- parse_semantic_anchor('@@ -1,3 +1,4 @@  ') returns None (whitespace-only treated as empty)
- parse_semantic_anchor('@@ -1,3 +1,4 @@ impl Display for Foo') returns Some('impl Display for Foo')

### R3: Implement entity-first tier ordering

apply_unified_tiered SHALL be restructured so that for each hunk: (1) IF a semantic anchor exists AND find_entity_range succeeds, scope the context search to that entity's line range; (2) IF no semantic anchor or entity not found, run full-file context-anchored search (exact, whitespace-insensitive, trim-insensitive); (3) IF context search fails, fall back to header correction + diffy. The function signature SHALL change to return Vec<HunkResolution> alongside the applied content. The existing find_entity_range and collect_named_containers functions SHALL continue to be used — NTP-003 replaces them.

**Acceptance:**
- A hunk with a valid @@ semantic anchor matching an entity in the file is resolved at tier 1 (entity-guided)
- A hunk with no semantic anchor is resolved at tier 2 (context search) or tier 3 (diffy)
- A hunk where entity is found but context does not match within entity range falls through to tier 2
- A hunk where context search fails falls through to tier 3 (diffy)
- The return type includes Vec<HunkResolution> with one entry per hunk

### R4: Context search with three fallback levels

Tier 2 context-anchored search SHALL try three matching strategies in order: exact match (current find_context_match), whitespace-insensitive match (collapse all runs of whitespace to single space before comparison), and trim-insensitive match (trim each line before comparison). Each strategy SHALL search the full file (or entity range if scoped by tier 1). The first successful match wins.

**Acceptance:**
- Exact match succeeds when context lines are byte-identical
- Whitespace-insensitive match succeeds when context differs only in whitespace quantity
- Trim-insensitive match succeeds when context has leading/trailing whitespace differences
- Strategies are tried in order: exact first, then whitespace-insensitive, then trim-insensitive
- Match confidence is Medium for exact, Medium for whitespace-insensitive, Low for trim-insensitive

### R5: Surface resolution metadata in tool output

patch.rs execute SHALL include the per-hunk HunkResolution data in the tool output JSON under a resolution_details key. Each entry SHALL include the file path, hunk index, tier used, entity matched, line drift, and confidence. This data is informational — it does not affect pass/fail.

**Acceptance:**
- Tool output JSON contains a resolution_details array
- Each entry has file, hunk_index, tier_used, entity_matched, matched_at_line, stated_line, drift, confidence
- A single-hunk single-file patch produces one resolution_details entry
- A multi-file patch produces one entry per hunk across all files
