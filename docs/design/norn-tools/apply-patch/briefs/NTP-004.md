## Purpose

Models need to preview patch impact before committing — blast radius, diagnostics, resolution details — without modifying any files on disk. Dry-run mode runs the exact same resolution and validation code path as actual application but skips the disk-write step, ensuring that a successful dry-run guarantees successful application. Results are stored in the action log for reference and can be applied via follow-up without re-specifying the patch.

## Task

Add a dry_run boolean parameter to the apply_patch input schema. When dry_run is true, execute runs the full resolution pipeline (entity matching, context search, AST validation, diagnostics preview) but does not write any files to disk. The tool output includes per-file resolution details, AST validation results, blast radius, file length impact, and diagnostics preview. committed is always false in dry-run mode. The code path SHALL be identical to the non-dry-run path up to the point of disk writes — there is no separate dry-run pipeline. Dry-run results are stored in the action log.

## Requirements

### R1: Add dry_run parameter to input schema and PatchArgs

The apply_patch input schema SHALL include a dry_run boolean parameter with default false. PatchArgs SHALL gain a #[serde(default)] dry_run: bool field. The parameter description SHALL state that dry-run previews the patch result without writing to disk.

**Acceptance:**
- input_schema() JSON includes dry_run property with type boolean and default false
- PatchArgs has dry_run: bool field with #[serde(default)]
- Omitting dry_run from the tool call arguments defaults to false (backward compatible)

### R2: Skip disk writes when dry_run is true

WHEN dry_run is true, execute SHALL run the full resolution and staging pipeline (read files, apply hunks, run AST validation, collect diagnostics) but SHALL NOT write staged files to disk, SHALL NOT call tokio::fs::write, SHALL NOT create parent directories, and SHALL NOT delete files. The tool output SHALL have committed: false. All resolution metadata and diagnostics SHALL still be computed and returned.

**Acceptance:**
- Dry-run does not modify any files on disk
- Dry-run does not create any directories
- Dry-run output has committed: false
- Dry-run output contains the same resolution_details as a real application would
- Dry-run output contains diagnostics array (if any would have been produced)

### R3: Include blast radius in dry-run output

Dry-run output SHALL include a blast_radius object with: per-file entries (lines_added, lines_removed, lines_modified, containing_symbols list), and totals (total_files, total_added, total_removed). containing_symbols SHALL list the names of entities (functions, structs, etc.) that contain the changed lines — derived from entity extraction if available, empty list if not. File length impact SHALL be included as file_length_before and file_length_after per file.

**Acceptance:**
- Dry-run output contains blast_radius.files array with per-file metrics
- Each file entry has lines_added, lines_removed, containing_symbols
- Each file entry has file_length_before and file_length_after (line counts)
- blast_radius.totals has total_files, total_added, total_removed
- containing_symbols is populated when entity extractor is available
- containing_symbols is empty array when entity extractor is not available

### R4: Include AST validation and diagnostics preview

Dry-run output SHALL include AST validation results per file (pass or fail with error details). Dry-run output SHALL include a diagnostics_preview array showing what DiagnosticsPostCheck would report if the patch were applied — the same diagnostics that would run in non-dry-run mode. The diagnostics preview SHALL run the same check_single_file logic on the staged content (in memory) without writing to disk first.

**Acceptance:**
- Dry-run output contains ast_validation array with per-file pass/fail
- AST failures include error position and message
- Dry-run output contains diagnostics_preview array
- Diagnostics preview produces the same output as post-validation would on a real apply
- Diagnostics that require the file to be on disk (e.g. clippy) report what they can from static analysis
