# NTP-005: Strict and structural resolution modes

> **Cluster:** norn-tools/apply-patch
> **Depends on:** NTP-002
> **Checklist:**
> - C35 — mode parameter added to apply_patch input schema with values auto, strict, structural
> - C36 — auto mode (default) uses entity-first resolution with full tier fallback
> - C37 — strict mode only applies when context lines match exactly at the stated line position
> - C38 — strict mode reports what structural matching would have found when exact matching fails
> - C39 — strict mode structural_alternative includes would_apply, entity, entity_range, matched_at_line, drift, confidence
> - C40 — structural mode skips diffy entirely and requires entity resolution for every hunk
> - C41 — Failed strict-mode result includes a follow-up action ID for applying with structural matching
> **Stories:**
> - S6 (AI Agent, Verifying Patch Accuracy) — As an AI agent verifying my own accuracy, I want to apply a patch in strict mode so that I can confirm my line numbers are correct before committing.
> - S7 (AI Agent, Verifying Patch Accuracy) — As an AI agent whose strict-mode patch failed, I want to see what structural matching would have done so that I can either accept the structural match or regenerate with correct line numbers.

## Purpose

Models need a way to verify their own patch accuracy (strict mode: did I get the line numbers right?) and to force structural resolution (structural mode: always use entity matching, never fall back to line numbers). Strict mode is a confidence tool — when it fails but structural would have succeeded, the result tells the model exactly what structural matching found, including the drift and a follow-up action to apply with structural matching. This feedback loop lets models calibrate their accuracy over time.

## Task

Add a mode parameter to the apply_patch input schema with three values: auto (default, current behavior after NTP-002 reorder), strict (exact line-position match only), and structural (entity resolution required for every hunk, no diffy fallback). When strict mode fails, check whether structural matching would have succeeded and include a structural_alternative object in the result. Include a follow-up action ID for applying the patch with structural matching without re-generating it.

## Requirements

### R1: Add mode parameter to input schema and PatchArgs

The apply_patch input schema SHALL include a mode string parameter with enum values ["auto", "strict", "structural"] and default "auto". PatchArgs SHALL gain a #[serde(default)] mode: PatchMode field. PatchMode SHALL be an enum with Auto, Strict, Structural variants, with Auto as the Default impl. The parameter description SHALL explain each mode's purpose.

**Acceptance:**
- input_schema() JSON includes mode property with enum constraint and default auto
- PatchMode enum exists with Auto, Strict, Structural variants
- PatchMode implements Default returning Auto
- Omitting mode from arguments defaults to Auto (backward compatible)

**Files:**
- modify: crates/norn/src/tools/patch.rs

**Checklist:**

### R2: Implement auto mode as the reordered default

WHEN mode is Auto, apply_unified_tiered SHALL use the entity-first resolution order established by NTP-002: tier 1 (entity-guided), tier 2 (context search), tier 3 (diffy). This is the existing behavior after NTP-002 lands. No changes needed beyond passing the mode through.

**Acceptance:**
- Auto mode behavior is identical to the NTP-002 tier ordering
- Auto mode is the default when mode parameter is omitted

**Files:**
- modify: crates/norn/src/tools/patch_apply.rs

**Checklist:**
- C36 — auto mode (default) uses entity-first resolution with full tier fallback

### R3: Implement strict mode with exact position matching

WHEN mode is Strict, apply_unified_tiered SHALL only apply a hunk when context lines match exactly at the stated line position (the @@ header's line numbers). It SHALL NOT fall back to context search or entity-guided placement. If exact matching fails, the hunk is NOT applied and the result includes applied: false for that hunk. The tool continues processing remaining hunks (does not abort the entire patch on first strict failure).

**Acceptance:**
- Strict mode applies a hunk only when context matches at the exact stated line
- Strict mode does not fall back to context search
- Strict mode does not fall back to entity-guided placement
- A strict-mode failure for one hunk does not abort the entire patch
- HunkResolution for a failed strict hunk has tier_used: 0 (none) and applied: false

**Files:**
- modify: crates/norn/src/tools/patch_apply.rs

**Checklist:**

### R4: Report structural alternative when strict fails

WHEN strict mode fails for a hunk, the system SHALL check whether entity-guided or context-search resolution would have succeeded. IF structural matching would have succeeded, the result SHALL include a structural_alternative object with: would_apply (bool), entity (matched entity name), entity_range (line range), matched_at_line (where context was found), drift (signed offset from stated line), and confidence (High/Medium/Low). IF structural matching would also have failed, structural_alternative SHALL be null.

**Acceptance:**
- Failed strict-mode hunk with a viable structural match includes structural_alternative object
- structural_alternative.would_apply is true when entity or context match succeeded
- structural_alternative.drift is the signed difference between matched_at_line and stated_line
- structural_alternative.entity is the entity name from the @@ anchor
- Failed strict-mode hunk with no viable structural match has structural_alternative: null

### R5: Implement structural mode

WHEN mode is Structural, apply_unified_tiered SHALL require entity resolution for every hunk. It SHALL NOT fall back to diffy (tier 3). If a hunk has no semantic anchor or the entity is not found, the hunk fails. Context search (tier 2) is still available within the entity's scoped range. The tool continues processing remaining hunks on failure (same as strict mode).

**Acceptance:**
- Structural mode requires entity resolution — hunks without semantic anchors fail
- Structural mode does not fall back to diffy
- Structural mode allows context search within the entity's scoped range
- A hunk with a valid entity anchor and context match within the entity succeeds
- A hunk without a semantic anchor fails with a clear error message

**Files:**
- modify: crates/norn/src/tools/patch_apply.rs

**Checklist:**

### R6: Follow-up action for strict-to-structural upgrade

WHEN a strict-mode patch fails with a viable structural_alternative, the result SHALL include a follow_up_id referencing a follow-up action that re-applies the patch with mode: auto (which will use structural matching). The follow-up action SHALL reuse the same patch text — no re-generation needed. This depends on the follow-up types from NTF-001 being available; if they are not yet landed, this requirement SHALL register the follow-up data in the tool output under a pending_follow_up key that can be wired once NTF-001 lands.

**Acceptance:**
- Failed strict-mode result with structural_alternative includes follow_up_id or pending_follow_up
- The follow-up references the same patch text with mode changed to auto
- The model can invoke the follow-up without re-specifying the patch content
- If NTF-001 is not yet landed, pending_follow_up contains the data needed to create the follow-up later
