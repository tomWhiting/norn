## Purpose

Tool calls are the primary activity of an agent session. Without per-tool rendering, all tool output is unstructured text. This brief defines the ToolRenderer trait and implements the Tier 1 (rich) renderers for tools that produce substantial, structured output: Bash, Edit (including rollback and override cases), ApplyPatch, Search, and Read.

## Task

Define the ToolRenderer trait in tools/mod.rs with header_line() and body() methods. Implement Tier 1 renderers in tools/rich.rs: Bash (streaming ANSI passthrough, exit code, duration), Edit (unified diff with red/green, blast-radius, AST BLOCKED rollback case, COMMITTED override case), ApplyPatch (diff hunks), Search (file:line:content with highlighted matches), Read (path + line range, body hidden by default). Each renderer produces a compact header line that doubles as the collapsed view.

## Requirements

### R1: ToolRenderer trait

The ToolRenderer trait SHALL define the interface for all tool renderers with header_line() for the compact summary and body() for the expanded output.

**Acceptance:**
- ToolRenderer trait with: header_line(&self, tool_output: &Value, caps: &TerminalCaps) -> String
- ToolRenderer trait with: body(&self, tool_output: &Value, caps: &TerminalCaps) -> Option<String>
- body() returns None for tools that have no expanded view
- renderer_for(tool_name: &str) -> Option<Box<dyn ToolRenderer>> factory function
- Test: renderer_for('bash') returns Some, renderer_for('unknown') returns None

### R2: Bash renderer

The Bash renderer SHALL show the command with a $ prefix, stream output with ANSI passthrough, and display exit code and duration on completion.

**Acceptance:**
- Header: '$ {command_preview}' truncated to terminal width, with exit code and duration on completion
- Body: full command output with ANSI escape sequences passed through (subprocess colours preserved)
- While running: header shows a spinner character
- On completion: header shows exit code (0 or non-zero) and duration in seconds
- Non-zero exit codes render with error styling
- Test: header_line for a completed bash call includes exit code and duration

### R3: Edit renderer — success case

WHEN an edit succeeds (committed: true, no AST errors), THE SYSTEM SHALL render a unified diff with red/green colouring and blast-radius symbols.

**Acceptance:**
- Header: '~ {path}' with duration
- Body: unified diff with removed lines in red (- prefix) and added lines in green (+ prefix)
- Blast-radius section shows containing symbols from the tool output's blast_radius field
- Test: header_line for a successful edit includes the file path

**Files:**

**Checklist:**

### R4: Edit renderer — AST rollback case

WHEN an edit is rolled back due to AST validation failure (kind: 'edit_blocked_by_ast', committed: false), THE SYSTEM SHALL render the header with '✗' prefix and 'AST BLOCKED (not committed)' AND the body SHALL show diagnostic errors, NOT the diff.

**Acceptance:**
- Header: '✗ ~ {path}  AST BLOCKED (not committed)'
- Body: diagnostic error messages from the tool output's diagnostics array
- The diff is NOT shown — showing a diff of uncommitted changes would be misleading
- Test: header_line for a rolled-back edit contains 'AST BLOCKED'

### R5: Edit renderer — AST override case

WHEN an edit is committed despite AST errors via AllowBrokenAst (committed: true, check_overrides non-empty), THE SYSTEM SHALL render the header with '⚠' prefix and show the override source attribution.

**Acceptance:**
- Header: '⚠ ~ {path}  COMMITTED (AST override: {source})'
- Source attribution comes from the first CheckOverride entry's source field
- Body: unified diff (same as success case) plus diagnostic warnings below the diff
- Test: header_line for an overridden edit contains the override source

- Test: header_line includes the file path
