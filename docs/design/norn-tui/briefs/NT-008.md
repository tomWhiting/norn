# NT-008: Session event rendering and structured output

> **Cluster:** norn-tui
> **Depends on:** NT-003, NT-006
> **Checklist:**
> - C66 — Assistant messages render as markdown through the D6 rendering pipeline
> - C67 — Thinking content renders with ANSI dim attribute, collapsed by default
> - C68 — Ctrl+E toggles thinking visibility and secondary schema fields for future output
> - C69 — Tool calls render through per-tool renderers (D7) with appropriate visual treatment per tool type
> - C70 — User messages render with a distinct prefix separating them from assistant output
> - C71 — Structured assistant messages with multiple schema fields render the primary field by default with secondary fields toggleable
> - C72 — Secondary schema fields render with labeled separators between sections
> - C73 — New event types with EventSchemaSet schemas render automatically via the labeled-section pattern
> **Stories:**
> - S17 (Developer, Interacting with Structured Output) — As a developer using event schemas, I want structured assistant messages to show the primary field by default with secondary fields toggleable so that the conversation is not cluttered with alternate renderings.
> - S18 (Developer, Interacting with Structured Output) — As a developer receiving a Question event, I want to see numbered options and reply inline so that I can answer the agent's question without switching contexts.
> - S19 (Developer, Interacting with Structured Output) — As a developer using spoken responses, I want the TTS-optimised text available as a secondary field so that I can review what would be spoken without enabling TTS.

## Purpose

The TUI needs to render four categories of session events (assistant messages, thinking, tool calls, user messages) and handle structured output with multiple schema fields. This brief is the dispatch layer that routes events to the correct renderer and handles the per-event schema system.

## Task

Implement session event rendering in events/schema_render.rs. Route assistant messages through the NT-003 markdown pipeline. Route tool calls through the NT-006/NT-007 per-tool renderers via the ToolRenderer dispatch. Render thinking content with ANSI dim (collapsed by default). Render user messages with a distinct prefix. Implement the Ctrl+E toggle for thinking visibility and secondary schema fields. Handle structured assistant messages with multiple fields: primary field renders by default, secondary fields render with labeled separators when toggled. Support automatic rendering of new event types via the labeled-section pattern.

Thinking content streams live via ThinkingDelta for real-time rendering. After the turn completes, the accumulated thinking text is persisted on the AssistantMessage.thinking field in the EventStore and is available for replay (NT-010).

## Requirements

### R1: Event rendering dispatch

The event renderer SHALL route each session event type to its appropriate rendering pipeline: assistant messages to markdown, tool calls to per-tool renderers, user messages to prefix renderer.

**Acceptance:**
- render_event(event, caps, verbosity, thinking_visible) dispatches to the correct renderer
- SessionEvent::AssistantMessage → markdown renderer (NT-003)
- SessionEvent::ToolResult → per-tool renderer via renderer_for (NT-006/NT-007)
- SessionEvent::UserMessage → prefix renderer (R5)
- Test: render_event for an AssistantMessage produces markdown-styled output

**Files:**
- create: crates/norn-tui/src/events/schema_render.rs

**Checklist:**
- C66 — Assistant messages render as markdown through the D6 rendering pipeline
- C69 — Tool calls render through per-tool renderers (D7) with appropriate visual treatment per tool type

### R2: Thinking content rendering

ThinkingDelta events SHALL render with ANSI dim attribute in the scroll region, collapsed by default. Thinking content streams live via ThinkingDelta for real-time rendering. After the turn completes, the accumulated thinking text is persisted on the AssistantMessage.thinking field in the EventStore and is available for replay.

**Acceptance:**
- ThinkingDelta text renders with dim SGR attribute when thinking_visible is true
- When thinking_visible is false (default), ThinkingDelta text is silently discarded — not written to scroll region
- A dim prefix (e.g. 'thinking: ') distinguishes thinking content from assistant text
- After the turn completes, accumulated thinking text is available via AssistantMessage.thinking in the EventStore
- During EventStore replay (NT-010), thinking content is read from AssistantMessage.thinking and rendered with the same dim treatment, controlled by the thinking_visible toggle
- Test: ThinkingDelta with thinking_visible=false produces no scroll region output

**Files:**
- modify: crates/norn-tui/src/events/schema_render.rs

**Checklist:**
- C67 — Thinking content renders with ANSI dim attribute, collapsed by default

### R3: Ctrl+E toggle for thinking and secondary fields

WHEN the user presses Ctrl+E, THE SYSTEM SHALL toggle thinking visibility AND secondary schema field visibility for future output.

**Acceptance:**
- DisplayToggles struct holds: thinking_visible (bool, default false), secondary_fields_visible (bool, default false)
- Ctrl+E flips both booleans simultaneously
- Future ThinkingDelta events respect the new thinking_visible value
- Future structured messages respect the new secondary_fields_visible value
- A brief status message ('thinking: on, details: on' / 'thinking: off, details: off') appears momentarily
- Test: toggling from default (false, false) produces (true, true)

**Files:**
- modify: crates/norn-tui/src/events/schema_render.rs

**Checklist:**
- C68 — Ctrl+E toggles thinking visibility and secondary schema fields for future output

### R4: Structured assistant messages with multiple schema fields

WHEN an assistant message conforms to a schema with multiple fields, THE SYSTEM SHALL render the primary field by default AND render secondary fields with labeled separators when secondary_fields_visible is true.

**Acceptance:**
- Primary field: the first field in the schema (or a field named 'text', 'content', 'written', or 'response') renders by default
- Secondary fields: all other fields render below the primary, each preceded by a labeled separator '─── {field_name} ───'
- When secondary_fields_visible is false, secondary fields are not rendered
- When secondary_fields_visible is true, all fields render in schema-definition order
- Test: a 2-field schema with secondary_fields_visible=false renders only the primary field

### R5: User message rendering

User messages SHALL render with a distinct prefix separating them from assistant output.

**Acceptance:**
- User messages render with a coloured '> ' prefix (e.g. blue or cyan)
- Multi-line user messages have the prefix on the first line only; continuation lines are indented
- User messages are visually distinct from assistant messages (which have no prefix)
- Test: user message 'hello' renders with '> hello'

**Files:**
- modify: crates/norn-tui/src/events/schema_render.rs

**Checklist:**
- C70 — User messages render with a distinct prefix separating them from assistant output

### R6: Schema-driven rendering for new event types

WHEN a new event type has a schema in the EventSchemaSet, THE SYSTEM SHALL render its content using the labeled-section pattern without requiring TUI code changes.

**Acceptance:**
- render_structured(value, schema, caps, secondary_visible) renders any JSON value against its schema
- Object fields render as labeled sections with separators
- Primary field heuristic: 'text', 'content', 'written', 'response', or first string-typed field
- Non-object values render as formatted JSON
- Test: a schema with fields 'summary' and 'detail' renders 'summary' as primary, 'detail' as secondary

**Files:**
- modify: crates/norn-tui/src/events/schema_render.rs

**Checklist:**
- C73 — New event types with EventSchemaSet schemas render automatically via the labeled-section pattern
