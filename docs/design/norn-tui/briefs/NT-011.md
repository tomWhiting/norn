# NT-011: Event loop and streaming integration

> **Cluster:** norn-tui
> **Depends on:** NT-003, NT-004, NT-008, NT-009
> **Checklist:**
> - C11 — TextDelta events write styled markdown output into the scroll region
> - C12 — ThinkingDelta events render with ANSI dim attribute in the scroll region
> - C13 — ToolCallDelta events accumulate per tool-call ID and render through the per-tool renderer on completion
> - C14 — Done events clear the streaming indicator and show usage summary in the fixed panel
> - C15 — Error events render with a distinct error style in the scroll region
> - C16 — Streaming indicator in the fixed panel shows generating status with elapsed time while the model is producing output
> **Stories:**
> - S5 (Developer, Interactive Agent Conversation) — As a developer, I want to see a streaming indicator while the model is generating so that I know the agent is working even when I have scrolled back.

## Purpose

This is the integration brief that connects all TUI subsystems into a working application. It implements the main tokio::select! event loop that multiplexes terminal input events, ProviderEvent broadcasts, agent status updates, and render ticks. When this brief lands, the TUI is functional end-to-end: the user types a prompt, the agent responds with streaming text and tool calls, and the output renders in the scroll region with the fixed panel showing status.

## Task

Implement AppState in app/state.rs as the central state object connecting all subsystems: input editor, display toggles (verbosity, thinking), streaming indicator, agent status panel, tab state. Implement the main event loop in app/event_loop.rs using tokio::select! over: termina's async event reader (keyboard, resize), the ProviderEvent broadcast channel, agent status watch channels, and a periodic render tick for elapsed time updates. Wire ProviderEvent dispatch: TextDelta → markdown buffer → scroll region, ThinkingDelta → dim render (ephemeral, not persisted), ToolCallDelta → accumulate per ID → tool renderer on completion, Done → clear streaming indicator + show usage summary, Error → error style in scroll region. Wire terminal resize to DECSTBM recalculation and fixed panel redraw. Complete the run_tui() entry point in lib.rs that builds the runtime (via norn-cli's build_runtime), constructs the provider, sets up the terminal (NT-001), and enters the event loop.

### R2: TextDelta streaming to scroll region

WHEN a TextDelta event arrives from the ProviderEvent broadcast, THE SYSTEM SHALL pass the text through the markdown renderer and write the styled output to the scroll region.

**Acceptance:**
- TextDelta text is accumulated in the markdown renderer's buffer
- Plain text flushes immediately to the scroll region
- Incomplete markdown spans buffer until their closing marker (per NT-003)
- The scroll region cursor advances with each write
- Test: TextDelta with 'hello' produces 'hello' in the scroll region output

**Files:**
- create: crates/norn-tui/src/app/event_loop.rs

**Checklist:**
- C11 — TextDelta events write styled markdown output into the scroll region

### R3: ThinkingDelta rendering (ephemeral)

WHEN a ThinkingDelta event arrives, THE SYSTEM SHALL render it with ANSI dim attribute when thinking is visible AND discard it when thinking is hidden. ThinkingDelta is ephemeral — it SHALL NOT be persisted.

**Acceptance:**
- When display_toggles.thinking_visible is true, ThinkingDelta text renders dimmed in the scroll region
- When display_toggles.thinking_visible is false, ThinkingDelta text is silently discarded
- ThinkingDelta content is NOT written to the EventStore — it exists only in the streaming broadcast
- Test: ThinkingDelta with thinking_visible=false produces no scroll region output

**Files:**
- modify: crates/norn-tui/src/app/event_loop.rs

**Checklist:**
- C12 — ThinkingDelta events render with ANSI dim attribute in the scroll region

### R4: ToolCallDelta accumulation and rendering

WHEN ToolCallDelta events arrive, THE SYSTEM SHALL accumulate arguments per tool-call ID AND render through the per-tool renderer when the tool call is complete (next TextDelta, new tool-call ID, or Done event).

**Acceptance:**
- Pending tool calls tracked by ID in a HashMap<String, PendingToolCall>
- PendingToolCall accumulates: name (from first delta), arguments (concatenated deltas)
- When a tool call is flushed (new ID arrives, TextDelta arrives, or Done), the accumulated call is rendered via renderer_for(name) from NT-006/NT-007
- The rendered output (header + optional body per verbosity setting) is written to the scroll region
- Test: two ToolCallDelta events for the same ID accumulate into one rendered tool call

**Files:**

### R7: Streaming indicator lifecycle

The streaming indicator SHALL show '● generating...' with elapsed time while the model is producing output AND transition to the usage summary on completion.

**Acceptance:**

- Note: run_agent_step runs a full multi-iteration loop — the broadcast channel emits events from ALL iterations, including tool results between provider calls. The streaming indicator should show 'generating...' during provider calls and may briefly show tool execution state between iterations
- On Done, transitions to Complete with usage summary
- After a configurable delay (e.g. 5 seconds), transitions back to Idle
- Test: receiving a TextDelta when Idle transitions to Generating

### R8: Event loop with tokio::select!

The main event loop SHALL multiplex terminal events, ProviderEvent broadcasts, and render ticks using tokio::select!.

**Acceptance:**
- tokio::select! branches: terminal events (termina async reader), provider events (broadcast receiver), render tick (tokio::time::interval for elapsed time updates)
- Terminal resize events trigger DECSTBM recalculation and fixed panel redraw
- Keyboard events dispatch through the keybinding mapper (NT-004) to actions
- Submit action sends the input text to run_agent_step and subscribes to its broadcast channel
- Ctrl+O, Ctrl+E keybindings update display toggles
- The loop exits on Exit action (Ctrl+C on empty input)
- Test: the event loop function compiles and accepts the required channel types

**Files:**
- modify: crates/norn-tui/src/app/event_loop.rs

### R9: run_tui() entry point

The run_tui() function in lib.rs SHALL build the runtime, construct the provider, set up the terminal, and enter the event loop.

**Acceptance:**
- run_tui(cli: &Cli) -> ExitCode is the public entry point called from norn-cli's main.rs
- Calls build_runtime() (from norn-cli) to assemble the RuntimeBundle
- Calls build_provider() (from norn-cli) to construct the provider
- Calls terminal setup (NT-001) to enter raw mode and establish scroll regions
- Enters the event loop (R8) with the runtime components
- On exit, restores terminal state and returns the appropriate ExitCode
- Test: run_tui compiles with the correct signature

**Files:**
- modify: crates/norn-tui/src/lib.rs
