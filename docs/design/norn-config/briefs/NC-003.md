## Purpose

Implement the settings loading pipeline: discover settings files at up to three paths, parse JSON, merge layers with field-type-appropriate strategies, and validate the result. This is the core of the configuration system — it turns files on disk into a merged NornSettings that downstream code can consume.

## Task

Create norn::config::loader with load_settings() that discovers and parses settings files. Create norn::config::merge with merge_settings() implementing five-layer precedence with field-type-specific merge strategies (scalar override, deny-additive, allow-concatenate, hook-extend, MCP-by-name, tool-deep-merge). Create norn::config::validate with validate_settings() that checks duration strings, permission patterns, and MCP server shapes.

## Requirements

### R1: Settings file discovery

load_settings() SHALL discover settings files at three paths: ~/.norn/settings.json (user), .norn/settings.json (project), .norn/settings.local.json (local). Missing files SHALL be treated as empty settings (all-None). The function SHALL accept an explicit working directory for project-level discovery.

**Acceptance:**
- load_settings(cwd: &Path) defined in norn::config::loader
- Returns Result<LoadedSettings, ConfigError> where LoadedSettings holds user, project, and local layers
- User path resolved via norn::config::paths::settings_file()
- Project path is cwd/.norn/settings.json
- Local path is cwd/.norn/settings.local.json
- Missing file produces NornSettings::default() for that layer (no error)
- Test: missing files produce all-None settings

### R3: Five-layer merge for scalar fields

merge_settings() SHALL merge NornSettings layers in order: compiled defaults, user, project, local, CLI overrides. For scalar fields (Option<T>), higher-precedence non-None value SHALL win.

**Acceptance:**
- merge_settings defined in norn::config::merge
- Accepts user, project, local layers plus a CLI override layer
- Returns a single merged NornSettings
- Scalar: local Some(x) wins over project Some(y) wins over user Some(z)
- Scalar: None at higher layer preserves lower layer's value
- Test: three-layer scalar merge with mixed None/Some values

### R4: Permission deny-additive merge

Permission deny arrays SHALL union across all layers. A deny rule at any layer cannot be removed by a higher-precedence layer. Permission allow arrays SHALL concatenate across layers and deduplicate.

**Acceptance:**
- Deny: user [A, B] + project [C] + local [] produces [A, B, C]
- Deny: local cannot remove a user-level deny
- Allow: user [X] + project [X, Y] produces [X, Y] (deduplicated)
- Ask: same concatenate-deduplicate as allow
- Test: deny-additive merge verified
- Test: allow-deduplicate merge verified

### R5: Hook and MCP merge strategies

Hook arrays SHALL merge by event type: project hooks extend user hooks within the same event. MCP servers SHALL merge by name: same-name project server overrides user server definition.

**Acceptance:**
- Hooks: user pre_tool [A] + project pre_tool [B] produces pre_tool [A, B]
- Hooks: project does not remove user hooks of the same event type
- MCP: user {s1: def_a} + project {s1: def_b, s2: def_c} produces {s1: def_b, s2: def_c}
- MCP: same-name project server fully replaces user server (not deep-merged)
- Test: hook extend merge verified
- Test: MCP by-name merge verified

validate_settings() SHALL validate the merged settings: duration strings via humantime, permission patterns for syntactic well-formedness, MCP server entries for required fields. Unknown top-level keys SHALL emit tracing::warn, not an error.

**Acceptance:**
