## Purpose

Define the typed settings structs that represent the JSON settings schema. These types are the shared vocabulary that the loader (NC-003), merger (NC-003), builder integration (NC-004), and downstream consumers (NC-005) all reference. Landing the types first allows NC-003 to import them without a circular dependency.

## Task

Create norn::config::types with NornSettings and all sub-structs: ProviderSettings, AgentSettings, RetrySettings, PermissionSettings, HookSettings, HookEntry, ToolSettings, WriteToolSettings, McpServerSettings, SkillsSettings, ContextSettings, SessionSettings. All structs derive Serialize, Deserialize, Debug, Default with all-None/empty defaults. Duration fields are Option<String> (parsed by humantime at validation time, not at deserialization time).

## Requirements

### R1: NornSettings top-level struct

NornSettings SHALL be the root settings type with optional fields for every configuration section: model, provider, agent, retry, permissions, hooks, tools, mcp_servers, skills, context, session, tui, env.

**Acceptance:**
- NornSettings defined in norn::config::types
- model field is Option<String>
- provider field is Option<ProviderSettings>
- agent field is Option<AgentSettings>
- retry field is Option<RetrySettings>
- permissions field is Option<PermissionSettings>
- hooks field is Option<HookSettings>
- tools field is Option<ToolSettings>
- mcp_servers field is Option<BTreeMap<String, McpServerSettings>>
- skills field is Option<SkillsSettings>
- context field is Option<ContextSettings>
- session field is Option<SessionSettings>
- tui field is Option<serde_json::Value> (opaque, reserved for TUI cluster)
- env field is Option<BTreeMap<String, String>>
- Derives Serialize, Deserialize, Debug, Default, Clone
- Default yields all-None/empty fields

**Files:**
- create: crates/norn/src/config/types.rs
- modify: crates/norn/src/config/mod.rs

**Checklist:**
- C1 — NornSettings struct defined in norn::config::types with all top-level sections: model, provider, agent, retry, permissions, hooks, tools, mcp_servers, skills, session, tui, env.
- C10 — All settings structs derive Serialize, Deserialize, Debug, Default. Default yields all-None/empty fields (no assumed defaults).

### R5: HookSettings and HookEntry structs

HookSettings SHALL have one field per HookRegistry trait. HookEntry SHALL describe a single hook command with optional matcher and timeout.

**Acceptance:**
- HookSettings defined with fields: pre_tool (Option<Vec<HookEntry>>), post_tool (Option<Vec<HookEntry>>), pre_llm (Option<Vec<HookEntry>>), post_llm (Option<Vec<HookEntry>>), session_event (Option<Vec<HookEntry>>)
- HookEntry defined with fields: matcher (Option<String>), command (String), timeout (Option<u64>)
- Both derive Serialize, Deserialize, Debug, Default, Clone
- HookEntry command is required (not Option) — a hook without a command is invalid
