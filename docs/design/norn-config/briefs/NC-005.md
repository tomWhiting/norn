# NC-005: Downstream config surfaces — skills, context, session, rate limit

> **Cluster:** norn-config
> **Depends on:** NC-004
> **Checklist:**
> - C42 — skills.search_paths in settings provides additional skill search directories.
> - C43 — context.search_paths in settings provides context discovery directories.
> - C44 — session.history_capacity in settings overrides the REPL history capacity.
> - C45 — provider.rate_limit in settings overrides the hardcoded 60 req/min (Tom-decision value for default).
> - C46 — cargo clippy -p norn -p norn-cli -- -D warnings passes clean.
> - C47 — No file exceeds 500 lines of code (excluding tests, comments, whitespace).
> - C48 — No .unwrap() or .expect() in library code (norn crate).
> **Stories:**
> - S1 (AI Agent, Operating Under Configuration) — As an AI agent, I want my tool permissions to be configured in a settings file so that the operator does not need to pass --allowed-tools on every invocation.
> - S12 (Human Operator, Configuring Norn Per-Project) — As an operator, I want to add project-specific skill search paths in .norn/settings.json so that project skills are discovered automatically.
> - S15 (Human Operator, Overriding Configuration at Invocation) — As an operator, I want NORN_HOME to redirect all Norn state to a custom directory so that I can run isolated instances for testing or CI.
> - S20 (Workflow Engine, Launching Norn Programmatically) — As a workflow engine, I want per-tool config (write limits, edit config) to be expressible in settings so that workflow-level tool policies do not require custom code.

## Purpose

Wire the remaining settings fields into their consumers: skill search paths, context search paths, session history capacity, and provider rate limit. These are the last-mile connections that make the config system end-to-end functional. After this brief, every settings field defined in NC-002 has a consumer.

## Task

Wire skills.search_paths from merged settings into the SkillSearchPaths extension on ToolContext. Wire context.search_paths into a discoverable extension. Wire session.history_capacity into REPL history construction. Wire provider.rate_limit into the OpenAI provider rate limiter. Verify all linting and quality gates pass.

## Requirements

### R1: Skill search paths from settings

WHEN merged settings contain skills.search_paths, THE SYSTEM SHALL include those directories in the SkillSearchPaths extension installed on the shared ToolContext, alongside the default scan directories.

**Acceptance:**
- Settings skills.search_paths values are prepended to the SkillSearchPaths vec
- Default scan directories (.norn/skills/, ~/.norn/skills/) are still included
- Settings paths are resolved relative to the working directory
- Test: settings skill paths appear in SkillSearchPaths extension

**Files:**
- modify: crates/norn-cli/src/runtime/wiring.rs

**Checklist:**
- C42 — skills.search_paths in settings provides additional skill search directories.

**Stories:**
- S12 (Human Operator, Configuring Norn Per-Project) — As an operator, I want to add project-specific skill search paths in .norn/settings.json so that project skills are discovered automatically.

### R2: Context search paths from settings

WHEN merged settings contain context.search_paths, THE SYSTEM SHALL make them available as a discoverable ToolContext extension so the context cluster can read them.

**Acceptance:**
- ContextSearchPaths struct defined (newtype over Vec<PathBuf>)
- ContextSearchPaths installed on the shared ToolContext when settings provide context.search_paths
- Settings paths are resolved relative to the working directory
- Test: settings context paths accessible via ToolContext extension

**Files:**

**Checklist:**

### R3: Session history capacity from settings

WHEN merged settings contain session.history_capacity, THE SYSTEM SHALL use that value for REPL history construction instead of the hardcoded HISTORY_CAPACITY constant.

**Acceptance:**

- Absent setting falls back to the existing HISTORY_CAPACITY constant
- Test: settings history_capacity overrides the default

### R4: Provider rate limit from settings

WHEN merged settings contain provider.rate_limit, THE SYSTEM SHALL use that value for the OpenAI provider rate limiter permits-per-minute instead of the hardcoded DEFAULT_PERMITS_PER_INTERVAL constant. The actual default value is a Tom-decision.

**Acceptance:**

- Absent setting falls back to the existing DEFAULT_PERMITS_PER_INTERVAL constant
