# NR-002: Norn session loop — async agent lifecycle

> **Cluster:** norn-runtime
> **Depends on:** NR-001
> **Checklist:**
> - C10 — run_norn_session_loop is an async function spawned via tokio::spawn (not spawn_blocking)
> - C11 — Norn session loop multiplexes AgentEvent broadcast, SessionCommand mpsc, idle timeout, and agent completion
> - C12 — SessionCommand::SendMessage delivers text via Norn InboundChannel
> - C13 — SessionCommand::Stop cancels via CancellationToken with graceful drain of remaining events
> - C14 — SessionCommand::Kill cancels via CancellationToken immediately
> - C15 — Idle timeout fires only after the agent has completed at least one turn
> - C16 — Session status transitions to Stopped when the Norn agent run completes
> - C17 — Completion watcher emits ServiceEvent::SessionCompleted or SessionFailed on the global EventBus
> **Stories:**
> - S1 (AI Agent, Running a Member-Facing Session) — As an AI agent, I want my session to run through Norn in-process so that I have typed tool access and structured event streaming instead of subprocess JSONL parsing.
> - S3 (AI Agent, Running a Member-Facing Session) — As an AI agent, I want to receive DMs as wake prompts through Norn so that I see my inbox immediately on wake without polling.
> - S4 (AI Agent, Running a Member-Facing Session) — As an AI agent, I want mid-session DMs to arrive via my inbound channel so that I can respond to messages while actively working.
> - S20 (System, Managing Session Resources) — As the system, I want idle timeout semantics to match between runtimes so that resource cleanup is predictable.

## Purpose

Implement the async Norn session loop that replaces the blocking Claude Runner session loop for Norn-powered sessions. This is the core runtime integration — it builds a Norn Agent, runs it via tokio::spawn, translates events into AssistantEvent, and handles session commands through Norn's native mechanisms.

## Task

Implement run_norn_session_loop in norn_session.rs and wire spawn_norn_loop in service.rs to call it. The loop multiplexes four event sources: Norn AgentEvent broadcast, SessionCommand mpsc, idle timeout, and agent completion. Commands are delivered via InboundChannel (SendMessage) and CancellationToken (Stop/Kill). The completion watcher emits lifecycle events on the global EventBus. This brief replaces the NR-001 stub.

Dependency setup: No new cargo dependencies. The norn crate is already a dependency of meridian-services (norn = { path = "../norn" }). All required tokio features (sync, time, rt-multi-thread) are already present.

### R2: AgentBuilder construction with InboundChannel

The InboundSender half of the channel SHALL be stored on the SessionHandle so the session loop can receive messages. THE SYSTEM SHALL call Agent::run_with(prompt) where prompt is the initial_message from SessionConfig.

**Acceptance:**
- AgentBuilder is constructed with provider, event sender, cancellation token, profile, and InboundChannel
- InboundSender is stored on SessionHandle for later SendMessage delivery
- Agent::run_with is called with the initial_message text
- The agent runs as an async task within the norn session loop
- InboundChannel is created by the server (not inherited from sub-agent machinery)

### R3: Event multiplexing select loop

The norn session loop SHALL use tokio::select! to multiplex: (a) AgentEvent from broadcast::Receiver — translated via NornTranslateState and forwarded to the session broadcast, (b) SessionCommand from mpsc::Receiver — dispatched to InboundChannel or CancellationToken, (c) idle timeout — fires after configured duration with no activity, (d) agent join handle — resolves when the Norn agent completes.

**Acceptance:**
- tokio::select! multiplexes all four sources
- AgentEvent events are translated and forwarded to broadcast::Sender<AssistantEvent>
- SessionCommand messages dispatch correctly
- Idle timeout branch stops the session
- Agent completion branch exits the loop

### R4: SendMessage bridge to InboundChannel with DeliveryMode

WHEN SessionCommand::SendMessage(text) is received, THE SYSTEM SHALL construct a ChannelMessage from the text, send it via InboundSender with DeliveryMode::Steer (injected at next tool boundary, interrupts current work), AND emit AssistantEvent::UserMessage on the session broadcast. DMs use DeliveryMode::Steer because they are intentional communication requiring attention. Notifications (if routed through a future path) SHOULD use DeliveryMode::FollowUp (surfaces only if agent would otherwise stop).

**Acceptance:**
- SendMessage constructs a ChannelMessage with the text content
- ChannelMessage is sent via InboundSender (the sender half stored on SessionHandle)
- DeliveryMode::Steer is used for DM-originated messages
- AssistantEvent::UserMessage is emitted with the text on the session broadcast
- Idle timeout timer is reset on SendMessage
- If InboundSender is closed (agent completed), SendMessage returns an error instead of silently dropping

### R5: Stop and Kill via CancellationToken

WHEN SessionCommand::Stop is received, THE SYSTEM SHALL cancel the CancellationToken and drain remaining events until the agent completes. WHEN SessionCommand::Kill is received, THE SYSTEM SHALL cancel the CancellationToken and exit the loop immediately without draining.

**Acceptance:**
- Stop sets a stopping flag, cancels the token, and continues draining events
- Kill cancels the token and breaks from the loop immediately
- Both emit AssistantEvent::StatusChanged with SessionStatus::Stopping then Stopped

### R6: Idle timeout with had_activity gate

WHEN an idle timeout is configured AND the agent has completed at least one turn (had_activity is true), THE SYSTEM SHALL stop the session after the timeout duration elapses with no events. The timer SHALL reset on every AgentEvent and every SendMessage command. WHEN had_activity is false, the timeout SHALL NOT fire.

**Acceptance:**
- Idle timeout uses tokio::time::sleep, not crossbeam::after
- Timer resets on AgentEvent receipt and SendMessage receipt
- Timer does not fire before the first turn completes (had_activity gate)
- Timeout causes session stop with SessionExitStatus::IdleTimeout equivalent

### R7: Session completion and status transitions

WHEN the Norn agent's run_with join handle resolves with Ok, THE SYSTEM SHALL emit StatusChanged(Stopped), mark the session as completed in the registry, emit ServiceEvent::SessionCompleted on the global EventBus, mark the agent session finished in PG, and release the member's wake claim. WHEN run_with returns Err (provider auth failure, permanent error), THE SYSTEM SHALL emit StatusChanged(Stopped) and ServiceEvent::SessionFailed with the error message. WHEN run_with panics, THE SYSTEM SHALL emit ServiceEvent::SessionFailed. All three paths SHALL release the member's wake claim.

**Acceptance:**
- Agent completion (Ok) emits StatusChanged(Stopped) + ServiceEvent::SessionCompleted
- Agent error (Err) emits StatusChanged(Stopped) + ServiceEvent::SessionFailed with error message
- Agent panic emits ServiceEvent::SessionFailed
- All three paths (Ok, Err, panic) release wake claim by setting member activity to Available
- All three paths mark the agent session finished in PG
- SessionRegistry is updated via mark_completed in all cases
