# NR-010: Frontend runtime awareness and session tab differentiation

> **Cluster:** norn-runtime
> **Depends on:** NR-001
> **Checklist:**
> - C85 — Frontend session types include runtime_kind field, surfaced through useAssistantSession and useActiveAgentSessions hooks
> - C86 — No hardcoded 'claude' model fallback strings in AssistantPanel.tsx — runtime-aware fallback instead
> - C87 — Session toolbar displays a runtime badge (Norn or Claude Runner) in the ProfileRow with distinct visual treatment
> **Stories:**
> - S15 (Human Developer, Debugging Agent Sessions) — As a developer, I want to see which runtime a session is using so that I can diagnose runtime-specific issues.
> - S24 (Human Developer, Monitoring and Configuring Agent Sessions) — As a developer, I want a member-facing Norn profile that configures namespace tools and system instructions for workspace member behavior so that I can wake members as Norn sessions without manual configuration.

## Purpose

The frontend currently has no concept of which runtime backs an agent session. The session API already returns runtime_kind (SessionInfoResponse at api/assistant.rs:103) but the frontend never reads it. Two model fallback strings hardcode 'claude'. The session tab in CommsInput renders identical content for both runtimes. This brief adds runtime awareness to the frontend so Tom can distinguish Norn from Claude Runner sessions and the UI adapts appropriately.

## Task

Add runtime_kind to the frontend session types and surface it in the assistant hooks. Fix the two hardcoded 'claude' model fallbacks in AssistantPanel.tsx. Add a runtime indicator to the session toolbar so the active runtime is visible. The SessionStartedEvent from the WebSocket should include runtime_kind if available, and the REST session response already provides it.

## Requirements

### R1: Runtime kind in frontend session types

WHEN the frontend receives session data from REST or WebSocket THE SYSTEM SHALL include a runtime_kind field ('norn' or 'claude_runner') AND it SHALL surface runtime_kind through useAssistantSession and useActiveAgentSessions hooks.

**Acceptance:**
- SessionStartedEvent type in features/assistant/types.ts includes optional runtime_kind?: 'norn' | 'claude_runner' field
- useAssistantSession hook exposes runtimeKind from the session_started event
- REST session response type includes runtime_kind field matching the backend SessionInfoResponse
- useActiveAgentSessions includes runtime_kind in the session info returned to consumers

**Files:**
- modify: apps/web/src/features/assistant/types.ts
- modify: apps/web/src/features/assistant/hooks/useAssistantSession.ts

### R2: Fix hardcoded model fallback strings

WHEN displaying the model name for an agent session THE SYSTEM SHALL use the model field from the session data AND it SHALL NOT fall back to a hardcoded 'claude' string. WHEN the model field is null or undefined, THE SYSTEM SHALL display the runtime kind as fallback ('Norn' or 'Claude Runner').

**Acceptance:**
- AssistantPanel.tsx line 118 no longer uses ?? 'claude' — uses runtime-aware fallback
- AssistantPanel.tsx line 334 no longer uses ?? 'claude' — uses runtime-aware fallback
- A Norn session with no model field displays 'Norn' not 'claude'
- A Claude Runner session with no model field displays 'Claude Runner' not 'claude'

**Files:**

### R3: Runtime indicator in session toolbar

WHEN the session toolbar renders for an active session THE SYSTEM SHALL display a runtime badge showing 'Norn' or 'Claude Runner' alongside the model and context window badges in the ProfileRow AND it SHALL use distinct visual treatment (color or icon) so the runtime is identifiable at a glance.

**Acceptance:**
- Session toolbar (session-toolbar.tsx ProfileRow, around line 73-82) includes a runtime badge
- Badge displays 'Norn' for Norn sessions and 'Claude Runner' for Claude Runner sessions
- Badge uses distinct color: Norn sessions get a different badge color than Claude Runner
- Badge is visible without expanding the profile row — it appears on the same row as model and context window badges

## Boundaries

- SHALL NOT change the backend SessionInfoResponse or AssistantEvent types — they already include runtime_kind
- SHALL NOT add Norn-specific event processing to useAssistantSession — the event stream is runtime-agnostic by design
- SHALL NOT change the session toolbar layout structure — only add the runtime badge within the existing ProfileRow
