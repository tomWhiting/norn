> - S19 (System, Managing Session Resources) — As the system, I want Norn sessions to persist to both JSONL and PG so that session resume and UI history are independently served.

## Purpose

Wire Norn's JsonlSink-backed EventStore into the assistant service so that session conversation history persists to disk and can be replayed on resume. Establish the dual persistence model: JSONL files for Norn's conversation continuity, PG events for the UI's history display. Both coexist independently.

## Task

Create a session file manager that handles path resolution, directory creation, EventStore construction for new/resume/fork modes, and JSONL file lifecycle. The session file path convention is {data_dir}/sessions/norn/{session_id}.jsonl. spawn_norn_loop calls this manager to obtain an EventStore before building the AgentBuilder. The existing PG persistence (persist_events) and Redis streaming (stream_deltas_to_redis) remain unchanged — they subscribe to the AssistantEvent broadcast channel.

Dependency setup: No new cargo dependencies. The norn crate (JsonlSink, EventStore, SessionEvent) and tempfile are already dependencies of meridian-services.

## Requirements

### R1: Session directory and path resolution

The system SHALL resolve Norn session JSONL file paths as {data_dir}/sessions/norn/{session_id}.jsonl where data_dir is the server's configured data directory. THE SYSTEM SHALL create the sessions/norn/ directory if it does not exist. THE SYSTEM SHALL NOT use ~/.norn/ (norn-cli local convention).

**Acceptance:**
- Session files are at {data_dir}/sessions/norn/{session_id}.jsonl
- Directory is created automatically if missing
- data_dir comes from server configuration (ServiceContext or ServerConfig)
- Path resolution is a pure function testable without filesystem

### R2: EventStore for new sessions

WHEN a new Norn session starts (SessionStartMode::New), THE SYSTEM SHALL create a JsonlSink at the session file path in write mode AND construct EventStore::with_sink(sink) with empty initial events.

**Acceptance:**
- New session creates a fresh JSONL file
- EventStore has no initial events
- JsonlSink is in write (create) mode
- The session file path follows the {session_id}.jsonl convention

**Files:**

**Checklist:**

### R3: EventStore for resumed sessions

WHEN an existing Norn session resumes (SessionStartMode::Resume), THE SYSTEM SHALL read the JSONL file at the session path, deserialize SessionEvent lines into a Vec<SessionEvent>, create a JsonlSink in append mode for the same file, and construct EventStore::with_sink_and_events(sink, events). IF the JSONL file does not exist, THE SYSTEM SHALL fall back to creating a new session (EventStore::with_sink, no error).

**Acceptance:**
- Resume reads existing JSONL and replays events
- JsonlSink is in append mode (does not truncate existing content)
- EventStore receives pre-populated events from the file
- Missing file falls back to new session behavior (no error, clean start)
- Corrupt lines are logged and skipped, not fatal
- Truncated trailing line (incomplete JSON from power failure mid-write) is treated as corrupt and skipped
- Empty file (0 bytes) falls back to new session behavior
- File containing valid JSON that is not SessionEvent is treated as corrupt per line

### R4: EventStore for forked sessions

WHEN a session is forked (SessionStartMode::Fork), THE SYSTEM SHALL copy the source session's JSONL file to a new path (new session ID), then construct an EventStore from the copy using the resume flow. IF the source file does not exist, THE SYSTEM SHALL create a new session instead.

**Acceptance:**
- Fork copies source JSONL to {new_session_id}.jsonl
- New file is independent from source (copy, not symlink)
- EventStore is built from the copy with resume semantics
- Missing source file falls back to new session

### R5: AgentBuilder receives EventStore

WHEN spawn_norn_loop builds the AgentBuilder, THE SYSTEM SHALL call .session(store) with the EventStore obtained from the session file manager. THE SYSTEM SHALL NOT call .session() with a bare EventStore::new() (the bug in make_norn_step_runner).

**Acceptance:**
- AgentBuilder::session(store) is called with a JsonlSink-backed EventStore
- EventStore::new() is never used for member-facing Norn sessions
- Session events are written to disk as the agent runs

### R6: PG and Redis persistence unchanged

The existing persist_events and stream_deltas_to_redis tasks SHALL continue to subscribe to broadcast::Sender<AssistantEvent> without modification. WHILE a Norn session runs, THE SYSTEM SHALL have both JSONL persistence (Norn-internal) and PG/Redis persistence (AssistantEvent broadcast) active simultaneously.

**Acceptance:**
- persist_events spawned for Norn sessions exactly as for Claude Runner sessions
- stream_deltas_to_redis spawned for Norn sessions exactly as for Claude Runner sessions
- Both subscribe to the same broadcast channel that the Norn session loop writes to
- No modifications to persistence.rs or streaming.rs

## Boundaries

- SHALL NOT modify persistence.rs or streaming.rs
- SHALL NOT modify the EventStore or JsonlSink implementations in the norn crate
- SHALL NOT modify the JSONL format — use Norn's existing SessionEvent serialization
- SHALL NOT implement cleanup/garbage collection of old session files (future scope)

## Verification

- New session creates a fresh JSONL file at the correct path
- Resume reads an existing JSONL file and pre-populates the EventStore
- Fork copies the source file and builds from the copy
- Missing source file for resume/fork falls back to new session without error
- AgentBuilder::session() is called (not bare EventStore::new())
- PG events appear in assistant_session_events for Norn sessions
- Redis stream receives delta events for Norn sessions
