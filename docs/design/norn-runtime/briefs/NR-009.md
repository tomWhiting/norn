# NR-009: Mid-session DM delivery to active Norn agents

> **Cluster:** norn-runtime
> **Depends on:** NR-002
> **Checklist:**
> - C76 — wake_trigger delivers DMs to active Norn sessions via SessionCommand::SendMessage instead of returning WakeSkipped
> - C77 — Mid-session DM formatted via format_agent_dm_envelope — identical envelope shape to wake prompt delivery, notification marked delivered after successful send
> **Stories:**
> - S4 (AI Agent, Running a Member-Facing Session) — As an AI agent, I want mid-session DMs to arrive via my inbound channel so that I can respond to messages while actively working.
> - S22 (AI Agent, Receiving Messages While Active) — As an AI agent with an active session, I want DMs that arrive mid-session to be delivered to my inbound channel immediately so that I can respond without waiting for a wake cycle.

## Purpose

When a DM arrives for an agent that is already running a Norn session, the message is silently dropped. The wake_trigger catches ServiceEvent::MessageSent, checks active_session_for_member, finds the session active, and returns WakeSkipped — nobody delivers the DM. The InboundChannel plumbing exists (SessionCommand::SendMessage works, norn_session.rs:325-356 constructs ChannelMessages with DeliveryMode::Steer), but the wake_trigger never calls it for active sessions. This brief connects the existing plumbing so DMs reach running agents immediately.

## Task

Modify the wake_trigger WakeSkipped path to deliver the DM content to the active session via registry.send_command(SessionCommand::SendMessage). The DM must be formatted using the same format_agent_dm_envelope function used by build_wake_prompt, so the agent sees identical DM format regardless of whether it was idle or active when the message arrived. The formatted envelope is delivered as a String via SessionCommand::SendMessage; the norn_session.rs handler wraps it as a ChannelMessage with DeliveryMode::Steer and author='user', and the agent picks it up at the next inbound drain point (runner.rs:589, 709, 723, or 839).

## Requirements

### R1: Deliver DM to active session instead of WakeSkipped

WHEN a DM triggers the wake_trigger AND the recipient has an active session THE SYSTEM SHALL deliver the DM content to the active session via SessionCommand::SendMessage instead of returning WakeSkipped AND it SHALL format the DM using format_agent_dm_envelope to match the wake prompt format.

**Acceptance:**
- wake_trigger.rs WakeSkipped path (around line 159) now calls registry.send_command(session_id, SessionCommand::SendMessage(formatted_dm))
- The formatted DM uses format_agent_dm_envelope(content, sender_name, timestamp, subject, Some(session_id), exchange_origin) — same function used by build_wake_prompt at wake_prompt.rs:118
- The DM content and sender name are fetched from storage before formatting — not using raw ServiceEvent fields which may lack full content
- The agent sees '--- DM from {sender} at {time} ---\n{content}\n---\n<reply_instructions>...' regardless of idle vs active delivery path

### R2: Fetch DM content for mid-session formatting

WHEN the wake_trigger delivers a DM to an active session THE SYSTEM SHALL fetch the full DM content from storage via DirectMessageStore::get_direct_message AND it SHALL resolve the sender name via MemberStore AND it SHALL NOT use truncated preview text from the notification.

**Acceptance:**
- Full DM content retrieved via storage.get_direct_message(&message_id)
- Sender name resolved from sender member record, not from notification preview
- If storage fetch fails, falls back to a brief notification-style message directing the agent to check its inbox (matching build_wake_prompt fallback at wake_prompt.rs:41-47)
- Subject extracted from DM if present, passed to format_agent_dm_envelope

### R3: Mark notification as delivered after mid-session injection

WHEN a DM is successfully delivered to an active session via SendMessage THE SYSTEM SHALL mark the corresponding notification as delivered so it does not resurface in future notification checks or wake prompts AND it SHALL NOT mark the notification if the send_command call fails.

**Acceptance:**
- After successful send_command, the notification ID is passed to storage.mark_notifications_delivered
- Failed send_command (session gone, channel closed) skips the mark — notification remains pending for next wake
- The notification does not appear in subsequent get_rich_inbox or get_pending_notifications calls for this member

## Boundaries

- SHALL NOT modify the InboundChannel, InboundSender, or ChannelMessage types in the norn crate
- SHALL NOT modify norn_session.rs — the existing SessionCommand::SendMessage handler already constructs the correct ChannelMessage
- SHALL NOT change the wake path for idle sessions — build_wake_prompt and resume_session are untouched
- SHALL NOT change format_agent_dm_envelope — use it as-is for format consistency
- SHALL NOT deliver DMs to Claude Runner sessions via this path — Claude Runner mid-session delivery uses the existing stdin write mechanism
