## Purpose

Build the namespace tool infrastructure (MeridianToolContext, registration, profile gating) and implement the three critical namespace tools that every member-facing agent needs: meridian_messaging, meridian_member, and meridian_workspace. These are the critical tool surface for a Norn-powered member session to communicate, look up other members, and query workspace context.

## Task

Create crates/meridian-tools/src/ module with the MeridianToolContext extension, the register_meridian_tools function, and the MessagingTool, MemberTool, and WorkspaceTool implementations. Each namespace tool implements norn::tool::Tool with a command discriminant. Tools call service-layer functions directly via ServiceContext. Profile filtering gates which namespace tools are available. The CallerContext extension carries member_id, session_id, and workspace_id for authorization.

## Requirements

### R1: norn_tools module and MeridianToolContext

The system SHALL create a norn_tools module in crates/meridian-services/src/ with a MeridianToolContext type that holds Arc<ServiceContext> and implements a trait-based service interface (per DESIGN.md D11). MeridianToolContext SHALL be installed as a ToolContext extension during AgentBuilder construction. Namespace tools SHALL extract it from ToolContext to access service-layer functions. Both host and VM provide in-process backends — host has the full service set, VM has a limited set (workflow + source). Tools calling unavailable services receive context-aware error messages per D11.

**Acceptance:**
- crates/meridian-tools/src/lib.rs exists with pub mod declarations and re-exports
- crates/meridian-tools/src/context.rs defines MeridianToolContext
- MeridianToolContext holds Arc<ServiceContext> for in-process service calls
- MeridianToolContext is retrievable from ToolContext via get_extension::<MeridianToolContext>()
- MeridianToolContext exposes trait methods (e.g. send_dm, get_inbox) that both host and VM backends implement in-process
- Host backend calls full ServiceContext; VM backend calls limited ServiceContext (workflow + source only)
- Tools calling unavailable services in VM context receive context-aware errors (e.g. 'messaging is not available in shared workspace mode')

### R2: CallerContext extension for authorization and identity

A CallerContext type SHALL carry member_id (Uuid), session_id (Uuid), workspace_id (Option<Uuid>), and agent_cert_fingerprint (Option<[u8; 32]>) as an Arc extension on ToolContext. CallerContext is set at AgentBuilder construction and is unforgeable — not passable as tool input. Namespace tools SHALL read CallerContext to scope storage queries, authorize operations to the calling member, and optionally sign exchange actions with the agent cert. This is the canonical CallerContext definition — other clusters (TA-001, W-006) reference this.

**Acceptance:**
- CallerContext struct has fields: member_id (Uuid), session_id (Uuid), workspace_id (Option<Uuid>), agent_cert_fingerprint (Option<[u8; 32]>)
- CallerContext installed as Arc extension on ToolContext during Norn session setup in AssistantService
- CallerContext is not constructable from tool input — only from session setup code
- Namespace tools extract CallerContext via get_extension::<CallerContext>() and use member_id for authorization scoping
- agent_cert_fingerprint is None for host-instance sessions, Some for VM/exchange sessions

**Files:**
- modify: crates/meridian-tools/src/context.rs

### R3: register_meridian_tools function

A register_meridian_tools function SHALL accept an AgentBuilder, ServiceContext, CallerContext, and a list of enabled tool names. It SHALL add each enabled namespace tool to the builder via .tool(). Tool names use the meridian_ prefix (e.g., meridian_messaging, meridian_member). WHEN a tool name is not in the enabled list, THE SYSTEM SHALL skip it.

**Acceptance:**
- register_meridian_tools function exists in crates/meridian-tools/src/lib.rs
- It adds Box<dyn Tool> instances for each enabled namespace tool
- Disabled tools are not added to the builder
- Tool registry names use meridian_ prefix

### R4: MessagingTool — meridian_messaging namespace

MessagingTool SHALL implement norn::tool::Tool. The command enum and per-command parameter structs SHALL use #[derive(ToolArgs)] from norn-macros to generate input_schema. The tool SHALL use tool_follow_ups! to register output-to-follow-on mappings per tool-overview.md. Commands: send, inbox, read, search, mark_read, snooze, respond, retry, channel_send, channel_mentions, notify_check, notify_summary. The execute method SHALL dispatch on the command value and call the corresponding service-layer function via MeridianToolContext. Notification commands (snooze, notify_check, notify_summary) and channel commands (channel_send, channel_mentions) are part of the messaging namespace, not separate tools. Aligned with TA-002.

**Acceptance:**
- MessagingTool implements Tool with name() returning 'meridian_messaging'
- Command enum and per-command parameter structs use #[derive(ToolArgs)]
- tool_follow_ups! macro registers output-to-follow-on mappings per tool-overview.md
- input_schema includes command enum with all 12 commands
- send command calls MessagingService::send_dm with content, recipient name
- inbox command retrieves conversations with metadata for the calling member
- read command fetches full message content by ID
- search command finds messages by query text
- mark_read command marks a conversation as read
- snooze command defers a message for later
- respond command replies to a message with content and optional structured data
- retry command retries a failed message delivery
- channel_send command sends a message to a named channel
- channel_mentions command queries @mentions for the calling member in a channel
- notify_check command returns pending notification count
- notify_summary command returns a summary of pending notifications
- All commands scope to CallerContext.member_id
- Service errors (PG connection failure, member not found, etc.) return ToolOutput with is_error=true and human-readable error message, not agent-level panics
