# NR-001: Runtime discriminant and session lifecycle branching

> **Cluster:** norn-runtime
> **Checklist:**
> - C1 — RuntimeKind enum with ClaudeRunner and Norn variants defined in types.rs
> - C2 — SessionConfig has an optional runtime field of type Option<RuntimeKind>
> - C3 — SessionHandle stores RuntimeKind and is queryable via the registry
> - C4 — AssistantError has NornBuildFailed and NornSessionFailed variants
> - C5 — AssistantService::start_session branches on RuntimeKind after token acquisition
> - C6 — spawn_norn_loop method builds AgentBuilder, creates event channels, spawns async task
> - C7 — Existing spawn_session_loop (Claude Runner path) is unchanged and continues to work
> - C8 — Both runtime paths register a SessionHandle with the same SessionRegistry
> - C9 — Both runtime paths produce broadcast::Sender<AssistantEvent> for downstream consumers
> **Stories:**
> - S12 (Human Developer, Configuring Agent Sessions) — As a developer, I want to switch a member's runtime between Norn and Claude Runner via wake config so that I can migrate incrementally.
> - S13 (Human Developer, Configuring Agent Sessions) — As a developer, I want both runtimes to produce the same event stream so that the frontend works identically regardless of runtime.
> - S15 (Human Developer, Debugging Agent Sessions) — As a developer, I want to see which runtime a session is using so that I can diagnose runtime-specific issues.

## Purpose

Introduce the RuntimeKind discriminant so that AssistantService can drive either Norn or Claude Runner behind a single session lifecycle contract. This is the foundation brief — every other brief in the cluster depends on the types and branching established here.

## Task

Add RuntimeKind enum to types.rs, add runtime field to SessionConfig, store RuntimeKind on SessionHandle, add Norn error variants to AssistantError, and modify start_session to branch on RuntimeKind. The Claude Runner path remains unchanged. The Norn path is stubbed (spawn_norn_loop exists but delegates to NR-002 for implementation). Both paths converge on the same SessionRegistry and broadcast::Sender<AssistantEvent>.

## Requirements

### R1: RuntimeKind enum

The system SHALL define a RuntimeKind enum with variants ClaudeRunner and Norn in types.rs. RuntimeKind SHALL derive Clone, Copy, Debug, PartialEq, Eq, Serialize, Deserialize. RuntimeKind SHALL implement Default with ClaudeRunner as the default.

**Acceptance:**
- RuntimeKind::ClaudeRunner and RuntimeKind::Norn are defined in types.rs
- RuntimeKind derives Clone, Copy, Debug, PartialEq, Eq, Serialize, Deserialize
- RuntimeKind::default() returns ClaudeRunner

**Files:**
- modify: crates/meridian-services/src/assistant/types.rs

**Checklist:**
- C1 — RuntimeKind enum with ClaudeRunner and Norn variants defined in types.rs

### R2: SessionConfig runtime field

SessionConfig SHALL have a field `runtime: Option<RuntimeKind>` that defaults to None. WHEN runtime is None, the session SHALL use the server default runtime. WHEN runtime is Some(kind), the session SHALL use that runtime regardless of defaults.

**Acceptance:**
- SessionConfig has runtime: Option<RuntimeKind> field
- SessionConfig::default() has runtime: None
- Existing SessionConfig construction sites compile without changes (field defaults to None)

**Files:**
- modify: crates/meridian-services/src/assistant/types.rs

**Checklist:**
- C2 — SessionConfig has an optional runtime field of type Option<RuntimeKind>

**Stories:**
- S12 (Human Developer, Configuring Agent Sessions) — As a developer, I want to switch a member's runtime between Norn and Claude Runner via wake config so that I can migrate incrementally.

### R3: SessionHandle stores RuntimeKind

SessionHandle SHALL store the resolved RuntimeKind. SessionInfo (the public snapshot) SHALL expose the runtime_kind field so that WS handlers and API responses can report which runtime a session uses.

**Acceptance:**
- SessionHandle has a runtime_kind: RuntimeKind field
- SessionInfo has a runtime_kind: RuntimeKind field
- SessionRegistry get() and list() return SessionInfo with populated runtime_kind

**Files:**
- modify: crates/meridian-services/src/assistant/registry.rs
- modify: crates/meridian-services/src/assistant/types.rs

**Checklist:**
- C3 — SessionHandle stores RuntimeKind and is queryable via the registry

**Stories:**
- S15 (Human Developer, Debugging Agent Sessions) — As a developer, I want to see which runtime a session is using so that I can diagnose runtime-specific issues.

### R4: Norn error variants

AssistantError SHALL have variants NornBuildFailed(String) and NornSessionFailed(String) for errors originating from the Norn runtime path. These SHALL NOT overlap with existing error variants.

**Acceptance:**
- AssistantError::NornBuildFailed(String) is defined in error.rs
- AssistantError::NornSessionFailed(String) is defined in error.rs
- Both variants have Display implementations via thiserror

**Files:**
- modify: crates/meridian-services/src/assistant/error.rs

**Checklist:**
- C4 — AssistantError has NornBuildFailed and NornSessionFailed variants

### R5: start_session branches on RuntimeKind

WHEN start_session is called with a SessionConfig whose resolved runtime is ClaudeRunner, THE SYSTEM SHALL follow the existing Claude Runner path (build_command, spawn_session_loop). WHEN the resolved runtime is Norn, THE SYSTEM SHALL follow the Norn path (spawn_norn_loop). Both paths SHALL register a SessionHandle in the same SessionRegistry and produce a broadcast::Sender<AssistantEvent>.

**Acceptance:**
- start_session resolves RuntimeKind from config.runtime (or server default if None)
- ClaudeRunner path calls existing build_command and spawn_session_loop unchanged
- Norn path calls spawn_norn_loop (stub in this brief, implemented in NR-002)
- Both paths register a SessionHandle with the resolved RuntimeKind
- Both paths create broadcast::channel<AssistantEvent>(256)

**Files:**
- modify: crates/meridian-services/src/assistant/service.rs

**Checklist:**
- C5 — AssistantService::start_session branches on RuntimeKind after token acquisition
- C7 — Existing spawn_session_loop (Claude Runner path) is unchanged and continues to work
- C8 — Both runtime paths register a SessionHandle with the same SessionRegistry
- C9 — Both runtime paths produce broadcast::Sender<AssistantEvent> for downstream consumers

**Stories:**

### R6: spawn_norn_loop stub

A spawn_norn_loop method SHALL exist on AssistantService that accepts session_id, config, and RuntimeKind::Norn-specific artifacts. In this brief it SHALL return AssistantError::NornBuildFailed with a message indicating the Norn path is not yet implemented. NR-002 replaces the stub with the real implementation.

**Acceptance:**
- spawn_norn_loop method exists on AssistantService
- Calling spawn_norn_loop returns Err(AssistantError::NornBuildFailed(...))
- The existing spawn_session_loop is renamed to spawn_claude_runner_loop for clarity

**Files:**
- modify: crates/meridian-services/src/assistant/service.rs

**Checklist:**

## Boundaries

- SHALL NOT implement event translation (NR-003 scope)
- SHALL NOT modify the Claude Runner session loop or translate layer
