# NA-004: M0 bridge and minimal UI — session list + one streamed turn

> **Cluster:** norn-app
> **Milestone:** M0 (spike) — completing this brief completes M0
> **Depends on:** NA-001 (shell), NA-002 (DTOs), NA-003 (session service)
> **Spec anchors:** §3 (bridge design), R1.1, R1.3, R3.1, R3.3, N3, N4, M0 acceptance, §9 (first bullet)

## Purpose

Wire the shell to the engine: `crates/norn-app`'s Tauri backend embeds `crates/norn` in-process via NA-003's session service, streams NA-002 DTO envelopes over the Tauri event channel, and the frontend renders a session list and a single chat pane where a real turn streams from the real `~/.norn` store. This proves the bridge — the entire point of M0.

## Task

Implement the Tauri commands (list sessions, open session, send turn, cancel turn), the event-channel subscription (one per window), the Zustand stores consuming envelopes, and two views (session list, chat pane). Against a real store: a session created in the TUI appears in the app without restart and a turn streams live.

## Requirements

### R1: Typed command surface

The backend SHALL register exactly four commands — `list_sessions`, `open_session`, `send_turn`, `cancel_turn` — each returning a typed result whose error side is a serde-serializable projection of the service's typed errors. No stringly errors cross the bridge; no command panics (N3: no `unwrap`/`expect` in the app backend).

**Acceptance:**
- IPC allow-list contains exactly these four commands (N4)
- each command has a test (Tauri command harness or direct fn test) covering success and at least one typed-error path
- lock-contention error from NA-003 R1 surfaces to the frontend as its own error variant (R1.3)

**Files:**
- create: `crates/norn-app/src-tauri/src/commands/…` (module discipline per CLAUDE.md)

### R2: Event channel

The backend SHALL forward NA-002 envelopes for the active session to the window over one Tauri event subscription per window (§3 bridge design). `Lagged` envelopes SHALL be forwarded, not swallowed (NA-002 R3 consumer contract). Subscription lifecycle SHALL be tied to the window: closing the window or switching sessions tears the subscription down (no leaked forwarders).

**Acceptance:**
- test or instrumented assertion: rapid session switch leaves exactly one live forwarder
- lag surfaced end-to-end: a test forces broadcast lag and the frontend store records a gap marker

**Files:**
- create: `crates/norn-app/src-tauri/src/bridge/…`

### R3: Frontend — session list view

The session list view SHALL render enumeration results (id, title, status, lineage indication) from a Zustand store populated via `list_sessions`, refreshing on a store-driven trigger (command re-invocation; no polling loop with an invented interval — refresh on focus and on explicit user action for M0). Selecting a session invokes `open_session` and routes the chat pane to it.

**Acceptance:**
- session created via TUI while app is open appears after refresh without app restart (M0 acceptance, §9 first bullet)
- views contain no IPC calls (MVVM: stores own commands, views render store state)

**Files:**
- create: frontend `stores/sessions.ts`, `views/SessionList…` (per NA-001 layout)

### R4: Frontend — chat pane, honest streaming

The chat pane SHALL render a streamed turn from DTO envelopes: text deltas appended live, tool-call start/completion shown as typed items, stream-retry resets partial output exactly as the DTO contract requires, completion shows factual usage figures (R3.5's numbers, M0 scope: display only). Cancel is one action: invokes `cancel_turn`, renders the preserved partials plus a truthful cancelled marker (R3.3). UI states map 1:1 to runtime states — no spinner theatre (principle 3).

**Acceptance:**
- manual acceptance run (recorded in PR description): real store, real provider, one streamed turn end-to-end; kill mid-stream via cancel; UI shows partials + cancelled state; timeline in `~/.norn` confirms partial preservation
- store test: `Lagged` envelope produces a visible gap marker in rendered state
- store test: retry envelope resets accumulated deltas

**Files:**
- create: frontend `stores/turn.ts`, `views/ChatPane…`

### R5: Gates and M0 acceptance

Workspace battery, clippy `-D warnings`, fmt: green. The M0 milestone definition (§8) SHALL be demonstrably met: shell boots, embeds norn, lists sessions from `~/.norn`, streams one turn end-to-end. Cold-start need not hit N1's <1s in the spike, but SHALL be measured and recorded in the PR description as the baseline for M1/M2 tracking (a number, not an adjective).

**Acceptance:**
- all gates green in CI-equivalent local run
- PR description carries: acceptance-run evidence, cold-start measurement, any engine findings surfaced by NA-002/NA-003 integration

**Files:**
- as needed

## Non-goals

- Tree rendering, timeline view, annotations (M1: R2.x)
- Agents panel, spawn/fork UI (M1/M2: R4.x)
- Auth/account UI (M1: R5.1); the spike uses whatever credentials the store already holds
- Steering/inbound queue (M1: R3.2)
- iOS anything (M3)

## Notes for the worker

- The bridge is the product of this brief. If the DTO subset (NA-002) or service surface (NA-003) proves insufficient mid-implementation, stop and report the gap — the fix lands in the library brief, not as app-side workaround code (O1 is absolute).
- One subscription per window is a spec decision (§3); do not multiplex through a global.
- Keep the frontend dependency set to what NA-001 established; no UI kit additions for a spike.
