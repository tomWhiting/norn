# NA-003: L2 slice — library session service: list, open, drive one turn

> **Cluster:** norn-app
> **Milestone:** M0 (spike)
> **Depends on:** NA-002 (event DTOs exist for the streaming surface)
> **Spec anchors:** L2, R1.1, R1.3, R3.1, R3.3, O1
> **Evidence base:** in-process driver surface exists (`AgentBuilder` `agent/builder.rs:85` → `Agent::into_parts` `agent/instance.rs:155` → `run_agent_step` `loop/runner/entry.rs:188`); multi-turn *policy* is what drivers duplicate (SPEC Appendix A). TUI rotation atomically rewires six dependents (`norn-tui/src/app/rotation.rs:42-175`) — the M1 completion of L2 must land *inside* the seam this brief creates.

## Purpose

Give embedding drivers (the app first, TUI/CLI convergence later) a library-level service for the M0 loop: enumerate the session index, open/resume a session, submit one user turn, stream its events, cancel it. This is the seam L2 completes in M1 (compact/clear/new, atomic rotation, checkpointing); M0 cuts only what the spike needs, but the shape must not preclude the M1 completion — rotation must be able to land inside this service, not beside it.

## Task

Add a public session-service module to `crates/norn` wrapping `AgentBuilder`/`AgentParts`/`run_agent_step` behind an API that NA-004's Tauri backend can consume without touching loop internals. Read paths honour the existing store, index, and inter-process lock semantics exactly as the CLI does.

## Requirements

### R1: Session enumeration

The service SHALL expose an operation returning the session index as structured data: session id, name/title, parent/fork lineage, status, fidelity, generation, and timestamps — the fields R2.1's tree view needs, sourced from the existing index machinery (no parallel index). Reads SHALL honour the inter-process index lock and its deadline semantics (R1.3); a lock-acquisition failure is a typed error, not a panic or a silent empty list.

**Acceptance:**
- test: enumeration over a fixture store returns lineage-correct entries
- test: lock contention path yields the typed error (deadline semantics observed, not reimplemented)
- no new index format, no bypass reads of index files

**Files:**
- create: `crates/norn/src/<service module>/…` (worker proposes module home; `mod.rs` re-exports only)

### R2: Open and drive one turn

The service SHALL open an existing session (or create a new root session) and submit a single user turn, driving the underlying step machinery through the same code path the print orchestrator uses (no forked policy). Events for the turn SHALL be delivered to the caller as NA-002 DTO envelopes (typed subscription handle, one per consumer).

**Acceptance:**
- integration test: fixture store → open → submit turn against a stubbed/replayed provider → DTO stream contains text deltas and turn completion in order → JSONL timeline on disk gained exactly the expected events
- the app-facing API surface contains no `AgentParts` internals (loop types stay crate-internal to the service consumer's view)

**Files:**
- as per module layout above

### R3: Cancellation

The service SHALL expose cancellation of an in-flight turn via the engine's cancellation token, with partial output preserved in the timeline (R3.3). Cancellation SHALL be idempotent and race-safe against natural completion: cancelling a turn that just completed is a no-op with a truthful typed outcome, not an error and not a lie.

**Acceptance:**
- test: cancel mid-stream → turn ends, timeline retains partials, outcome reports cancelled
- test: cancel after completion → typed already-completed outcome

**Files:**
- as per module layout above

### R4: Seam shape holds room for M1 rotation

The service's type SHALL own (or hold exclusive rebindable references to) every store-coupled dependent that TUI rotation rewires (store handle, action log, action-log tree, child infra binding, cache key, prompt environment — per `rotation.rs:42-175`), so that M1's `rotate` operation can swap them behind the service boundary atomically. This brief does NOT implement rotation; it SHALL include a doc comment on the service type naming the six dependents and stating the M1 contract, plus a compile-visible placeholder (e.g. unimplemented typed method returning a `NotYetSupported` error — NOT `todo!()`/panic) so the surface is reserved.

**Acceptance:**
- doc comment enumerates the rotation dependents with the rotation.rs citation
- placeholder method exists, returns typed not-supported error, has a test asserting exactly that
- no `unwrap`/`expect` outside tests; workspace battery, clippy `-D warnings`, fmt green

**Files:**
- as per module layout above

## Non-goals

- compact/clear/new/model/name actions (M1, L2 completion)
- Multi-window / multi-consumer arbitration beyond one subscription handle per consumer
- Any change to TUI or CLI to consume this service (convergence is M1+; drivers align when the seam is complete)
- Fork as a UI action (R2.5, M1)

## Notes for the worker

- Investigate before shaping: read the print orchestrator's turn loop and the TUI's session-open path first; the service must be a *lift*, not a third implementation (O1). Where print and TUI genuinely differ, surface the difference in your report and propose which behaviour is canonical — do not pick silently (no invented defaults).
- Provider stubbing for the integration test should reuse whatever replay/stub infrastructure the workspace battery already uses; introduce nothing bespoke without flagging it.
- If honouring the index lock from a second in-process consumer surfaces a latent engine assumption (single-consumer), stop and report — that is an engine finding, not something to paper over in the service.
