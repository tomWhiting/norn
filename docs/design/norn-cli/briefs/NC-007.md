## Purpose

Users need to see agent output as it happens, not after the step completes. This brief delivers the streaming display renderer that subscribes to the broadcast::Sender<ProviderEvent> channel from run_agent_step and renders text deltas, thinking deltas, tool call summaries, truncated tool results, schema validation feedback, and token usage summaries to the terminal in real time. Without this, the CLI would buffer the entire response before showing anything, violating the streaming-first constraint (CO4).

## Task

Create a display module under print/ (crates/norn-cli/src/print/display.rs) that spawns a tokio task to receive ProviderEvent values from the broadcast channel. NOTE: The crate was reorganised — see DESIGN.md Structure section. Import paths use crate::cli, crate::config, crate::runtime, crate::print, not the old flat module names. TextDelta events are written to stdout inline. ThinkingDelta events are rendered dimmed (ANSI dim escape codes) to stderr. ToolCallDelta events are accumulated per tool call ID until the call is complete (name is known and arguments are fully assembled), then a one-line summary is printed to stderr showing the tool name and key arguments. Tool result content is displayed with head/tail truncation: for output exceeding the configurable line threshold (default 20 lines), show the first 10 and last 10 lines with a '... (N lines omitted)' separator. Schema validation failures (detected from ProviderEvent::Error with schema-related errors) are printed to stderr. On ProviderEvent::Done, a usage summary line is printed to stderr showing input tokens, output tokens, and elapsed wall-clock time.

## Requirements

### R1: Subscribe to ProviderEvent broadcast channel

WHEN run_agent_step is called, the system SHALL subscribe to the broadcast::Sender<ProviderEvent> channel and spawn a tokio task that receives events until the channel closes. The display task SHALL run concurrently with the agent step, rendering events as they arrive without buffering.

**Acceptance:**
- A broadcast::Receiver<ProviderEvent> is obtained via sender.subscribe() before run_agent_step
- The display task processes events concurrently with agent execution
- When the agent step completes and the sender is dropped, the display task exits cleanly

### R2: Text delta streaming to stdout

WHEN a ProviderEvent::TextDelta is received, the system SHALL write the text fragment to stdout immediately and flush. Text deltas SHALL be rendered inline without additional formatting, allowing the model's output to appear character-by-character as it streams.

**Acceptance:**
- Text appears on stdout incrementally as TextDelta events arrive
- stdout is flushed after each TextDelta write
- Multiple TextDelta events concatenate naturally into readable text
- A newline is printed after the final TextDelta to separate from the next prompt

### R3: Thinking delta rendering on stderr

WHEN a ProviderEvent::ThinkingDelta is received, the system SHALL write the text fragment to stderr with ANSI dim styling (ESC[2m prefix, ESC[0m suffix per fragment). Thinking deltas SHALL be visually distinct from text deltas so users can distinguish reasoning from output.

**Acceptance:**
- Thinking text appears on stderr with dimmed styling
- Thinking text is visually distinguishable from regular text output on stdout
- When --quiet is set, thinking deltas are suppressed (not written to stderr)

### R4: Tool call summary display

WHEN ProviderEvent::ToolCallDelta events are received, the system SHALL accumulate the tool call's name and arguments by ID. WHEN the tool call is complete (a subsequent TextDelta, ThinkingDelta, ToolCallDelta with a different ID, or Done event arrives), the system SHALL print a one-line summary to stderr in the format '> {tool_name}: {key_arguments}' where key_arguments is a truncated preview of the JSON arguments (first 120 characters, ellipsized if longer).

**Acceptance:**
- Tool call summary line appears on stderr after tool call deltas complete
- Summary shows tool name and argument preview (e.g., '> bash: find crates/ -name *.rs | wc -l')
- Arguments longer than 120 characters are truncated with '...'
- When --quiet is set, tool call summaries are suppressed

### R5: Tool result head/tail truncation

WHEN a tool result is rendered and its text content exceeds the configurable line threshold (default 20 lines), the system SHALL display the first 10 lines, a separator line '... ({N} lines omitted)', and the last 10 lines. The line counts (total threshold, head count, tail count) SHALL be configurable via a DisplayConfig struct. WHEN output is at or below the threshold, it SHALL be displayed in full.

**Acceptance:**
- Tool output of 5 lines is displayed in full
- Tool output of 50 lines shows first 10, '... (30 lines omitted)', last 10
- Line counts are configurable (not hardcoded)
- Truncation applies per tool result, not globally

### R6: Schema validation feedback on stderr

WHEN a ProviderEvent::Error is received that indicates a schema validation failure (the error relates to structured output schema enforcement), the system SHALL print a diagnostic message to stderr indicating the schema attempt failed and a retry is in progress. The message SHALL include the validation error details.

**Acceptance:**
- Schema validation failure prints a message to stderr (e.g., 'Schema validation failed: missing required field "answer". Retrying...')
- The message includes enough detail for the user to understand what went wrong
- When --quiet is set, schema feedback is suppressed

### R7: Token usage summary after each step

WHEN a ProviderEvent::Done is received, the system SHALL print a usage summary line to stderr showing input tokens, output tokens, and elapsed wall-clock time for the completed step. The format SHALL be '[{input_tokens} in / {output_tokens} out, {elapsed}]' where elapsed is formatted as seconds with one decimal place (e.g., '3.2s') or minutes and seconds for longer durations (e.g., '2m 15.0s').

**Acceptance:**
- After each agent step, a usage line appears on stderr (e.g., '[1,234 in / 567 out, 3.2s]')
- Token counts come from the Usage field in ProviderEvent::Done
- Elapsed time measures wall-clock duration from step start to Done event
- When --quiet is set, the usage summary is suppressed

## Boundaries

- SHALL NOT implement print mode output formatting (text/json/stream-json envelope) -- that is NC-003 or a later brief
- SHALL NOT implement the REPL loop -- that is NC-005; this brief provides the display renderer that the REPL spawns
- SHALL NOT implement tool result content rendering beyond text -- binary or image tool outputs are out of scope
- SHALL NOT implement clickable or interactive elements -- the display is write-only terminal output
- SHALL NOT buffer the full response before display -- streaming-first per constraint CO4
