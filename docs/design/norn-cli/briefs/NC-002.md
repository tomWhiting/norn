## Purpose

Without persistence, every conversation vanishes when the process exits. This brief builds the JSONL-based persistence layer that serializes SessionEvent records to disk, maintains a session index for listing and lookup, and provides resume and fork operations that reconstruct an EventStore from a saved file. This is the storage engine that session subcommands (NC-008) and the REPL's auto-save depend on.

## Task

Implement JSONL serialization and deserialization of SessionEvent records, a session index file, XDG-compliant path resolution (with macOS fallback), an append protocol that flushes events after each run_agent_step call, resume from a JSONL file that reconstructs an EventStore, and fork that copies events into a new session with a Fork event appended. All paths must respect XDG_DATA_HOME and fall back to ~/Library/Application Support/norn/ on macOS when XDG vars are unset.

## Requirements

### R1: XDG-compliant path resolution

WHEN the CLI needs to resolve a data directory for session storage, THE SYSTEM SHALL check XDG_DATA_HOME first and use $XDG_DATA_HOME/norn/sessions/ when set. IF XDG_DATA_HOME is not set and the platform is macOS, THE SYSTEM SHALL fall back to ~/Library/Application Support/norn/sessions/. IF XDG_DATA_HOME is not set and the platform is not macOS, THE SYSTEM SHALL fall back to ~/.local/share/norn/sessions/. WHEN the CLI needs to resolve a config directory, THE SYSTEM SHALL check XDG_CONFIG_HOME first and fall back to ~/Library/Application Support/norn/ on macOS or ~/.config/norn/ otherwise.

**Acceptance:**
- With XDG_DATA_HOME=/tmp/xdg, session_data_dir() returns /tmp/xdg/norn/sessions/
- On macOS without XDG_DATA_HOME, session_data_dir() returns ~/Library/Application Support/norn/sessions/
- On Linux without XDG_DATA_HOME, session_data_dir() returns ~/.local/share/norn/sessions/
- config_dir() respects XDG_CONFIG_HOME in the same pattern
- A unit test verifies each branch with mocked env vars and platform detection

### R2: JSONL serialization of SessionEvent

WHEN the system persists a SessionEvent, THE SYSTEM SHALL serialize it as a single JSON line using serde_json::to_string (no pretty printing) and append it to the session JSONL file followed by a newline character. WHEN the system reads a session file, THE SYSTEM SHALL deserialize each non-empty line as a SessionEvent using serde_json::from_str, collecting them in insertion order. IF a line fails to deserialize, THE SYSTEM SHALL return an error identifying the line number and the parse failure reason.

**Acceptance:**
- A round-trip test serializes 5 different SessionEvent variants to JSONL, reads them back, and verifies equality
- Each line in the file is valid JSON ending with a newline
- A corrupted line at position N produces an error message containing the line number N
- Empty lines are skipped without error

### R3: Session index maintenance

THE SYSTEM SHALL maintain an index file at {session_data_dir}/index.jsonl where each line is a JSON object containing: id (UUID v7 string), name (optional string), model (string), working_dir (string), created_at (ISO 8601 timestamp), updated_at (ISO 8601 timestamp), event_count (u64), and status (active | completed). WHEN a new session is created, THE SYSTEM SHALL append a new index entry. WHEN events are appended to a session, THE SYSTEM SHALL rewrite the index file atomically (write to a .tmp file, then rename) with the updated entry. WHEN a session is removed, THE SYSTEM SHALL rewrite the index file without the removed entry.

**Acceptance:**
- After creating a session and appending 3 events, the index file contains exactly one entry with event_count = 3
- The index file is valid JSONL — each line parses as a SessionIndexEntry
- Atomic write: if the process crashes mid-update, the index file still contains the previous valid state (verified by checking .tmp file is used)
- A test creates 3 sessions and verifies the index lists all 3 with correct metadata

### R4: Append protocol

WHEN run_agent_step completes and produces new SessionEvent records, THE SYSTEM SHALL open the session JSONL file in append mode, serialize each new event as a JSON line, write them in order, and flush the file. THE SYSTEM SHALL then update the session index entry with the new event_count and updated_at timestamp. IF the session file does not exist (first append), THE SYSTEM SHALL create the directory structure and file. IF --no-session is set, THE SYSTEM SHALL skip all persistence operations.

**Acceptance:**
- After a step producing 4 events, the session file contains exactly 4 lines
- A second step producing 2 more events results in a file with 6 lines total (append, not overwrite)
- The index entry's event_count matches the total lines in the session file
- With --no-session, no files are created or modified
- Directory creation is verified by writing to a path where the parent directory does not yet exist

### R5: Resume from JSONL file

WHEN the user specifies --resume with a session ID or name, THE SYSTEM SHALL locate the session in the index, read the corresponding JSONL file, deserialize all events, and construct a new EventStore by appending each event in order. THE SYSTEM SHALL return the populated EventStore and the list of deserialized events so the caller can rebuild the message history for the provider. IF --resume is specified without a value, THE SYSTEM SHALL resolve to the most recently updated session from the index. IF the session ID is a partial prefix (minimum 8 characters), THE SYSTEM SHALL disambiguate against the index and return an error if the prefix matches zero or multiple sessions.

**Acceptance:**
- Resuming a session with 5 events produces an EventStore with len() == 5
- All 5 events are present in EventStore::events() in the original insertion order
- Resuming with no argument resolves to the session with the latest updated_at timestamp
- A partial prefix of 8+ characters matching exactly one session resolves successfully
- A partial prefix matching zero sessions returns an error naming the prefix
- A partial prefix matching two or more sessions returns an error listing the ambiguous IDs

### R6: Fork a session

WHEN the user specifies --fork with a session ID or name, THE SYSTEM SHALL read the source session's events from its JSONL file, create a new session with a fresh UUID v7 ID, write all source events to the new session's JSONL file, then append a Fork event with source_event_id set to the last event's ID and forked_session_id set to the new session ID. THE SYSTEM SHALL create a new index entry for the forked session and continue execution with the new session. IF --fork is specified without a value, THE SYSTEM SHALL resolve to the most recently updated session, following the same resolution rules as --resume.

**Acceptance:**
- Forking a session with 5 events creates a new JSONL file with 6 lines (5 original + 1 Fork event)
- The Fork event's source_event_id matches the last event of the original session
- The Fork event's forked_session_id matches the new session's ID in the index
- The original session file is unmodified after the fork
- The index contains entries for both the original and forked sessions
- Fork with no argument resolves to the most recent session

## Boundaries

- SHALL NOT implement slash commands for session naming (/name) — those are in the REPL brief
- SHALL NOT implement session export to markdown or JSON formats — export is in NC-008
- SHALL NOT add any provider or agent-loop calls — this is pure I/O and data structure work
- SHALL NOT implement REPL integration — the persistence layer exposes functions that the REPL brief will call
