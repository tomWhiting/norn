## Purpose

The CLI needs to wire the DiagnosticCollector into LoopContext and ToolContext, configure per-tool validation thresholds from profiles and CLI overrides, and render diagnostics in all three output formats. N-025 adds the LoopContext field and runner push sites in libnorn. This brief wires them from the CLI side: construct the collector, set it on LoopContext, publish it on ToolContext, load tool_config from profiles, apply -c overrides for tool thresholds, and drain/render at step completion.

## Task

Construct an Arc<DiagnosticCollector> and set it on LoopContext::diagnostics before calling run_agent_step. Publish the same collector on ToolContext via ToolRegistry's shared context (insert_extension). Parse a tool_config section from the profile to configure per-tool thresholds (write.max_code_lines, glob overrides). Support -c write.max_code_lines=N CLI overrides that take precedence over profile values. Construct WriteTool with the resolved LengthLimit (None when no threshold configured). Drain the collector after run_agent_step and render per output format: text mode renders on stderr, json mode includes in the diagnostics envelope field, stream-json mode emits as diagnostic NDJSON events before the completed event.

## Requirements

### R1: Construct and wire DiagnosticCollector

THE SYSTEM SHALL construct an `Arc<DiagnosticCollector>` at CLI startup (before the first `run_agent_step` call). THE SYSTEM SHALL set it on `LoopContext::diagnostics` as `Some(Arc::clone(&collector))`. THE SYSTEM SHALL publish the same collector on the `ToolRegistry`'s shared `ToolContext` via `insert_extension(Arc::clone(&collector))`. THE SYSTEM SHALL retain a clone of the `Arc` for draining after step completion.

**Acceptance:**
- LoopContext::diagnostics is Some when run_agent_step is called
- ToolContext has a DiagnosticCollector available via get_extension after registry construction
- The same collector instance is shared between LoopContext and ToolContext
- The CLI can drain the collector after run_agent_step returns

### R2: Profile tool_config section for per-tool thresholds

THE SYSTEM SHALL parse an optional `tool_config` section from the profile. WHEN `tool_config.write.max_code_lines` is set, THE SYSTEM SHALL construct a `LengthLimit` with `default: Some(value)`. WHEN `tool_config.write.length_overrides` is an array of `{pattern, limit}` objects, THE SYSTEM SHALL add them as glob overrides to the `LengthLimit`. WHEN `tool_config.write` is absent, THE SYSTEM SHALL construct `LengthLimit::none()`. THE SYSTEM SHALL construct `WriteTool::with_length_limit(limit)` using the resolved `LengthLimit`.

**Acceptance:**
- Profile with tool_config.write.max_code_lines = 500 constructs LengthLimit with default Some(500)
- Profile with tool_config.write.length_overrides = [{pattern: 'tests/**', limit: 800}] adds glob override
- Profile without tool_config section constructs LengthLimit::none()
- WriteTool is constructed with the resolved LengthLimit

### R3: CLI -c overrides for tool thresholds

THE SYSTEM SHALL recognise `-c write.max_code_lines=N` as a tool-specific config override. WHEN this override is present, THE SYSTEM SHALL set `LengthLimit.default` to `Some(N)`, taking precedence over the profile value. Unknown `-c write.*` keys SHALL produce a warning and be ignored. THE SYSTEM SHALL NOT accept `-c` tool overrides for tool names that are not registered.

**Acceptance:**
- -c write.max_code_lines=800 sets LengthLimit.default to Some(800)
- -c write.max_code_lines=800 overrides profile value of 500
- -c write.unknown_key=foo produces a warning and is ignored
- Without -c write.max_code_lines, profile value is used; without profile, LengthLimit::none()

### R4: Drain and render diagnostics per output format

AFTER `run_agent_step` returns, THE SYSTEM SHALL drain the `DiagnosticCollector`. IN text mode: non-empty diagnostics SHALL be rendered on stderr with severity, code, message, and suggestion. IN json mode: diagnostics SHALL be included in the output envelope under a `diagnostics` key as an array of objects with severity, code, message, source_tool, file_path, and suggestion fields. IN stream-json mode: diagnostics SHALL be emitted as `{"type":"diagnostic","severity":"...","code":"...","message":"..."}` NDJSON events before the completed event. WHEN diagnostics are empty, no diagnostic output SHALL be produced.

**Acceptance:**
- Text mode with diagnostics renders them on stderr with severity prefix
- Text mode with no diagnostics produces no diagnostic output on stderr
- JSON mode includes diagnostics array (empty when none present)
- Stream-json mode emits diagnostic events before completed event
- Each diagnostic object includes severity, code, and message at minimum
