## Purpose

The CLI's subcommands provide essential non-conversational operations: session management (list, show, resume, fork, export, remove), authentication (login, logout, status), MCP server mode, system health checks, and shell completion generation. Without these, users cannot manage sessions outside the REPL, authenticate, expose Norn as an MCP server, diagnose setup problems, or generate shell completions. Each subcommand is a thin handler that delegates to libnorn APIs.

## Task

Replace the stub subcommand handlers from NC-001 with real implementations, organized as a folder module at crates/norn-cli/src/commands/ with session.rs, auth.rs, mcp.rs, doctor.rs, and completion.rs. Session subcommands read and manipulate the JSONL session store (from NC-002). Auth subcommands delegate to norn::provider::auth::login(), logout(), and read ~/.codex/auth.json for status. MCP serve delegates to norn::integration::mcp_server::McpServer::serve_stdio() with the full tool registry. MCP connect performs an initialize handshake and tools/list against an external server. Doctor checks auth status, provider connectivity, and working directory permissions. Completion uses clap_complete to generate scripts for bash, zsh, and fish.

### R2: Session list subcommand

WHEN the user runs 'norn session list', the system SHALL read the session index and display sessions from the current working directory, sorted newest-first. IF --all is set, the system SHALL show sessions from all directories. IF --limit is set, the system SHALL display at most that many sessions. IF --format is 'json', the system SHALL output a JSON array to stdout. IF --format is 'table' (default), the system SHALL output a formatted table with columns: ID (first 8 characters), name, model, turns, updated, working directory.

**Acceptance:**
- 'norn session list' shows sessions filtered to the current directory
- 'norn session list --all' shows all sessions
- 'norn session list --limit 5' shows at most 5 sessions
- 'norn session list --format json' outputs valid JSON array to stdout
- Sessions are sorted newest-first by updated timestamp
- When no sessions exist, the table output shows a 'No sessions found.' message

### R5: Session fork subcommand

WHEN the user runs 'norn session fork <ID|NAME>', the system SHALL copy the source session's events into a new session file, append a Fork event, and enter the REPL with the forked session. The new session gets a new UUID v7 ID. This is functionally equivalent to 'norn --fork <ID>'.

**Acceptance:**
- 'norn session fork <ID>' creates a new session with copied events and enters the REPL
- The forked session has a new ID distinct from the source
- A Fork event is present in the new session's event list
- The source session is not modified

### R13: Doctor subcommand

WHEN the user runs 'norn doctor', the system SHALL perform the following checks and report pass/fail for each with actionable remediation messages: (1) Auth status: whether OAuth credentials exist and the token is valid. (2) Provider connectivity: attempt an HTTP HEAD or lightweight request to the provider API endpoint to verify reachability. (3) Working directory permissions: verify the current directory is readable and writable. Each check SHALL print a status line to stderr in the format '[PASS] description' or '[FAIL] description: remediation'.

**Acceptance:**
- 'norn doctor' prints status for auth, connectivity, and permissions checks
- Each check shows [PASS] or [FAIL] with a clear description
- [FAIL] lines include actionable remediation advice (e.g., 'Run `norn auth login` to authenticate.')
- Exit code is 0 if all checks pass, 1 if any check fails
