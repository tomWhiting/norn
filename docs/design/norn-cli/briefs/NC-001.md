## Purpose

Establish the norn-cli crate as a compilable workspace member with the full clap argument tree, subcommand stubs, mode detection logic, and exit code conventions. This is the foundation every subsequent brief depends on — nothing can be built until the crate compiles, the binary runs, and the REPL/print mode fork is decided.

## Task

Create a new crate crates/norn-cli/ with a Cargo.toml declaring norn as a dependency, a binary target named 'norn', and a clap-derive argument parser defining the full command tree (main flags + session/auth/mcp/doctor/completion subcommands). Implement mode detection: REPL when both stdin and stdout are TTYs and -p is not set; print mode otherwise. Subcommand handlers are stubs that print 'not yet implemented' and exit. The main command dispatches to either a placeholder REPL loop (read line, echo, repeat) or a placeholder print handler (echo prompt, exit). Exit codes: 0 success, 1 agent error, 2 argument error, 3 auth error.

### R3: Implement mode detection logic

The main function SHALL determine the execution mode as follows: (1) If --print is set: print mode. (2) If stdin is not a TTY (std::io::stdin().is_terminal() returns false): print mode. (3) If stdout is not a TTY: print mode. (4) Otherwise: interactive REPL mode. The mode detection SHALL be a pure function that accepts the parsed CLI args and TTY state and returns an enum Mode { Repl, Print }.

**Acceptance:**
- `norn -p 'hello'` runs in print mode (outputs and exits)
- `echo hello | norn` runs in print mode (piped stdin detected)
- `norn 'hello' | cat` runs in print mode (piped stdout detected)
- `norn` in a TTY starts interactive REPL mode (placeholder loop)
- Mode detection is testable: a unit test verifies the logic with mock TTY state

### R4: Implement placeholder REPL and print handlers

The main function SHALL dispatch to either a REPL handler or a print handler based on mode detection. The REPL handler SHALL read lines from stdin in a loop, echo them, and exit on '/exit' or EOF. The print handler SHALL read the positional PROMPT args, concatenate them, print them to stdout, and exit with code 0. Both are placeholders — subsequent briefs replace them with real implementations.

**Acceptance:**
- `norn` in a TTY accepts input and echoes it back
- `norn` exits on '/exit' input
- `norn -p 'hello world'` prints 'hello world' to stdout and exits 0
- Subcommand stubs print 'not yet implemented' to stderr and exit 1

### R5: Define exit code conventions

The CLI SHALL use typed exit codes: 0 for success, 1 for agent errors (provider failure, tool error, schema unreachable), 2 for CLI argument errors (clap handles this automatically), 3 for authentication errors. An ExitCode enum SHALL be defined with From<ExitCode> for std::process::ExitCode.

**Acceptance:**
- `norn -p 'hello'` exits with code 0
- `norn --invalid-flag` exits with code 2 (clap default)
- ExitCode enum has Success(0), AgentError(1), ArgumentError(2), AuthError(3) variants
- A unit test verifies the numeric values
