## Purpose

Print mode is the headless execution path that makes Norn usable in shell pipelines, automation, and orchestration workflows. This brief wires the full path from CLI arguments through provider construction, runtime component assembly (token estimator, context edits, retry policy, diagnostics), stdin handling, agent execution via run_agent_step, and formatted output in text/json/stream-json modes. Without this, Norn cannot execute a prompt and produce output — it is the first brief that actually calls libnorn.

## Task

Implement provider construction (OpenAiProvider default with OAuth, ClaudeRunnerAdapter via --provider flag), runtime wiring (SimpleTokenEstimator into LoopContext::token_estimator, ContextEdits::new() into LoopContext::context_edits, RetryPolicy from -c overrides, DiagnosticCollector, IterationMonitorConfig from profile), stdin handling (auto-detect piped stdin, read all, prepend with <stdin> delimiters when positional PROMPT also present), output schema parsing (inline JSON detection by leading {, else file path), per-event schema parsing (--event-schema TYPE=JSON|PATH), and three output formatters: text (final output to stdout, progress to stderr), json (single envelope to stdout), stream-json (NDJSON events to stdout). Also implement -o file output and -q quiet mode.

## Requirements

### R1: Provider construction — OpenAI default

WHEN the CLI executes in print mode and no --provider flag is set or --provider is 'openai', THE SYSTEM SHALL construct an OpenAiProvider by building a ProviderConfig with auth_source set to AuthSource::OAuth { auth_root: None }, base_url derived from -c base_url override or None (letting OpenAiProvider auto-detect from auth source), timeout from -c request_timeout override or a default of 120 seconds, max_retries from -c max_retries override or a default of 2, and provider_options from -c provider_options override parsed as JSON or None. THE SYSTEM SHALL call OpenAiProvider::new(config).await and propagate any ProviderError as exit code 3 for auth failures or exit code 1 for other provider errors.

**Acceptance:**
- With no flags, provider is constructed with AuthSource::OAuth { auth_root: None }
- With -c base_url=http://localhost:8080, ProviderConfig.base_url is Some("http://localhost:8080")
- With -c max_retries=5, ProviderConfig.max_retries is 5
- With -c request_timeout=30s, ProviderConfig.timeout is Duration::from_secs(30)
- With -c provider_options='{"key":"val"}', ProviderConfig.provider_options is Some(json)
- AuthenticationFailed errors produce exit code 3

### R4: Stdin handling — auto-detect and prepend with delimiters

WHEN stdin is not a TTY (detected via std::io::IsTerminal), THE SYSTEM SHALL read all content from stdin before execution begins. IF positional PROMPT arguments are also provided, THE SYSTEM SHALL construct the effective prompt by prepending the stdin content wrapped in <stdin> delimiters: '<stdin>\n{stdin_content}\n</stdin>\n\n{positional_prompt}'. IF no positional PROMPT is provided, THE SYSTEM SHALL use the stdin content as the prompt directly. WHEN stdin is a TTY, THE SYSTEM SHALL NOT read from stdin in print mode.

**Acceptance:**
- echo 'data' | norn -p 'Summarise' produces effective prompt '<stdin>\ndata\n</stdin>\n\nSummarise'
- echo 'data' | norn -p (no positional prompt) uses 'data' as the prompt
- In a TTY with -p 'hello', stdin is not read
- Large stdin input (1MB) is read in full before execution
- A unit test verifies delimiter formatting with both stdin and positional prompt present

### R5: Output schema parsing — inline JSON and file path

WHEN -s / --output-schema is provided with a value starting with '{', THE SYSTEM SHALL parse the value as inline JSON using serde_json::from_str and use the parsed Value as the output schema for run_agent_step. WHEN -s is provided with a value not starting with '{', THE SYSTEM SHALL treat it as a file path, read the file, and parse its contents as JSON. IF parsing fails (inline or file), THE SYSTEM SHALL return an error and exit with code 2. WHEN --event-schema TYPE=VALUE is provided, THE SYSTEM SHALL parse TYPE as an EventType variant (assistant_message, spoken_response, tool_call_envelope, stop_output, question, handoff, review, progress), parse VALUE as inline JSON (starts with '{') or file path, and build an EventSchemaSet with the per-type schemas. THE SYSTEM SHALL merge profile event schemas with CLI --event-schema flags, with CLI flags taking precedence.

**Acceptance:**
- -s '{"type":"object"}' parses as inline JSON schema
- -s schema.json reads and parses the file at schema.json
- -s '{invalid' exits with code 2 and an error message
- --event-schema spoken_response=tts.json reads the file and maps to EventType::SpokenResponse
- --event-schema assistant_message='{"type":"object"}' parses inline JSON for EventType::AssistantMessage
- Profile event schemas are present in the final EventSchemaSet when no CLI override exists for that type
- CLI --event-schema overrides profile event schemas for the same EventType

### R6: Output format — text mode

WHEN --output-format is 'text' or not specified, THE SYSTEM SHALL write the final model output to stdout after run_agent_step completes. IF an output schema was set and structured output was produced, THE SYSTEM SHALL pretty-print the JSON value. WHILE the agent step is executing, THE SYSTEM SHALL write tool call summaries and progress information to stderr. IF -q / --quiet is set, THE SYSTEM SHALL suppress all stderr output. IF -o / --output is set, THE SYSTEM SHALL write the final output to the specified file path instead of stdout. WHEN diagnostics are present after step completion, THE SYSTEM SHALL render them on stderr with severity, code, message, and suggestion fields.

**Acceptance:**
- norn -p 'hello' writes the model's text output to stdout
- norn -p -s '{"type":"object"}' 'extract' pretty-prints the structured JSON output
- Tool progress appears on stderr, not stdout
- norn -p -q 'hello' produces no stderr output
- norn -p -o out.txt 'hello' writes to out.txt
- Diagnostics with severity=warning appear on stderr in text mode

### R7: Output format — json mode

WHEN --output-format is 'json', THE SYSTEM SHALL write a single JSON object to stdout after run_agent_step completes, containing: output (the model output value), usage (object with input_tokens and output_tokens), model (string), session_id (UUID string or null if --no-session), events (array of SessionEvent objects from the step), result (string: 'completed', 'schema_unreachable', 'max_iterations', or 'timed_out'), and diagnostics (array of diagnostic objects, empty if none). THE SYSTEM SHALL NOT write any other output to stdout. IF -o is set, THE SYSTEM SHALL write the JSON envelope to the file.

**Acceptance:**
- --output-format json produces a single JSON object on stdout
- The JSON object contains all required fields: output, usage, model, session_id, events, result, diagnostics
- usage.input_tokens and usage.output_tokens are integers
- result is 'completed' on successful completion
- result is 'schema_unreachable' when AgentStepResult::SchemaUnreachable is returned
- diagnostics array is empty when no diagnostics are collected
- diagnostics array contains objects with severity, code, message when present
- With --no-session, session_id is null

### R8: Output format — stream-json mode

WHEN --output-format is 'stream-json', THE SYSTEM SHALL subscribe to the broadcast::Sender<ProviderEvent> channel and write NDJSON events to stdout as they arrive: {"type":"text_delta","text":"..."} for TextDelta events, {"type":"tool_call","id":"...","name":"...","arguments":"..."} for complete tool calls, {"type":"tool_result","id":"...","output":{...}} for tool results, {"type":"thinking_delta","text":"..."} for ThinkingDelta events, and {"type":"completed","output":{...},"usage":{...}} as the final event. WHEN diagnostics are present, THE SYSTEM SHALL emit {"type":"diagnostic",...} events before the completed event. IF -q is set, THE SYSTEM SHALL still write all events to stdout (quiet only affects stderr in text mode).

**Acceptance:**
- --output-format stream-json produces one JSON object per line on stdout
- TextDelta events appear as {"type":"text_delta","text":"..."}
- The final line is a completed event with output and usage
- Each line is valid JSON parseable by serde_json::from_str
- Events arrive incrementally — not buffered until completion
- Diagnostic events appear before the completed event when present

### R9: Print mode orchestration — end-to-end execution path

WHEN the CLI determines print mode via mode detection, THE SYSTEM SHALL execute the following sequence: (1) read stdin if piped and construct effective prompt, (2) parse output schema if -s is set, (3) construct the provider via R1/R2, (4) wire the LoopContext via R3 and NC-004's config/profile integration, (5) construct an EventStore (empty for new session, populated for --resume/--fork via NC-002), (6) call run_agent_step with the provider, executor, store, prompt, tools, schema, model, config, event broadcast sender, and loop context, (7) handle the AgentStepResult by routing to the appropriate output formatter (R6/R7/R8), (8) persist session events via NC-002 (unless --no-session), (9) exit with the appropriate exit code (0 for Completed, 1 for SchemaUnreachable/MaxIterationsReached/TimedOut). THE SYSTEM SHALL construct a broadcast::Sender<ProviderEvent> and pass it to run_agent_step for streaming output in stream-json mode.

**Acceptance:**
- norn -p 'hello' executes the full path and produces output on stdout
- norn -p --output-format json 'hello' produces a JSON envelope
- norn -p --output-format stream-json 'hello' produces NDJSON
- Exit code is 0 on Completed, 1 on SchemaUnreachable
- Session events are persisted after execution (unless --no-session)
- The broadcast sender is created and passed to run_agent_step

## Boundaries

- SHALL NOT implement the interactive REPL — this is print mode only
- SHALL NOT implement slash command processing — that is a REPL brief concern
- SHALL NOT implement streaming display for REPL mode (dimmed thinking, tool summaries) — those are REPL brief concerns
- SHALL NOT implement session subcommands (list, show, export, remove) — those are in NC-008
- SHALL NOT implement reedline integration — print mode reads stdin once and exits
- SHALL NOT hardcode rate limits, token caps, or context window defaults — all configurable values come from flags, -c overrides, or profile
