## Purpose

Define the rules engine data model and trigger evaluation logic. Rules are contextual guidance that fires based on conditions — path globs, tool invocations, command matches — and delivers content via one of three modes. The rules engine is a passive consumer: it does not modify prompt construction. It consumes generic path-change and tool-invocation events, tracks which rules are currently in context via a presence set rebuilt from content tags, and produces injection requests. This brief covers the types, parsing, trigger evaluation, delivery mode definitions, and lifecycle tracking mechanism. It does not implement the shell execution feature for dynamic rule content (deferred to N-008 if scoped separately).

## Task

Define Rule struct with RuleId, trigger conditions, delivery mode, timing (before/after), and body content. Implement rule file parsing from YAML front matter (trigger globs, command patterns, delivery mode, timing) plus plain text body. Implement TriggerCondition evaluation against RuntimeEvent (path-change and tool-invocation events). Define DeliveryMode enum with three variants matching D7. Define RulePresenceSet that tracks which rules are currently in the active context, rebuilt synchronously from content tags during prompt construction. Define RuleInjection as the output type representing a rule ready for delivery. Testing layer: unit tests (Layer 1) — all components testable with synthetic events, no provider needed.

- Rule derives Clone, Debug, Serialize, Deserialize

### R2: Define TriggerCondition enum

TriggerCondition SHALL be an enum with variants for path glob matches and command/tool invocation matches. Path glob triggers match against file paths on read or write events. Command triggers match against bash command strings. Tool triggers match against tool invocation names. Each variant SHALL carry its match pattern as a string.

**Acceptance:**
- TriggerCondition enum has variants: PathGlob { pattern: String }, BashCommand { pattern: String }, ToolInvocation { tool_name: String }
- TriggerCondition derives Clone, Debug, Serialize, Deserialize
- PathGlob pattern is a glob pattern (supports *, **, ?)
- BashCommand pattern is a substring match pattern for command matching
- ToolInvocation tool_name is an exact tool name match

### R4: Define RuntimeEvent for trigger evaluation

The rules module SHALL define a RuntimeEvent enum representing the generic events that the rules engine consumes. The rules engine SHALL NOT subscribe to rule-specific events — it consumes generic path-change and tool-invocation events per D7's dependency inversion principle.

**Acceptance:**
- RuntimeEvent enum has variants: PathChanged { path: String, operation: PathOperation }, ToolInvoked { tool_name: String, arguments: Option<serde_json::Value> }, BashCommandRun { command: String }
- PathOperation enum has variants: Read, Write
- RuntimeEvent derives Clone, Debug
- No rule-specific event types — the rules engine is a passive consumer of generic events

### R5: Implement rule file parsing from YAML front matter

The parser module SHALL parse rule files consisting of YAML front matter (delimited by ---) followed by a plain text body. The front matter SHALL specify trigger conditions (path globs, command patterns, tool names), delivery mode, timing, and an optional rule name. The parser SHALL return a Result<Rule, RulesError>.

**Acceptance:**
- parse_rule_file function signature: fn parse_rule_file(id: RuleId, content: &str) -> Result<Rule, RulesError>
- YAML front matter is delimited by --- on its own line at the start and end of the header
- Front matter supports keys: name (optional String), triggers (array of trigger objects), delivery (string: system_context, context_injection, message), timing (string: before, after — defaults to before)
- Trigger objects in YAML have a type field (path_glob, bash_command, tool) and a pattern field

### R6: Implement trigger evaluation against runtime events

The triggers module SHALL implement evaluation of TriggerConditions against RuntimeEvents. PathGlob triggers SHALL match using glob pattern matching against file paths. BashCommand triggers SHALL match using substring containment against command strings. ToolInvocation triggers SHALL match using exact tool name equality. Evaluation SHALL return whether the trigger matched and the timing (before/after).

**Acceptance:**
- evaluate_triggers function signature: fn evaluate_triggers(rule: &Rule, event: &RuntimeEvent) -> Option<TriggerMatch>
- TriggerMatch struct has fields: rule_id (RuleId), timing (TriggerTiming), delivery (DeliveryMode)
- PathGlob { pattern: "**/*.rs" } matches RuntimeEvent::PathChanged { path: "src/main.rs", operation: Read }
- PathGlob { pattern: "**/mod.rs" } does not match RuntimeEvent::PathChanged { path: "src/main.rs", operation: Read }
- BashCommand { pattern: "cargo test" } matches RuntimeEvent::BashCommandRun { command: "cargo test --workspace" }
- ToolInvocation { tool_name: "Write" } matches RuntimeEvent::ToolInvoked { tool_name: "Write", .. }
- ToolInvocation { tool_name: "Write" } does not match RuntimeEvent::ToolInvoked { tool_name: "Read", .. }
- Cargo.toml includes glob = { workspace = true } for glob pattern matching
- A unit test exercises all three trigger types with matching and non-matching events

### R7: Define RulePresenceSet for lifecycle tracking

The lifecycle module SHALL define a RulePresenceSet that tracks which rules are currently present in the active context. The presence set SHALL be rebuilt synchronously from content tags during each prompt construction pass. WHEN a rule was previously in context but is absent from the current tag set, AND the rule's trigger fires again, the rules engine SHALL produce a re-injection request. WHEN a rule is already in context, the engine SHALL NOT re-inject it.

**Acceptance:**
- RulePresenceSet struct wraps HashSet<RuleId>
- rebuild method signature: fn rebuild(&mut self, tags: &[ContentTag]) — extracts RuleIds from ContentTag::Rule variants
- is_present method signature: fn is_present(&self, id: &RuleId) -> bool
- Rebuilding from tags containing ContentTag::Rule("rust-conventions") marks that rule as present
- Rebuilding from tags NOT containing a previously present rule marks it as absent
- A unit test: build presence set with rule A present, rebuild without rule A, verify is_present returns false

**Files:**
- modify: crates/norn/src/rules/lifecycle.rs

**Checklist:**

### R8: Define RuleEngine and RuleInjection output

The rules module SHALL define a RuleEngine struct that holds loaded rules and the presence set. WHEN a RuntimeEvent is submitted, the engine SHALL evaluate all rule triggers, check the presence set, and return a list of RuleInjection values for rules that matched and are not already in context. RuleInjection SHALL carry the rule ID, delivery mode, timing, and the rule body content.

**Acceptance:**
- RuleEngine struct has fields: rules (Vec<Rule>), presence (RulePresenceSet)
- RuleEngine::new accepts Vec<Rule>
- RuleEngine::add_rule appends a rule
- process_event method signature: fn process_event(&self, event: &RuntimeEvent) -> Vec<RuleInjection>
- RuleInjection struct has fields: rule_id (RuleId), delivery (DeliveryMode), timing (TriggerTiming), content (String)
- process_event returns empty vec when no triggers match
- process_event returns empty vec when a trigger matches but the rule is already in context (presence set)
- process_event returns a RuleInjection when a trigger matches and the rule is not in context
- A unit test: add a rule, process a matching event twice — first returns an injection, call rebuild with the rule present, second returns empty
