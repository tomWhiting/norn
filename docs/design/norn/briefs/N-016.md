## Purpose

Wire Norn into the broader Meridian ecosystem. Claude Runner integration maps agent steps to Claude's legitimate subscription path. MCP client connects to external tool servers. MCP server exposes Norn tools to any MCP consumer. Rhai builtins expose agent operations for workflow scripting. Lifecycle hooks provide pre/post interception points. Extensions integrate via Meridian extension protocol. Session variables and diagnostics complete the integration surface.

## Task

Implement the seven integration/ modules: claude.rs (Claude Runner integration producing StepOutcome), mcp_client.rs (connect to MCP servers, register tools dynamically), mcp_server.rs (expose Norn tools via MCP protocol), rhai.rs (Rhai host functions: blocking read_file/run_cmd + handle-returning spawn_agent/send_message), hooks.rs (lifecycle hooks for pre/post tool and LLM calls), variables.rs (session variables with shell execution), diagnostics.rs (diagnostic-grade reporting of tool failures and schema violations). Testing: Layer 1 unit tests, Layer 2 mock integration tests.

### R5: Rhai builtins for agent operations

Rhai builtins SHALL expose agent operations for workflow scripting. Host functions split into blocking (read_file, run_cmd, parse_json, to_json, write_json) and handle-returning (spawn_agent, send_message, wait_agent) categories per grilling refinement.

**Acceptance:**
- register_norn_builtins(engine, context) registers all host functions on a Rhai Engine
- Blocking functions: read_file(path) -> String, run_cmd(cmd) -> String, read_json(path) -> Dynamic, parse_json(s) -> Dynamic, to_json(val) -> String, write_json(path, val)
- Handle-returning functions: spawn_agent(config) -> AgentHandle, send_message(to, content) -> (), wait_agent(handle) -> Dynamic, fork_agent(config) -> Dynamic
- AgentHandle is an opaque Rhai type wrapping a Uuid
- All handle-returning functions access AgentRegistry and Mailbox via shared context
- StepOutcome bridge: Rhai Dynamic values converted to serde_json::Value for validation
- Test: register builtins, run Rhai script calling read_file on temp file, verify content

### R10: Rules shell execution for dynamic content

Rules with a shell source SHALL execute a shell command at injection time and use the output as the rule content. This enables rules that include dynamic context (team status, task lists, file trees) computed fresh each time the rule fires.

**Acceptance:**
- Rule struct (from N-007) gains an optional shell_source field: Option<String>
- When shell_source is set, the rule engine executes the command via tokio::process::Command
- Command output replaces the rule's static content at injection time
- Shell execution has a configurable timeout (default 5s) — timeout produces a diagnostic, not a panic
- Non-zero exit code produces a diagnostic warning and uses the static content as fallback
- Test: create rule with shell_source 'echo hello', fire it, verify injected content is 'hello'

## Boundaries

- SHALL NOT implement Claude Runner binary — only the adapter that calls it
- SHALL NOT implement MCP protocol from scratch — use an existing MCP crate or define the JSON-RPC interface
- SHALL NOT implement Rhai language features — only register host functions on the existing rhai crate's Engine
- SHALL NOT implement diagnostic rendering/formatting — only data structures and collection
- SHALL NOT embed QuickJS, Lua, or Deno — extensions are external processes per D12
- SHALL NOT implement Meridian extension SDKs — only the Norn-side discovery and proxy
- SHALL NOT add todo!() or unimplemented!() macros anywhere
