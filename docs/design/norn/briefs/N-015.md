## Purpose

Implement the tools that agents use to coordinate with other agents, manage tasks, activate skills, run scripts, search the tool catalog, and apply patches. Agent coordination tools (SpawnAgent, Fork, SendMessage/WaitAgent/CloseAgent) wrap the infrastructure from N-014. Standalone tools (Task, Skill, RunScript, ToolSearch, ApplyPatch) are self-contained and only depend on the Tool trait from N-006.

## Task

Implement 8 tools across 6 files in tools/: agent.rs (SpawnAgent, Fork, SendMessage, WaitAgent, CloseAgent — 5 tools wrapping N-014 infrastructure), task.rs (Task tool with CRUD ops), skill.rs (Skill tool loading prompt templates), script.rs (RunScript executing inline Rhai), tool_search.rs (ToolSearch with BM25), patch.rs (ApplyPatch applying unified diffs). All implement the Tool trait from N-006. Testing: Layer 1 unit tests.

## Requirements

### R1: SpawnAgent tool

SpawnAgent tool SHALL create a sub-agent by calling AgentRegistry::reserve and then launching the agent loop. The tool returns when the sub-agent completes, providing its structured output.

**Acceptance:**
- SpawnAgentTool implements Tool with name 'spawn_agent', effect Process
- Input schema: { task: String, model: String, role: String, tools: Option<Vec<String>>, path: Option<String> }
- Tool accesses AgentRegistry via Arc from ToolContext runtime_args
- Calls registry.reserve(), spawns agent loop via tokio::spawn, returns SpawnGuard handle
- Output includes agent_id, result (sub-agent's structured output), usage
- Error when registry is not available in context
- Test with MockProvider: spawn sub-agent, verify registry entry created and cleaned up

**Files:**
- modify: crates/norn/src/tools/agent.rs
- modify: crates/norn/src/tools/mod.rs

### R3: SendMessage, WaitAgent, CloseAgent tools

Three coordination tools: SendMessage sends to another agent's mailbox, WaitAgent blocks until a specific agent completes, CloseAgent requests an agent to wrap up.

**Acceptance:**
- SendMessageTool implements Tool with name 'send_message', effect Process
- SendMessage input: { to: String, content: Value, trigger_turn: Option<bool> }
- SendMessage calls Mailbox::send with the sender's agent ID from context
- WaitAgentTool implements Tool with name 'wait_agent', effect Process
- WaitAgent input: { agent_id: String, timeout: Option<u64> }
- WaitAgent polls registry until agent status is Completed or Failed, with configurable timeout
- CloseAgentTool implements Tool with name 'close_agent', effect Process
- CloseAgent input: { agent_id: String, reason: Option<String> }
- CloseAgent sets target agent status to Completing
- Test: send message, verify it arrives in target mailbox

- List action supports filtering by status

### R6: RunScript tool for inline Rhai execution

RunScript tool SHALL execute inline Rhai scripts with sandboxed host functions. This is lightweight ad-hoc scripting, not heavy orchestration. Host functions split into blocking (read_file, run_cmd) and handle-returning (spawn_agent) categories per grilling refinement.

**Acceptance:**
- RunScriptTool implements Tool with name 'run_script', effect Process
- Input schema: { script: String, args: Option<Value> }
- Creates a Rhai Engine with registered host functions
- Blocking host functions: read_file(path) -> String, run_cmd(cmd) -> String, parse_json(s) -> Dynamic, to_json(val) -> String
- Script args injected as a Rhai scope variable named 'args'
- Script return value serialized to serde_json::Value via serde_json (StepOutcome bridge pattern)
- Script execution has a configurable timeout (default 30s)
- Cargo.toml adds rhai crate dependency
- Test: run script that reads a temp file, verify output matches file content

### R7: ToolSearch with BM25 ranking

ToolSearch tool SHALL perform BM25 keyword search over the tool catalog (name + description) and return ranked results. This is useful when large tool registries exist, especially MCP servers with many tools.

**Acceptance:**
- ToolSearchTool implements Tool with name 'tool_search', effect ReadOnly
- Input schema: { query: String, max_results: Option<u32> }
- BM25 scoring computed over tool name and description text
- Returns ranked list of { name, description, score } objects
- Default max_results is 10
- Empty query returns all tools sorted alphabetically
- Test: register 5 tools with varied descriptions, search for keyword, verify top result is most relevant

**Files:**
- modify: crates/norn/src/tools/tool_search.rs
- modify: crates/norn/src/tools/mod.rs

**Checklist:**
- C55 — ToolSearch tool performs BM25 search over the tool catalog

### R8: ApplyPatch tool for unified diffs

ApplyPatch tool SHALL apply unified diff patches to files. Post-validate runs AST validation via tree-sitter (same mechanism as Edit tool from N-010) using Gate semantics — if AST breaks, the patch is rolled back.

**Acceptance:**
- ApplyPatchTool implements Tool with name 'apply_patch', effect Write
- Input schema: { patch: String, working_dir: Option<String> }
- Parses unified diff format (---/+++ headers, @@ hunks)
- Applies hunks to target files, respecting context lines for positioning
- post_validate_mode returns Gate (same as Edit tool)
- Post-validate calls the same AST validation used by Edit (tree-sitter parse, check for ERROR nodes)
- pre_validate checks all target files exist and have been read (same read-tracking as Edit)
- Returns diff summary: files modified, hunks applied, lines added/removed
- Test: create a file, apply a valid patch, verify file content changed correctly
- Test: apply a patch with bad context lines, verify error returned

**Files:**
- modify: crates/norn/src/tools/patch.rs
- modify: crates/norn/src/tools/mod.rs

## Boundaries

- SHALL NOT implement tree-sitter grammar loading — reuse the grammar infrastructure from N-010
- SHALL NOT implement heavy Rhai orchestration (workflow engine) — RunScript is lightweight ad-hoc only
- SHALL NOT implement semantic/vector search in ToolSearch — BM25 keyword only
- SHALL NOT implement handle-returning Rhai host functions (spawn_agent) — those require N-014 integration and are deferred to N-016
- SHALL NOT add todo!() or unimplemented!() macros anywhere
