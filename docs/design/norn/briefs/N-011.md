# N-011: Wire ToolExecutor, iteration monitors, and inbound channels into agent loop

> **Cluster:** norn
> **Depends on:** N-005, N-006, N-008, N-009
> **Checklist:**
> - C33 — Effect-based parallel scheduling runs read-only tools concurrently, serializes write tools
> - C35 — Dynamic tool availability changes available tools based on profile, workflow stage, or task state
> - C36 — ToolRegistry supports adding, removing, and querying tools at runtime
> **Stories:**
> - S3 (AI Agent, Executing a Workflow Step) — As an AI agent, I want to receive inbound messages and diagnostics at tool boundaries so that I can adjust my approach without being interrupted mid-thought.
> - S4 (AI Agent, Executing a Workflow Step) — As an AI agent approaching a token budget, I want the runtime to tell me to wrap up and summarize progress so that I can hand off cleanly to a continuation.
> - S22 (Human Developer, Orchestrating Agents via Workflows) — As a workflow developer, I want to configure which tools an agent has access to based on its profile so that I can enforce role discipline (e.g., team leads cannot edit files).
> - S26 (Human Developer, Steering a Running Agent) — As a human operator, I want to send a steering message to a running agent that gets delivered at the next tool boundary so that I can redirect without interrupting.

## Purpose

Connect the independently-built mechanisms into the running agent loop. ToolRegistry implements ToolExecutor. Iteration monitors called during the loop. Inbound channels drained at tool boundaries. Dynamic tool availability wired to profile configuration.

## Task

Implement three integration points plus dynamic availability: (1) ToolRegistry implements ToolExecutor dispatching through full lifecycle. (2) Wire evaluate_iteration into loop cycle. (3) Wire inbound channel drain into tool boundaries. (4) Dynamic tool set management. Testing: Layer 2 mock provider tests.

## Requirements

### R1: ToolRegistry implements ToolExecutor

ToolRegistry SHALL implement the ToolExecutor trait, dispatching through pre-validate/execute/post-validate/on-success with gate/report handling.

**Acceptance:**
- ToolRegistry implements ToolExecutor
- Pre-validate Block prevents execution
- Post-validate Gate failure returns error without committing
- Post-validate Report failure returns output with errors
- On-success runs only when post-validate passes
- ToolNotFound for unregistered tools

**Files:**
- modify: crates/norn/src/tool/registry.rs
- modify: crates/norn/src/tool/mod.rs

**Checklist:**
- C33 — Effect-based parallel scheduling runs read-only tools concurrently, serializes write tools

### R2: Wire iteration monitors into agent loop

The loop SHALL call evaluate_iteration after each cycle. Signals appended as system events. HandoffTriggered injects wrap-up guidance.

**Acceptance:**
- evaluate_iteration called once per loop iteration
- TokenWarning appends system event
- HandoffTriggered injects wrap-up UserMessage
- Test triggers a token warning and verifies event in store

**Files:**
- modify: crates/norn/src/loop/runner.rs

**Stories:**
- S4 (AI Agent, Executing a Workflow Step) — As an AI agent approaching a token budget, I want the runtime to tell me to wrap up and summarize progress so that I can hand off cleanly to a continuation.

### R3: Wire inbound channels into tool boundaries

The loop SHALL drain the inbound channel after each tool batch. Steer messages injected before next provider call. Follow-up messages held until model would stop.

**Acceptance:**
- Inbound channel drained after tool batch
- Steer messages become UserMessage events before next call
- Follow-up messages injected only when loop would return Completed
- Test sends Steer mid-loop, verifies in next context
- Test sends Follow-up, verifies loop continues

**Files:**
- modify: crates/norn/src/loop/runner.rs

**Stories:**
- S3 (AI Agent, Executing a Workflow Step) — As an AI agent, I want to receive inbound messages and diagnostics at tool boundaries so that I can adjust my approach without being interrupted mid-thought.
- S26 (Human Developer, Steering a Running Agent) — As a human operator, I want to send a steering message to a running agent that gets delivered at the next tool boundary so that I can redirect without interrupting.

### R4: Dynamic tool availability

ToolRegistry SHALL support set_available to restrict active tools without removing them.

**Acceptance:**
- set_available(names) restricts get() results
- Unavailable tools return ToolNotFound
- reset_available() restores all
- Test: register 3, set available 1, verify only 1 accessible

**Files:**
- modify: crates/norn/src/tool/registry.rs

**Checklist:**
- C35 — Dynamic tool availability changes available tools based on profile, workflow stage, or task state

**Stories:**
- S22 (Human Developer, Orchestrating Agents via Workflows) — As a workflow developer, I want to configure which tools an agent has access to based on its profile so that I can enforce role discipline (e.g., team leads cannot edit files).

### R5: ToolRegistry runtime mutation

ToolRegistry SHALL support adding and removing tools at runtime.

**Acceptance:**
- register() adds immediately available tool
- remove() returns Option<Box<dyn Tool>>
- get() after remove returns None
- Test: add tool at runtime, verify execution

**Files:**
- modify: crates/norn/src/tool/registry.rs

**Checklist:**
- C36 — ToolRegistry supports adding, removing, and querying tools at runtime
