## Purpose

The tool validation pipeline produces diagnostics (AST errors, length violations, post-validate failures) but has no way to surface them to the orchestrator beyond the individual tool result. The DiagnosticCollector exists but is not wired into LoopContext or the runner. File length counting uses naive non-blank line counting instead of tokei. The LengthLimit has a hardcoded default that should be caller-configured. PostValidateMode is fixed per tool type instead of configurable at runtime. This brief closes all of these gaps so that any tool — not just Write/Edit/Patch — can participate in a unified diagnostics pipeline with structured output and configurable strictness.

## Task

Add DiagnosticCollector as an optional field on LoopContext. Wire the runner to push NornDiagnostic at three sites (schema validation failure, pre-validate block, post-validate failure). Replace naive line counting in Write tool with tokei-based code-line counting via the diagnostics crate's count_code_lines(). Make LengthLimit.default an Option<usize> where None means no check runs. Make PostValidateMode overridable at runtime via ToolContext flags (RejectBrokenAst promotes Report to Gate, AllowBrokenAst demotes Gate to Report — extend this pattern so any tool's mode is adjustable). Update tool output format to include structured diagnostic JSON. Publish the DiagnosticCollector on ToolContext typed infrastructure so RuntimePostValidateCheck implementations can push diagnostics directly.

## Requirements

### R1: DiagnosticCollector on LoopContext

THE SYSTEM SHALL add a field `pub diagnostics: Option<Arc<DiagnosticCollector>>` to `LoopContext`. WHEN the field is `None`, no diagnostic collection occurs. WHEN `Some`, diagnostics are pushed into the collector at sites specified in R2. THE SYSTEM SHALL update `LoopContext::new()` to set `diagnostics: None` and update `LoopContext::default()` correspondingly. THE SYSTEM SHALL add tests verifying the field round-trips correctly.

**Acceptance:**
- LoopContext::new('base').diagnostics is None
- LoopContext::default().diagnostics is None
- Setting diagnostics to Some(Arc::new(DiagnosticCollector::new())) and draining it returns the correct diagnostics
- Existing tests still pass with no behavioral change

### R2: Runner pushes diagnostics at three sites

WHEN `loop_context.diagnostics` is `Some`, THE SYSTEM SHALL push a `NornDiagnostic` at three sites in the agent loop: (1) Schema validation failure — after schema validation fails during the retry loop, push via `NornDiagnostic::from_schema_error()`. (2) Pre-validate block — when `ToolRegistry::execute` returns `ToolError::PreValidationFailed`, push via `NornDiagnostic::from_tool_block()`. (3) Post-validate failure — when `ToolRegistry::execute` returns `ToolError::PostValidationFailed`, push via `NornDiagnostic::from_tool_error()`. THE SYSTEM SHALL NOT alter the agent loop's control flow — diagnostics are observational, not behavioral.

**Acceptance:**
- Schema validation failure pushes a diagnostic with code 'schema-violation' and severity Error
- Pre-validate block pushes a diagnostic with code 'tool-blocked' and severity Warning
- Post-validate failure pushes a diagnostic with code matching the ToolError variant
- With diagnostics set to None, no push occurs and the loop behaves identically to before
- The collector contains the correct number of diagnostics after a step with mixed failures

### R3: tokei-based code line counting in file-modification tools

THE SYSTEM SHALL add the `tokei` crate as a workspace dependency of the `norn` crate. THE SYSTEM SHALL replace the `non_blank_line_count()` function in `write.rs` with a call to `tokei::LanguageType::from_path()` and `LanguageType::parse()` to count code lines (excluding comments and blanks). WHEN tokei does not recognise the file's language, THE SYSTEM SHALL fall back to counting non-blank lines. THE SYSTEM SHALL expose a `pub fn count_code_lines(path: &Path, content: &str) -> u64` function in a shared module that Write, Edit, and ApplyPatch can all call. THE SYSTEM SHALL remove the `non_blank_line_count()` function.

**Acceptance:**
- A Rust file with 100 code lines, 50 comments, and 20 blanks reports 100 code lines
- A Python file with 80 code lines and 30 comments reports 80 code lines
- An unrecognised file extension (.xyz) falls back to non-blank line counting
- non_blank_line_count() no longer exists in write.rs
- Write, Edit, and ApplyPatch all call the same count_code_lines function

### R4: LengthLimit made optional with no hardcoded default

THE SYSTEM SHALL remove `DEFAULT_MAX_LINES`. THE SYSTEM SHALL rename `default_workspace()` to `none()` returning `LengthLimit { default: None, overrides: vec![] }`. THE SYSTEM SHALL update the `Default` impl to return `LengthLimit::none()`. THE SYSTEM SHALL update `limit_for()` to return `Option<usize>` instead of `usize`.

**Acceptance:**
- DEFAULT_MAX_LINES no longer exists

- LengthLimit { default: Some(500), overrides: vec![] }.limit_for(any_path) returns Some(500)
- LengthLimit with default None but a matching glob override returns Some(override_value)
- WriteTool::new() constructs with LengthLimit::none()
- WriteTool::with_length_limit() still accepts caller-specified limits

### R5: Configurable PostValidateMode via ToolContext flags

THE SYSTEM SHALL make PostValidateMode overridable at runtime via ToolContext flags. THE SYSTEM SHALL add a `ToolFlag::ForceGate` variant that promotes Report to Gate semantics. THE SYSTEM SHALL update `ToolRegistry::execute` to check for `ToolFlag::ForceGate` (promotes Report to Gate), `ToolFlag::AllowBrokenAst` (demotes Gate to Report), and `ToolFlag::RejectBrokenAst` (promotes Report to Gate for AST specifically). The priority SHALL be: explicit flags override the tool's declared default. THE SYSTEM SHALL record a `CheckOverride` when a flag overrides the tool's default mode.

**Acceptance:**
- Write tool (default Report) with ForceGate flag returns ToolError::PostValidationFailed on AST errors
- Edit tool (default Gate) with AllowBrokenAst flag returns Ok with errors in the tool result
- A tool with no flags uses its declared default PostValidateMode
- CheckOverride is recorded with flag and source when a mode override occurs
- ForceGate + AllowBrokenAst on the same context: ForceGate wins (explicit gate takes priority)

### R6: Structured diagnostic JSON in tool output

THE SYSTEM SHALL update the tool output format for Write, Edit, and ApplyPatch to include a `diagnostics` array with structured objects containing: `code` (string), `line` (u64), `severity` (string: error/warning/info/hint), `message` (string), and optional `fix` (string) and `do_not` (array of strings) fields. THE SYSTEM SHALL convert existing syntax_errors arrays into this format. THE SYSTEM SHALL include file length violations in the diagnostics array when a threshold is configured and exceeded. THE SYSTEM SHALL set `is_error` on the ToolOutput when any diagnostic has severity 'error'.

**Acceptance:**
- Write tool output for a file with syntax errors includes diagnostics array with code, line, severity, message
- Write tool output for a file exceeding length limit includes a diagnostic with code 'file-length-exceeded'
- Edit tool output for a file with syntax errors includes the same structured format
- diagnostics array is empty when file passes all checks
- is_error is true when any diagnostic has severity 'error', false otherwise

### R7: DiagnosticCollector published on ToolContext typed infrastructure

WHEN a DiagnosticCollector is present on LoopContext, THE SYSTEM SHALL publish it on ToolContext via `insert_extension(Arc::clone(&collector))` before tool dispatch. THE SYSTEM SHALL update tool_dispatch.rs to propagate the collector from LoopContext to the ToolRegistry's shared ToolContext. RuntimePostValidateCheck implementations SHALL be able to retrieve the collector via `ctx.get_extension::<DiagnosticCollector>()` and push diagnostics directly.

**Acceptance:**
- When LoopContext::diagnostics is Some, ToolContext has a DiagnosticCollector available via get_extension
- When LoopContext::diagnostics is None, ToolContext has no DiagnosticCollector
- A RuntimePostValidateCheck can retrieve and push to the collector
- Diagnostics pushed by RuntimePostValidateCheck appear in the collector's drain

## Boundaries

- SHALL NOT implement external linter integration (clippy, cargo check) — that is a CLI/profile concern
- SHALL NOT implement bypass detection — that is a separate brief
- SHALL NOT add the diagnostics crate as a dependency — tokei is added directly
- SHALL NOT implement per-tool CLI configuration — that is NC-009
- SHALL NOT modify the agent loop's control flow based on diagnostics — the collector is observational only
- SHALL NOT add new tool types or modify tool registration
