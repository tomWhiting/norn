## Purpose

Implement the three file tools that define Norn's differentiators: Read (with line numbers, binary detection), Write (with read-before-overwrite, AST validation, file length limits), and Edit (with read-before-edit, staged AST validation with rollback, blast radius reporting). These are the first tools to exercise the Tool trait lifecycle from N-006 with real compile-time pre-validate and post-validate phases. Write uses report semantics (commit then report errors). Edit uses gate semantics (stage in memory, rollback on AST failure). This is where medical-grade code quality starts.

## Task

Implement three tools in the tools module, each implementing the Tool trait from N-006. Read is straightforward: read a file, return content with line numbers, detect binary files and images. Write enforces read-before-overwrite in pre-validate (compile-time check using ToolContext to track read files), validates AST via tree-sitter in post-validate (report semantics: file written, error reported), and enforces file length limits. Edit enforces read-before-edit in pre-validate, applies string replacement, validates AST in post-validate (gate semantics: staged in memory, rollback on failure), and reports blast radius via symbol analysis. tree-sitter is required for AST parsing. Testing: Layer 1 unit tests with real files in temp directories.

## Requirements

### R1: Implement Read tool

Read tool SHALL implement the Tool trait. It SHALL read a file and return its content with line numbers (cat -n format). It SHALL detect binary files and return a message instead of binary content. It SHALL support offset and limit parameters for partial reads. It SHALL detect image files by extension and return an appropriate indicator.

**Acceptance:**
- ReadTool implements Tool trait with name 'read', effect ReadOnly
- Input schema: { path: String, offset: Option<u64>, limit: Option<u64> }
- Output includes line numbers in 'N\t<content>' format
- Binary files (detected by null bytes in first 8192 bytes) return ToolOutput with is_error=false and content indicating binary file
- Image files (.png, .jpg, .jpeg, .gif, .svg, .webp) return indicator with file path
- offset=10, limit=5 returns lines 10-14 of a file
- Reading a nonexistent file returns ToolOutput with is_error=true
- Unit test: write a temp file, read it, verify line numbers in output

### R3: Implement Write tool with read-before-overwrite

Write tool SHALL implement the Tool trait. Pre-validate SHALL block writes to existing files that have not been read (checked via ToolContext.has_read_file). Writing new files (file does not exist) SHALL be allowed without a prior read. This check SHALL respect the AllowOverwrite context flag. WHEN AllowOverwrite is set, the check is skipped and a CheckOverride is recorded.

**Acceptance:**
- WriteTool implements Tool trait with name 'write', effect Write
- Input schema: { path: String, content: String }
- Pre-validate blocks if file exists AND has_read_file is false: returns PreValidateOutcome::Block with reason
- Pre-validate allows if file does not exist (new file creation)
- Pre-validate allows if file exists AND has_read_file is true
- WHEN AllowOverwrite flag is set, pre-validate allows even without prior read, and records CheckOverride with check='read_before_overwrite', flag=AllowOverwrite, source from FlagEntry
- Unit test: attempt write to existing unread file, verify Block. Read file, retry, verify Proceed

### R4: Implement Write tool AST validation (report semantics)

Write tool post-validate SHALL parse the written file with tree-sitter and check for syntax errors. Post-validate uses report semantics: the file IS written to disk, then errors are reported in the tool result. The post-validate phase SHALL NOT prevent the file from being written. tree-sitter language detection SHALL be based on file extension.

**Acceptance:**
- WriteTool.post_validate_mode() returns PostValidateMode::Report
- After writing, tree-sitter parses the file content using the language grammar matching the file extension
- If tree-sitter finds ERROR nodes: PostValidateOutcome::Fail with errors listing line/column and error description
- If no ERROR nodes: PostValidateOutcome::Pass
- Supported extensions for v1: .rs, .py, .js, .ts, .json (focused set — additional grammars added in later briefs)
- Unsupported extensions: post-validate returns Pass (no validation, not failure)
- Unit test: write a .rs file with missing brace, verify Fail with line number. Write valid .rs, verify Pass

### R5: Implement Write tool file length enforcement

Write tool post-validate SHALL check the written file's line count against a configurable limit (default 500, matching CO5). Path-specific overrides SHALL be supported (e.g., test files can be longer). Exceeding the limit SHALL be reported as a PostValidateOutcome::Fail.

**Acceptance:**
- Default file length limit is 500 lines (excluding blank lines only — comment detection deferred to a later brief when tree-sitter line classification is built out)
- Limit is configurable via ToolContext or WriteTool configuration
- Path-specific overrides: patterns like '**/tests/**' or '**/*_test.rs' can have different limits
- Exceeding the limit produces Fail with message including current count and limit
- Unit test: write a 600-line file with default limit, verify Fail. Write a 400-line file, verify Pass

### R6: Implement Edit tool with read-before-edit

Edit tool SHALL implement the Tool trait. It SHALL apply exact string replacement (old_string → new_string). Pre-validate SHALL block edits on files that have not been read. Pre-validate SHALL verify old_string appears exactly once in the file content (ambiguous match = block).

**Acceptance:**
- EditTool implements Tool trait with name 'edit', effect Write
- Input schema: { path: String, old_string: String, new_string: String }
- Pre-validate blocks if has_read_file is false: returns Block with reason
- Pre-validate blocks if old_string is not found in the file: returns Block with 'old_string not found'
- Pre-validate blocks if old_string appears more than once: returns Block with 'old_string is ambiguous (N matches)'
- Execute replaces the single occurrence of old_string with new_string
- Unit test: create file, read it, edit with valid old_string, verify content changed

### R7: Implement Edit tool AST validation (gate semantics)

Edit tool post-validate SHALL parse the edited content with tree-sitter. Post-validate uses gate semantics: the edit is staged in memory. IF AST validation fails, the original file is preserved — the edit is NOT committed to disk. IF AST validation passes, the edited content is committed. WHEN AllowBrokenAst flag is set, gate semantics downgrade to report semantics and a CheckOverride is recorded.

**Acceptance:**
- EditTool.post_validate_mode() returns PostValidateMode::Gate
- Edit is staged: old_string → new_string applied to in-memory content, NOT written to disk during execute
- Post-validate parses the staged content with tree-sitter
- If ERROR nodes found: PostValidateOutcome::Fail, original file content preserved on disk, tool result reports the syntax error with line/column
- If no ERROR nodes: PostValidateOutcome::Pass, edited content committed to disk
- WHEN AllowBrokenAst is set: edit is committed even on AST failure, CheckOverride recorded with check='ast_validation', flag=AllowBrokenAst, source from FlagEntry
- Unit test: edit a valid .rs file to introduce syntax error, verify original file unchanged on disk. Edit with valid change, verify file updated
