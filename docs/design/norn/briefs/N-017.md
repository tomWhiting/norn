# N-017: Wire rules engine and hooks into the agent loop via LoopContext

> **Cluster:** norn
> **Depends on:** N-007, N-008, N-011, N-016
> **Stories:**
> - S25 (Human Developer, Orchestrating Agents via Workflows) — As a workflow developer, I want to write rules that fire before or after specific tools or commands so that agents receive contextual guidance without me modifying the tools themselves.
> - S12 (AI Agent, Searching and Navigating Code) — As an AI agent entering a new directory, I want to see a file tree listing activated by a rule so that I know the directory structure before I start editing.

## Purpose

The rules engine and lifecycle hooks exist as standalone modules but are not connected to the agent loop. run_agent_step has no parameters for rules or hooks and never calls them. This brief creates a LoopContext struct that bundles optional components (rules, hooks, mutable system instruction), wires the rules engine to emit RuntimeEvents from tool calls and apply RuleInjections to the prompt, and wires all five hook types into their correct firing points. After this brief, rules actually fire when files are edited and hooks actually intercept tool and LLM calls.

## Task

Create LoopContext struct bundling RuleEngine, HookRegistry, and composable system instruction. Refactor run_agent_step to accept LoopContext. Wire RuleEngine: emit RuntimeEvents after tool execution (PathChanged, ToolInvoked, BashCommandRun), call process_event, apply RuleInjections (SystemContextAppend to mutable system sections, ContextInjection as UserMessage, MessageDelivery as UserMessage). Wire HookRegistry: PreLlmHook before call_provider (Block returns error), PostLlmHook after with LlmCallSummary, PreToolHook before execute (Block skips tool), PostToolHook after execute, SessionEventHook on every store.append. Make system_instruction owned and composable (Vec<String> sections joined per iteration). Update all existing tests for new signatures. Testing: Layer 2 mock provider tests with rules and hooks exercised.

## Requirements

### R1: Create LoopContext struct

LoopContext SHALL bundle optional components that run_agent_step needs beyond the core parameters. This avoids parameter explosion on the main function.

**Acceptance:**
- LoopContext struct in crates/norn/src/loop/runner.rs or new loop/loop_context.rs
- Fields: rules (Option<RuleEngine>), hooks (Option<HookRegistry>), system_sections (Vec<String> — mutable system instruction parts), event_schemas (Option<EventSchemaSet>), iteration_monitor (Option<IterationMonitorConfig>)
- LoopContext::new() takes the base system instruction as first section
- LoopContext::system_instruction() joins all sections with double newline
- LoopContext::append_system_section(content) adds a section
- LoopContext::clear_dynamic_sections() removes all sections after the first (base instruction)
- Default: all Optional fields None, single system section
- run_agent_step signature changes: replaces system_instruction, event_schemas, iteration_monitor params with &mut LoopContext
- All 24+ existing tests updated to construct LoopContext instead of passing individual params

**Files:**
- modify: crates/norn/src/loop/runner.rs
- modify: crates/norn/src/loop/mod.rs

### R2: Wire RuleEngine — emit RuntimeEvents from tool execution

After each tool execution, the loop SHALL construct a RuntimeEvent and pass it to RuleEngine::process_event. Tool calls that read/write files emit PathChanged events. Bash tool calls emit BashCommandRun events. All tool calls emit ToolInvoked events.

**Acceptance:**
- New helper: build_runtime_event(tool_name, arguments, output) -> Vec<RuntimeEvent>
- Read tool -> RuntimeEvent::PathChanged { path, kind: Read }
- Write/Edit/Patch tool -> RuntimeEvent::PathChanged { path, kind: Write }
- Bash tool -> RuntimeEvent::BashCommandRun { command } extracted from arguments
- All tools -> RuntimeEvent::ToolInvoked { tool_name }
- process_event called after execute_tool_call in helpers.rs
- Vec<RuleInjection> collected and returned alongside tool output
- Test: register a path-glob rule for '*.rs', execute a mock Write tool on 'foo.rs', verify rule fires

**Files:**
- modify: crates/norn/src/loop/helpers.rs
- modify: crates/norn/src/loop/runner.rs

**Stories:**

### R3: Wire RuleEngine — apply RuleInjections to prompt

RuleInjection results SHALL be applied to the prompt based on their DeliveryMode. SystemContextAppend adds to the mutable system sections. ContextInjection and MessageDelivery inject as UserMessages.

**Acceptance:**
- New helper: apply_rule_injections(loop_context, injections, messages)
- SystemContextAppend: calls loop_context.append_system_section(content)
- ContextInjection: appends as UserMessage with role='system' content to messages vec
- MessageDelivery: appends as UserMessage with '[Rule: {rule_id}] {content}' to messages vec
- Dynamic sections cleared at start of each iteration (clear_dynamic_sections) so rules re-fire fresh
- TriggerTiming::Before injections applied before call_provider
- TriggerTiming::After injections applied after tool execution before next iteration
- Test: rule with SystemContextAppend fires, verify system instruction contains rule content on next provider call

### R4: Wire PreToolHook and PostToolHook

Before each tool execution, PreToolHook SHALL fire. If any hook returns Block, the tool is skipped and the block reason returned as the tool result. After each tool execution, PostToolHook SHALL fire with the tool output.

**Acceptance:**
- execute_tool_call in helpers.rs accepts Option<&HookRegistry>
- Before dispatch: if hooks present, call hooks.run_pre_tool(tool_name, arguments).await
- HookOutcome::Block { reason } -> return reason as error tool result, skip execution
- HookOutcome::Proceed -> continue to execution
- After dispatch: if hooks present, call hooks.run_post_tool(tool_name, arguments, output).await
- PostToolHook is observation-only — its return value does not modify the output
- Test: register PreToolHook that blocks 'bash', execute bash, verify skipped with block reason

**Files:**
- modify: crates/norn/src/loop/helpers.rs

### R5: Wire PreLlmHook and PostLlmHook

Before each provider call, PreLlmHook SHALL fire with the request summary. If blocked, the loop SHALL return an error. After each provider call, PostLlmHook SHALL fire with a call summary.

**Acceptance:**
- Before call_provider: if hooks present, call hooks.run_pre_llm(request_summary).await
- LlmRequestSummary struct: model (String), message_count (usize), tool_count (usize), has_schema (bool)
- HookOutcome::Block -> return NornError::HookBlocked { hook_type, reason }
- After call_provider returns assembled response: call hooks.run_post_llm(call_summary).await
- LlmCallSummary struct: stop_reason (StopReason), usage (Usage), tool_call_count (usize), has_text (bool)
- Test: register PreLlmHook that blocks after 3 calls, verify loop terminates with HookBlocked

**Files:**
- modify: crates/norn/src/loop/runner.rs
- modify: crates/norn/src/loop/classify.rs

### R6: Wire SessionEventHook

Every time a SessionEvent is appended to the EventStore, SessionEventHook SHALL fire with the event. This provides a universal observation point for all session activity.

**Acceptance:**
- New helper: append_and_notify(store, event, hooks) — calls store.append then hooks.run_on_event if present
- All store.append call sites in runner.rs and helpers.rs replaced with append_and_notify
- SessionEventHook receives a reference to the event after it's stored
- Hook failures are logged at warn level but do not interrupt the loop
- Test: register SessionEventHook that counts events, run a two-tool loop, verify event count matches

**Files:**
- modify: crates/norn/src/loop/runner.rs
- modify: crates/norn/src/loop/helpers.rs
