# N-024: OAuth authentication via Codex login crate — replace API key auth

> **Cluster:** norn
> **Depends on:** N-004
> **Stories:**
> - S35 (System Integrator, Connecting Norn to Other Tools) — As a system integrator, I want to use Norn as a library crate without depending on Meridian so that I can integrate it into standalone projects.

## Purpose

The OpenAI provider currently authenticates via an API key (SecretString in ProviderConfig). Tom's directive: OAuth is the primary auth path, no API keys. Codex uses OAuth 2.0 Authorization Code with PKCE against auth.openai.com, storing tokens at ~/.codex/auth.json. The codex-login crate (Apache-2.0) implements the full flow. This brief integrates codex-login as a dependency and replaces the API key auth path with OAuth token management.

## Task

Add codex-login as a path dependency. Create an AuthProvider trait abstracting token acquisition. Implement OAuthAuthProvider wrapping codex-login's AuthManager — handles login flow, token storage, proactive refresh, 401 recovery. Replace ProviderConfig's api_key with an AuthSource enum (OAuth or LegacyApiKey for testing). OpenAI provider's request pipeline calls auth_provider.apply_auth() before every HTTP request. Test with mock auth provider. Remove API key as the default path.

## Requirements

### R1: AuthProvider trait and AuthSource enum

An AuthProvider trait SHALL abstract how authentication headers are applied to HTTP requests. AuthSource SHALL distinguish between OAuth (primary) and legacy API key (testing only).

**Acceptance:**
- AuthProvider trait in crates/norn/src/provider/auth.rs: async fn apply_auth(&self, request: reqwest::RequestBuilder) -> Result<reqwest::RequestBuilder, ProviderError>
- AuthProvider trait requires Send + Sync for Arc sharing
- AuthSource enum: OAuth { codex_home: Option<PathBuf> }, ApiKey { key: SecretString }
- AuthSource::OAuth is the default construction path
- AuthSource::ApiKey exists for env-gated integration tests only, NOT as a production path
- ProviderConfig replaces api_key: SecretString with auth_source: AuthSource
- Test: AuthProvider trait is object-safe (compiles as Arc<dyn AuthProvider>)

**Files:**
- create: crates/norn/src/provider/auth.rs
- modify: crates/norn/src/provider/mod.rs
- modify: crates/norn/src/provider/request.rs

### R2: OAuthAuthProvider wrapping codex-login

OAuthAuthProvider SHALL use the codex-login crate's AuthManager to acquire, cache, and refresh OAuth tokens. It reads from ~/.codex/auth.json (shared with Codex CLI — single login works for both).

**Acceptance:**
- Cargo.toml adds codex-login as a path dependency: codex-login = { path = '../../tools/codex/codex-rs/login' } (or appropriate relative path)
- OAuthAuthProvider wraps codex-login's AuthManager
- apply_auth sets Authorization: Bearer {access_token} header
- apply_auth sets ChatGPT-Account-ID header when account_id is present
- Proactive token refresh: checks JWT exp before each request, refreshes if expired
- Token storage: reads from ~/.codex/auth.json or $CODEX_HOME/auth.json
- If no stored token exists, returns a clear error directing the user to run login
- Test: mock AuthManager returns token, verify Bearer header set correctly

**Files:**
- modify: crates/norn/src/provider/auth.rs
- modify: crates/norn/Cargo.toml

### R3: Login flow integration

Norn SHALL expose a login function that triggers the OAuth PKCE flow via codex-login. This is called by the CLI/orchestrator when no stored token exists.

**Acceptance:**
- pub async fn login(config: LoginConfig) -> Result<(), NornError> wraps codex-login's run_login_server or device code flow
- LoginConfig struct: codex_home (Option<PathBuf>), device_code (bool for headless)
- On success, tokens are persisted to auth.json (compatible with Codex CLI)
- On failure, returns NornError with clear message (browser failed to open, user cancelled, etc.)
- pub async fn logout(config: LoginConfig) -> Result<(), NornError> revokes tokens and clears storage
- Test: verify LoginConfig construction with defaults

**Files:**
- modify: crates/norn/src/provider/auth.rs

### R4: Wire auth into OpenAI provider request pipeline

The OpenAI provider SHALL use AuthProvider to authenticate every HTTP request instead of directly setting a Bearer token from an API key.

**Acceptance:**
- OpenAiProvider stores Arc<dyn AuthProvider> instead of SecretString api_key
- build_from_auth_source(auth_source) -> Arc<dyn AuthProvider> factory function
- Every reqwest request in SenderProvider::execute passes through auth_provider.apply_auth()
- 401 responses trigger auth_provider.on_unauthorized() for reactive refresh
- SecretString api_key construction path still compiles (for ApiKey auth source) but is documented as testing-only
- All existing provider tests updated to use a MockAuthProvider
- Test: provider request includes Bearer header from auth provider

**Files:**
- modify: crates/norn/src/provider/openai/mod.rs

### R5: 401 recovery with token refresh

When the OpenAI provider receives a 401 response, it SHALL attempt token refresh via the AuthProvider before failing.

**Acceptance:**
- AuthProvider trait gains: async fn on_unauthorized(&self) -> Result<bool, ProviderError> — returns true if token was refreshed
- OAuthAuthProvider::on_unauthorized triggers codex-login's refresh flow
- On successful refresh, the original request is retried once with the new token
- On failed refresh (expired refresh token, revoked), returns ProviderError::AuthenticationFailed with message directing user to re-login
- ApiKeyAuthProvider::on_unauthorized always returns false (no refresh possible)
- Test: mock provider returns 401, auth provider refreshes, retry succeeds

**Files:**
- modify: crates/norn/src/provider/auth.rs
- modify: crates/norn/src/provider/openai/mod.rs

## Boundaries

- SHALL NOT reimplement OAuth — use the codex-login crate
- SHALL NOT modify the codex-login crate source
- SHALL NOT store API keys in auth.json — OAuth tokens only
- SHALL NOT remove the API key code path entirely (needed for testing) but SHALL NOT expose it as a default or recommended configuration
- SHALL NOT implement the device code flow in this brief — login via browser PKCE only (device code can be a follow-up)
- SHALL NOT add todo!() or unimplemented!() macros anywhere
