# N-020: Profiles, capabilities, slash commands, and runtime prompt commands

> **Cluster:** norn
> **Depends on:** N-017
> **Stories:**
> - S22 (Human Developer, Orchestrating Agents via Workflows) — As a workflow developer, I want to configure which tools an agent has access to based on its profile so that I can enforce role discipline (e.g., team leads cannot edit files).
> - S23 (Human Developer, Orchestrating Agents via Workflows) — As a workflow developer, I want to configure post-edit diagnostics (clippy, tests, formatters) per profile so that different workspaces get appropriate validation without changing tool code.
> - S24 (Human Developer, Orchestrating Agents via Workflows) — As a workflow developer, I want to declare output schemas for each workflow step so that every step produces predictable, typed output.
> - S28 (Human Developer, Steering a Running Agent) — As a human operator, I want the agent to produce both a written and a spoken version of its response so that I can listen via text-to-speech without awkward dot-point cadence.

## Purpose

Feature 19 (profiles/capabilities), 38 (skills/slash commands/profiles/prompt templates), 39 (session variables in prompts), and 40 (runtime system prompt commands) are all unimplemented. The Profile struct is the missing glue that configures what model, tools, hooks, rules, instructions, and capabilities an agent gets. Slash commands pre-process user input before it reaches the model. Runtime system prompt commands populate dynamic prompt sections from shell commands. This brief implements all of them.

## Task

Define Profile struct with model, reasoning_effort, tools, hooks, rules, system_instructions, settings, capabilities. Define Capability as a composable unit of tools + instructions + hooks. Implement from_profile() that constructs a LoopContext (from N-017) and configures the ToolRegistry from a Profile. Implement slash command pre-processing in the loop (detect /command prefix, expand to skill or tool invocation). Implement runtime prompt commands (shell commands registered as system sections that execute at prompt construction time). Thread reasoning_effort from Profile through to ProviderRequest.

## Requirements

### R1: Profile struct and serialization

Profile SHALL configure all aspects of an agent's behavior: model, reasoning effort, tools, hooks, rules, system instructions, settings, and composable capabilities.

**Acceptance:**
- Profile struct in crates/norn/src/profile.rs
- Fields: name (String), model (String), reasoning_effort (Option<ReasoningEffort>), tools (Option<Vec<String>>), system_instructions (Vec<String>), capabilities (Vec<Capability>), settings (HashMap<String, Value>)
- Profile derives Clone, Debug, Serialize, Deserialize
- Loads from TOML or JSON file via Profile::from_file(path)
- Test: round-trip serialize/deserialize a profile with all fields set

**Files:**
- create: crates/norn/src/profile.rs
- modify: crates/norn/src/lib.rs

### R2: Capability composition

Capabilities SHALL be composable units that contribute tools, system instructions, and hooks. A Profile's capabilities are merged to produce the final configuration.

**Acceptance:**
- Capability struct: name (String), tools (Vec<String>), system_instructions (Vec<String>), disallowed_patterns (Vec<String>)
- Profile::resolved_tools() merges Profile.tools with all Capability.tools, deduplicates
- Profile::resolved_instructions() concatenates Profile.system_instructions with all Capability.system_instructions
- Profile::resolved_disallowed() collects all Capability.disallowed_patterns
- Test: profile with 2 capabilities, verify merged tools and instructions

**Files:**
- modify: crates/norn/src/profile.rs

**Stories:**

### R3: from_profile builds LoopContext and configures registry

from_profile SHALL construct a fully configured LoopContext and restrict the ToolRegistry based on the Profile.

**Acceptance:**
- from_profile(profile, registry, rules, hooks) -> (LoopContext, ToolRegistry)
- Calls registry.set_available(profile.resolved_tools()) to restrict tools
- Constructs LoopContext with profile.resolved_instructions() as system sections
- Sets reasoning_effort on the LoopContext for threading to ProviderRequest
- If profile has rules, loads them into the RuleEngine
- Test: from_profile with tools=['read','write'], verify only those tools available

**Files:**
- modify: crates/norn/src/profile.rs
- modify: crates/norn/src/loop/runner.rs

**Stories:**

### R4: Thread reasoning_effort to ProviderRequest

**Acceptance:**

- call_provider reads reasoning_effort from LoopContext
- ProviderRequest.reasoning is set from the LoopContext value

### R5: Slash command preprocessing

User input starting with / SHALL be intercepted before reaching the model and expanded into a skill activation or tool invocation.

**Acceptance:**
- SlashCommand struct: name (String), handler (SlashCommandHandler enum)
- SlashCommandHandler enum: Skill { skill_name: String }, Tool { tool_name: String, args: Value }, Custom { handler: Box<dyn Fn> }
- SlashCommandRegistry in loop/commands.rs
- preprocess_input(input, registry) -> PreprocessResult enum { Passthrough(String), Expanded { messages: Vec<Message> } }
- Called in run_agent_step before the user prompt is added to messages
- /compact, /help, /status as built-in examples
- Test: input '/review foo.rs' expands to skill activation for 'review' with arg 'foo.rs'

**Files:**
- create: crates/norn/src/loop/commands.rs
- modify: crates/norn/src/loop/runner.rs
- modify: crates/norn/src/loop/mod.rs

### R6: Runtime system prompt commands

Profiles SHALL be able to register shell commands that run at prompt construction time and contribute dynamic sections to the system prompt.

**Acceptance:**
- PromptCommand struct: name (String), command (String), cache_ttl (Option<Duration>)
- Profile gains field: prompt_commands (Vec<PromptCommand>)
- At each iteration start, LoopContext evaluates prompt_commands via tokio::process::Command
- Command stdout becomes a system section (append_system_section)
- Cache: if cache_ttl set and previous result is fresh, reuse without re-executing
- Command timeout: 5s default, failure produces diagnostic warning and skips section
- Test: prompt command 'echo Current dir: $(pwd)' appears in system instruction

**Files:**

## Boundaries

- SHALL NOT implement profile persistence or a profile registry — profiles are constructed by the caller
- SHALL NOT implement capability discovery from filesystem — capabilities are defined in profile files
- SHALL NOT implement profile-based provider routing (choosing Claude vs OpenAI based on profile) — that is caller responsibility
- SHALL NOT add todo!() or unimplemented!() macros anywhere
