## Purpose

Implement the core agent loop — the function call that runs an agent step. This is the heart of Norn: take a provider, tools, context, instruction, and output schema, execute the prompt-tool cycle until the model produces schema-valid structured output or the budget is exhausted, streaming events throughout.

## Task

Implement the agent loop in the loop module. The loop sends a ProviderRequest to the provider, streams the response, assembles the complete response (text + tool calls as one assistant message), executes tool calls if present, and repeats until the model produces valid structured output via the dynamic schema tool or stops without tool calls. Schema enforcement uses a combined attempt budget covering both nudges and validation retries. The schema tool is terminal. The loop emits ProviderEvents for observability. All session events are appended to the EventStore. Testing: Layer 2 mock provider tests.

## Requirements

### R1: Define agent loop entry point

The loop module SHALL define a ToolExecutor trait as its minimal tool dispatch interface: an async method that takes a tool name and arguments (Value) and returns a Result<Value, ToolError>. The loop module SHALL expose a public async function `run_agent_step` that accepts a shared provider, a &dyn ToolExecutor, EventStore, ContextEdits, instruction, optional output schema, and configuration. It SHALL return Result<AgentStepResult, NornError>. N-006 (ToolRegistry) will implement ToolExecutor.

**Acceptance:**
- run_agent_step is a public async function in the loop module
- AgentLoopConfig struct has fields: schema_attempt_budget (u32, default 3), max_iterations (Option<u32>), schema_tool_name (String, default 'structured_output')
- AgentStepResult enum has variants: Completed { output: Value, usage: Usage }, SchemaUnreachable { best_attempt: Option<Value>, validation_errors: Vec<String>, attempts: u32, usage: Usage }, MaxIterationsReached { usage: Usage }
- ToolExecutor trait defined in loop module: async fn execute(&self, name: &str, arguments: Value) -> Result<Value, ToolError>
- ToolExecutor is object-safe (usable as &dyn ToolExecutor)
- A MockToolExecutor (HashMap<String, closure>) is provided for testing
- The function compiles and can be called with MockProvider and MockToolExecutor

### R3: Implement response assembly from stream

The loop SHALL assemble a complete response from ProviderEvents. TextDelta accumulated into text. ToolCallDelta accumulated into tool calls. Classification happens after Done, not during streaming.

**Acceptance:**
- [TextDelta('hello '), TextDelta('world'), Done] -> text='hello world', 0 tool calls
- [TextDelta('x'), ToolCallDelta(id=1,name='read',args='{"path":'), ToolCallDelta(id=1,args='"f"}'), Done(ToolUse)] -> text='x', 1 tool call with args='{"path":"f"}'
- [ToolCallDelta(id=1,...), ToolCallDelta(id=2,...), Done] -> 2 tool calls
- Classification does NOT happen during streaming

### R4: Implement response classification (four cases)

After assembly, classify: (1) schema tool + valid = Completed, (2) schema tool + invalid = retry, (3) other tools only = execute + continue (no budget), (4) no tools + stopped = nudge + budget.

**Acceptance:**
- Case 1: schema tool with valid output -> Completed
- Case 2: schema tool invalid then valid -> Completed, EventStore has validation error between calls
- Case 3: Read tool then schema tool -> Read executes, Completed. No budget consumed for Read
- Case 4: text-only stop then schema tool after nudge -> nudge injected, Completed. 1 budget consumed
- Schema tool identified by configurable name (default 'structured_output')
