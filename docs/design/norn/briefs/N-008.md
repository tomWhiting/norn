# N-008: Per-event schemas and inbound channels

> **Cluster:** norn

> - S26 (Human Developer, Steering a Running Agent) — As a human operator, I want to send a steering message to a running agent that gets delivered at the next tool boundary so that I can redirect without interrupting.

## Purpose

Extend the agent loop with two features: per-event output schemas that validate different event types (assistant messages, spoken responses, tool call envelopes, etc.) against configurable schemas, and inbound channels that deliver messages to a running agent at controlled tool boundaries without interrupting mid-thought. Two delivery modes control when inbound messages are injected: steer (after tool batch, before next LLM call) and follow-up (only when the agent would otherwise stop). Together these enable human steering, inter-agent coordination, and TTS-friendly spoken output.

## Task

Define event type and schema mapping types. Define inbound channel types with delivery modes. Integrate both into the agent loop: validate events against per-event schemas when configured, drain the inbound channel at tool boundaries, inject steer messages before the next provider call, and inject follow-up messages only when the model would stop. Testing: Layer 2 mock provider tests for all delivery modes and schema validation paths.

## Requirements

### R1: Define EventType enum and EventSchemaSet

THE SYSTEM SHALL define an EventType enum covering the event types that can have per-event output schemas. EventSchemaSet SHALL map each EventType to an optional JSON Schema. Profiles configure which schemas are active by populating the set.

**Acceptance:**
- EventType enum defined in crates/norn/src/loop/event_schemas.rs with variants: AssistantMessage, SpokenResponse, ToolCallEnvelope, StopOutput, Question, Handoff, Review, Progress
- EventType derives Clone, Copy, Debug, PartialEq, Eq, Hash, Serialize, Deserialize
- EventSchemaSet struct wraps a HashMap<EventType, serde_json::Value> where each value is a JSON Schema
- EventSchemaSet::new() creates an empty set (no schemas configured)
- EventSchemaSet::set(event_type, schema) registers a schema for an event type
- EventSchemaSet::get(event_type) returns Option<&serde_json::Value>
- EventSchemaSet::has(event_type) returns bool
- EventSchemaSet::event_types() returns an iterator over configured event types
- EventSchemaSet derives Clone, Debug

### R2: Per-event schema validation

THE SYSTEM SHALL validate event content against configured per-event schemas using the same jsonschema validation as the output schema. Validation returns a typed result with the event type and any errors. When no schema is configured for an event type, validation passes unconditionally.

**Acceptance:**
- EventValidationResult struct defined with fields: event_type (EventType), passed (bool), errors (Vec<String>)
- EventSchemaSet::validate(&self, event_type: EventType, content: &serde_json::Value) returns EventValidationResult
- When no schema configured for the event type, validate returns passed=true with empty errors
- When schema configured and content is valid, validate returns passed=true
- When schema configured and content is invalid, validate returns passed=false with jsonschema error messages
- Unit test: set AssistantMessage schema requiring { "text": "string" }, validate matching content passes
- Unit test: validate non-matching content fails with descriptive errors
- Unit test: validate against unconfigured event type passes

### R3: Spoken response via dynamic tool

THE SYSTEM SHALL support a SpokenResponse event type whose schema defines the shape of TTS-friendly output. When a SpokenResponse schema is configured in the EventSchemaSet, the agent loop SHALL register a dynamic 'spoken_response' tool (same pattern as the structured_output schema tool from N-005). The model calls this tool to produce spoken output, and the tool's execute phase validates the arguments against the SpokenResponse schema. Valid spoken content is emitted as a SpokenResponseEvent session event. Invalid content returns a validation error as the tool result for retry. The spoken_response tool is NOT terminal — the model can call it alongside regular tools and the structured_output tool.

**Acceptance:**
- SpokenResponse is a variant of EventType (defined in R1)
- SpokenResponseEvent session event variant defined in session/events.rs with fields: base (EventBase), content (serde_json::Value)
- When EventSchemaSet has a SpokenResponse schema, a 'spoken_response' ToolDefinition is added to the tool list passed to the provider (same mechanism as build_schema_tool in N-005's schema.rs)
- The spoken_response tool definition includes the SpokenResponse schema as its parameters schema
- When the model calls the spoken_response tool, arguments are validated against the SpokenResponse schema
- Valid arguments: appended to EventStore as SpokenResponseEvent, tool result returns success acknowledgment
- Invalid arguments: tool result returns validation errors and the schema for retry (same format as schema validation feedback)
- The spoken_response tool is NOT terminal — it does not end the loop, the model continues with other tools or the structured_output tool
- spoken_response tool name is configurable via AgentLoopConfig (default 'spoken_response')
- Unit test: model calls spoken_response with valid content, SpokenResponseEvent appears in EventStore
- Unit test: model calls spoken_response with invalid content, validation error returned as tool result

### R4: Define InboundChannel and DeliveryMode types

THE SYSTEM SHALL define an InboundChannel that receives messages destined for a running agent. Each inbound message carries a delivery mode (Steer or FollowUp) that determines when it is injected into the conversation. The channel is an async multi-producer single-consumer queue backed by a tokio mpsc channel.

**Acceptance:**
- DeliveryMode enum defined in crates/norn/src/loop/inbound.rs with variants: Steer, FollowUp
- DeliveryMode derives Clone, Copy, Debug, PartialEq, Eq, Serialize, Deserialize
- ChannelMessage struct defined with fields: author (String), content (String), delivery (DeliveryMode), timestamp (chrono::DateTime<chrono::Utc>)
- ChannelMessage derives Clone, Debug, Serialize, Deserialize
- InboundChannel struct wraps a tokio::sync::mpsc::Receiver<ChannelMessage>
- InboundSender struct wraps a tokio::sync::mpsc::Sender<ChannelMessage> and is Clone + Send
- inbound_channel(buffer: usize) function returns (InboundSender, InboundChannel)
- InboundSender::send(&self, msg: ChannelMessage) is async and returns Result
- InboundChannel::drain(&mut self) returns Vec<ChannelMessage> (all currently buffered messages, non-blocking)
- Unit test: send 3 messages, drain returns all 3 in order
- Unit test: drain on empty channel returns empty Vec

### R5: Inbound channel drain at tool boundaries

WHEN a tool batch completes execution, the agent loop SHALL drain the inbound channel. Messages are collected and separated by delivery mode. This is the only point at which inbound messages enter the conversation — they never interrupt a running tool or a streaming provider response.

**Acceptance:**
- run_agent_step accepts optional inbound: Option<&mut InboundChannel>
- After executing a ToolsOnly tool batch, the loop drains the inbound channel before the next provider call
- After executing pre-schema tools in ToolsAndSchemaValid, inbound is drained
- Drain occurs AFTER tool results are appended and BEFORE the next provider request
- Drained messages are partitioned into steer_messages and follow_up_messages by their DeliveryMode
- MockProvider test: send steer message during tool execution, verify it appears in the next provider call's messages
- No drain occurs during streaming or between individual tool calls within a batch

### R6: Steer delivery mode

WHEN the inbound channel is drained at a tool boundary and contains Steer messages, THE SYSTEM SHALL inject them as user messages immediately — after the current tool batch completes, before the next LLM call. Each steer message becomes a UserMessage event in the store and a User-role message in the conversation. The loop then continues normally (sends the next provider request with the steering context included).

**Acceptance:**
- Steer messages from drain are formatted as user messages with author attribution: '[Inbound from {author}]: {content}'
- Each steer message is appended to the EventStore as a UserMessage event
- Each steer message is appended to the local messages vec as a User-role message
- Multiple steer messages from one drain are injected in timestamp order
- After steer injection, the loop continues to the next provider call (does not stop)
- MockProvider test: agent executes tool, steer message injected, next provider call includes the steer content
- MockProvider test: 2 steer messages injected in correct order

### R7: Follow-up delivery mode

WHEN the agent would otherwise stop (TextStopNoSchema in no-schema mode, or Completed with valid schema output), THE SYSTEM SHALL check for buffered follow-up messages. If follow-up messages exist, they are injected as user messages and the loop continues instead of returning. If no follow-up messages exist, the loop returns normally. Follow-up messages are collected from drains at every tool boundary and held until a stop condition.

**Acceptance:**
- Follow-up messages from drains are accumulated in a buffer across the step, not injected immediately
- When the model stops (would return Completed or text-stop), the loop checks the follow-up buffer AND drains the channel one final time
- If follow-up messages exist, they are injected as UserMessage events with author attribution: '[Follow-up from {author}]: {content}'
- After follow-up injection, the loop continues (does not return Completed)
- If no follow-up messages exist at stop time, the loop returns normally
- Follow-up injection does NOT consume schema budget
- MockProvider test: model produces valid schema output, but follow-up message exists — loop continues with follow-up injected, model produces new schema output
- MockProvider test: no-schema mode, model stops with text, follow-up injected, model produces new text response — final Completed has the second response
- MockProvider test: model stops with no follow-up messages — returns Completed normally

## Boundaries

- SHALL NOT implement profile configuration loading — EventSchemaSet is constructed by the caller
- SHALL NOT implement AST validation, diagnostics, or any tool-specific logic
- SHALL NOT implement inter-agent messaging transport — InboundChannel is a local tokio channel; the caller connects it to external sources
- SHALL NOT implement token threshold detection or soft handoff (C23-C24)
- SHALL NOT implement repeated failure detection or semantic quality signals (C25-C26)
- SHALL NOT implement rules engine integration (C56-C64) — rules are a separate subsystem
- SHALL NOT modify the schema enforcement mechanism from N-005 — per-event schemas supplement it, they do not replace it
- SHALL NOT add todo!() or unimplemented!() macros anywhere
