## Purpose

Build the multi-agent infrastructure that all agent coordination depends on. AgentRegistry tracks active agents with hierarchical paths and no concurrency limits. Mailbox provides inter-agent messaging with sequence numbers. Fork logic inherits filtered context and returns structured audit results. RunMonitored delegates background task watching to a lightweight model. Goal tracking manages objectives, budgets, and continuation policies. This is infrastructure — the tools that wrap it (SpawnAgent, Fork tool, etc.) are in N-015.

## Task

Implement the five agent/ modules: registry.rs (AgentRegistry with two-phase spawn, RAII cleanup, hierarchical paths, no concurrency cap), mailbox.rs (Mailbox with trigger_turn flag and sequence numbers for wait-for-any), fork.rs (fork logic with filtered context inheritance, model switching, structured ForkAudit result), monitor.rs (RunMonitored with lightweight model delegation and query handle), goals.rs (GoalTracker with objectives, token/time budgets, continuation policies). Testing: Layer 1 unit tests for each module.

## Requirements

### R1: AgentRegistry with hierarchical paths and statuses

AgentRegistry SHALL track active agents by path (hierarchical, e.g. /workflow/dev/agent-1) with statuses (Spawning, Active, Completing, Completed, Failed). No hardcoded concurrency limits.

**Acceptance:**
- AgentRegistry struct in crates/norn/src/agent/registry.rs
- AgentEntry struct: id (Uuid), path (String), role (String), status (AgentStatus), model (String), spawned_at (DateTime<Utc>), parent_id (Option<Uuid>)
- AgentStatus enum: Spawning, Active, Completing, Completed, Failed
- get(id) returns Option<&AgentEntry>
- get_by_path(path) returns Option<&AgentEntry>
- list() returns iterator over all entries
- children(parent_id) returns iterator over direct children
- No field or constant caps agent count (C66)
- Test: register 100 agents, verify all accessible

### R2: Two-phase spawn reservation with RAII cleanup

Spawning an agent SHALL be two-phase: reserve (allocates ID, sets Spawning status, returns SpawnGuard) then confirm (sets Active). If the SpawnGuard drops without confirm, the entry is removed.

**Acceptance:**
- reserve(path, role, model, parent_id) returns Result<SpawnGuard>
- SpawnGuard holds Arc<parking_lot::RwLock<AgentRegistry>> and agent ID
- SpawnGuard::confirm(self) transitions status to Active and consumes the guard
- Drop impl on SpawnGuard removes the entry if not confirmed
- Duplicate path during reserve returns error
- Test: reserve without confirm — verify entry cleaned up on drop
- Test: reserve then confirm — verify entry persists with Active status

### R3: Mailbox with trigger_turn and sequence numbers

Mailbox SHALL support inter-agent messaging with a trigger_turn flag for immediate vs deferred delivery. Sequence numbers on each mailbox SHALL enable efficient wait-for-any without polling.

**Acceptance:**
- Mailbox struct in crates/norn/src/agent/mailbox.rs
- MailboxMessage struct: id (Uuid), from (Uuid), to (Uuid), content (serde_json::Value), trigger_turn (bool), sequence (u64), timestamp (DateTime<Utc>)
- send(from, to, content, trigger_turn) appends message and increments sequence
- recv(agent_id) returns Vec<MailboxMessage> draining pending messages
- peek(agent_id) returns count of pending messages without draining
- wait_for_any(agent_ids, since_sequences) returns when any mailbox has new messages past the given sequence numbers — uses tokio::sync::Notify
- Test: send 3 messages, recv drains all 3, second recv returns empty
- Test: wait_for_any wakes when message arrives

### R6: Goal tracking with budgets and continuation

GoalTracker SHALL manage objectives, token budgets, time budgets, and continuation policies. When a budget threshold is reached, it signals the appropriate continuation action.

**Acceptance:**
- GoalTracker struct in crates/norn/src/agent/goals.rs
- Goal struct: description (String), token_budget (Option<u64>), time_budget (Option<Duration>), continuation (ContinuationPolicy)
- ContinuationPolicy enum: Stop, Handoff { summary_prompt: String }, Continue { recapitulate: bool }
- GoalTracker::check(usage, elapsed) returns GoalSignal
- GoalSignal enum: OnTrack, BudgetWarning { pct_used: f32 }, BudgetExceeded { policy: ContinuationPolicy }
- BudgetWarning triggers at configurable threshold (default 80%)
- Test: set token budget 1000, report 800 used, verify BudgetWarning
- Test: set token budget 1000, report 1100 used, verify BudgetExceeded with correct policy

### R7: Session-dispatched cron scheduling

ScheduleEntry SHALL define cron-like schedules that re-launch agent sessions without keeping them alive between runs. The scheduler stores entries; execution is handled by the integration layer.

**Acceptance:**
- ScheduleEntry struct: id (Uuid), cron_expr (String), agent_config (serde_json::Value), enabled (bool), last_run (Option<DateTime<Utc>>), next_run (Option<DateTime<Utc>>)
- Scheduler struct stores entries in a HashMap<Uuid, ScheduleEntry>
- Scheduler::add(entry) registers a schedule
- Scheduler::remove(id) removes a schedule
- Scheduler::due_entries(now) returns entries where next_run <= now and enabled
- Scheduler::advance(id, now) updates last_run and computes next_run from cron_expr
- Cargo.toml adds cron crate for expression parsing
- Test: add entry with '*/5 * * * *', verify due_entries fires at correct interval

## Boundaries

- SHALL NOT implement agent tools (SpawnAgent, Fork tool, SendMessage, etc.) — those are N-015
- SHALL NOT implement the agent loop — only the infrastructure it uses
- SHALL NOT implement provider selection or model routing — fork uses a provider passed by the caller
- SHALL NOT implement session persistence to disk — registry and mailbox are in-memory
- SHALL NOT implement actual cron execution — only the schedule store and due-entry queries
- SHALL NOT add todo!() or unimplemented!() macros anywhere
