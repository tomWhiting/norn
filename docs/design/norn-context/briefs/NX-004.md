## Purpose

Subdirectories can have their own NORN.md files that provide directory-specific conventions. Rather than building a parallel activation system, these are registered as synthetic rules in the existing RuleEngine. This reuses trigger evaluation, presence tracking, and compaction recovery without new machinery.

## Task

Extend context/scanner.rs with a function that, given a PathChanged event, checks if the file's directory ancestry contains a NORN.md that hasn't been registered yet. When found, create a synthetic Rule with a PathGlob trigger matching the directory and descendants, register it in the RuleEngine via add_rule(). Track which directories have been scanned to avoid duplicate registration.

## Requirements

### R1: Nested NORN.md detection on PathChanged

WHEN a RuntimeEvent::PathChanged occurs, the scanner SHALL check if any directory in the file's ancestry (up to the project root) contains a NORN.md file that has not yet been registered as a synthetic rule.

**Acceptance:**
- Reading src/api/handler.rs checks src/api/ for NORN.md, then src/ for NORN.md
- The check stops at the project root (cwd), does not walk above it
- Directories already scanned are skipped (O(1) lookup via a HashSet)

**Files:**
- modify: crates/norn/src/context/scanner.rs

**Checklist:**
- C22 — context/scanner.rs detects NORN.md files in directory ancestry when a PathChanged event occurs.

**Stories:**

### R2: Synthetic rule creation

WHEN a NORN.md is found in a subdirectory, the scanner SHALL create a Rule with: ID 'norn-md:{relative-path}', a PathGlob trigger matching '{subdir}/**', DeliveryMode::SystemContextAppend, TriggerTiming::After, and the file content as the body.

**Acceptance:**
- NORN.md in src/api/ creates a Rule with id 'norn-md:src/api'
- The trigger pattern is 'src/api/**'
- The delivery mode is SystemContextAppend
- The timing is After
- The body is the full content of the NORN.md file
- shell_source is None

### R3: Single registration per directory

A nested NORN.md SHALL be registered as a synthetic rule exactly once. Subsequent PathChanged events in the same directory or its descendants SHALL NOT re-register the rule.

**Acceptance:**
- First PathChanged in src/api/ registers the synthetic rule
- Second PathChanged in src/api/ does not create a duplicate rule
- PathChanged in src/api/v2/ (subdirectory) does not re-register src/api/NORN.md

### R4: Compaction recovery via presence tracking

Synthetic rules SHALL recover from compaction via the RuleEngine's existing presence tracking (RulePresenceSet). WHEN a synthetic rule is lost from context during compaction, it SHALL re-activate on the next PathChanged event in the matching directory.

**Acceptance:**
- After compaction clears dynamic sections, the synthetic rule is no longer in the presence set
- A subsequent PathChanged in the matching directory triggers re-injection via the rules engine
- The re-injected content matches the original NORN.md content

### R5: RuleEngine integration

Synthetic rules SHALL be added to the RuleEngine via the public add_rule() method. The scanner SHALL NOT modify RuleEngine internals.

**Acceptance:**
- scanner calls engine.add_rule(rule) for each synthetic rule
- The synthetic rule appears in engine.process_event() results when a matching PathChanged occurs
- No modifications to rules/engine.rs internals are needed

## Boundaries

- SHALL NOT modify the RuleEngine struct or its internal methods — use add_rule() only
- SHALL NOT modify the Rule type or TriggerCondition type
- SHALL NOT implement file watching — detection is reactive on PathChanged events
- SHALL NOT handle always-on context files — that is NX-001
