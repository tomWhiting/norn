## Purpose

All context infrastructure is built but nothing is connected. This brief wires context file loading, rule directory scanning, and mtime staleness detection into build_runtime. It assembles the base system instruction in the correct layering order and verifies that compaction survival works via the existing split-prompt architecture.

## Task

In norn-cli's build_runtime: load always-on context files via the context loader, scan rule directories and populate the RuleEngine, assemble system_sections[0] in the correct order (Norn base prompt, profile instructions, user NORN.md, project NORN.md, skill catalog listing). Wire mtime staleness checking into the iteration loop. Verify that always-on context survives compaction (part of system_sections[0]) and that active rules re-activate via presence tracking.

## Requirements

### R1: Context file loading in build_runtime

WHEN build_runtime executes, it SHALL construct a context loader, load always-on NORN.md files, and store the loader on LoopContext for staleness checking.

**Acceptance:**
- build_runtime calls context::loader to load ~/.norn/NORN.md and {cwd}/NORN.md
- The context loader is stored on LoopContext (accessible for mtime checks)
- Missing context files do not cause build_runtime to fail

**Files:**
- modify: crates/norn-cli/src/runtime/builder.rs

**Checklist:**
- C42 — Context files and rule files are loaded during build_runtime.

**Stories:**
- S17 (Workflow Author, Orchestrating Agents with Context) — As a workflow author, I want the context loader available as an API so that I can programmatically load context for agent steps.

### R2: Rule directory scanning in build_runtime

WHEN build_runtime executes, it SHALL scan rule directories and add all parsed rules to the RuleEngine on LoopContext.

**Acceptance:**
- build_runtime calls context::scanner to scan {cwd}/.norn/rules/ and ~/.norn/rules/
- All successfully parsed rules are added to LoopContext.rules via add_rule()
- Parse failures for individual rule files are logged but do not block startup

### R3: Base instruction layering order

system_sections[0] SHALL be assembled in this order: (1) Norn base prompt from build_system_prompt(), (2) profile system_instructions, (3) user-level NORN.md content, (4) project-root NORN.md content, (5) skill catalog listing (when norn-skills is wired).

**Acceptance:**
- The base instruction contains the Norn identity line before any NORN.md content
- Profile instructions appear before NORN.md content
- User NORN.md content appears before project NORN.md content
- When no context files exist, the order is base prompt + profile instructions only

### R5: Compaction survival verification

Always-on context files SHALL survive compaction because they are part of system_sections[0]. Active rules SHALL re-activate after compaction via the RuleEngine's presence tracking.

**Acceptance:**
- After simulated compaction (clear_dynamic_sections), system_sections[0] still contains NORN.md content
- After compaction, a PathChanged event re-triggers a previously-active rule
- No special compaction logic is added — the existing split-prompt architecture handles this

**Checklist:**

### R6: Mtime check wiring

The agent loop's iteration setup SHALL call the context loader's staleness check at the start of each iteration. WHEN staleness is detected, system_sections[0] SHALL be rebuilt.

**Acceptance:**
- The staleness check runs alongside clear_dynamic_sections and evaluate_prompt_commands
- A modified NORN.md file is picked up within one iteration
- The mtime check adds no more than two stat() syscalls per iteration
