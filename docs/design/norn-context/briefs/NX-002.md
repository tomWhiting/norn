# NX-002: Mtime staleness detection and rule file discovery

## Purpose

Context files may be edited mid-session, and rule files on disk need to be discovered and loaded into the RuleEngine. This brief combines mtime-based staleness detection (so the runtime picks up NORN.md changes without a file watcher) with rule directory scanning (bridge between rules on disk and the running engine). Both are filesystem operations in the same narrow domain — staleness can't be tested without discovery.

## Task

Add a staleness check method to the context loader that stats always-on context files and re-reads on mtime change. Integrate the check point into LoopContext. Create context/scanner.rs to scan ordered rule directories (.norn/rules/, ~/.norn/rules/, optionally .claude/rules/), parse .md files via parse_rule_file(), derive RuleId from file stem, and apply first-found-wins on ID collision. Add rules_dir() path helpers.

## Requirements

### R1: Staleness check method

The context loader SHALL provide a check_staleness() method that stats both always-on context files and compares mtime against the last-seen values.

**Acceptance:**
- check_staleness() stats ~/.norn/NORN.md and {cwd}/NORN.md
- Returns false (no change) when neither file's mtime has changed
- Returns true (changed) when either file's mtime differs from last-seen
- Records the new mtime after detecting a change

**Files:**
- modify: crates/norn/src/context/loader.rs

**Checklist:**
- C18 — At the start of each iteration, the context loader stats both always-on context files.
- C19 — If a context file's mtime differs from the last-seen mtime, the file is re-read.

**Stories:**
- S9 (Human Operator, Configuring Project Context) — As a human operator, I want to edit NORN.md mid-session and have the agent pick up the changes so that I can refine conventions without restarting.

### R2: Re-read on staleness

WHEN check_staleness() detects a change, it SHALL re-read the changed file(s) and update the stored content. A subsequent call to the content method SHALL return the new content.

**Acceptance:**
- After modifying a context file and calling check_staleness(), the content reflects the new file
- Only the changed file is re-read (the unchanged file retains its cached content)
- The mtime for the changed file is updated to the new value

**Files:**
- modify: crates/norn/src/context/loader.rs

**Checklist:**
- C20 — When a context file changes, system_sections[0] is rebuilt with the new content.

**Stories:**
- S9 (Human Operator, Configuring Project Context) — As a human operator, I want to edit NORN.md mid-session and have the agent pick up the changes so that I can refine conventions without restarting.

### R3: Missing file handling

IF a context file does not exist when check_staleness() is called, the stat SHALL complete without error. IF a file that previously existed is now missing, the stored content for that file SHALL be cleared.

**Acceptance:**
- check_staleness() with neither file existing returns false and no error
- A previously-existing file that is now deleted results in cleared content for that file
- A file that appears for the first time mid-session is detected and loaded

**Files:**
- modify: crates/norn/src/context/loader.rs

**Checklist:**
- C21 — If a context file does not exist, the stat check completes without error.

### R4: LoopContext integration point

LoopContext SHALL hold an optional reference to the context loader. A method SHALL be available to check staleness and trigger a rebuild of system_sections[0] when content has changed.

**Acceptance:**
- LoopContext gains an optional context loader field
- A method on LoopContext calls check_staleness() and rebuilds system_sections[0] if true
- The method is callable at the start of each iteration (alongside evaluate_prompt_commands)

**Files:**
- modify: crates/norn/src/loop/loop_context.rs

**Acceptance:**

### R6: Rule directory scanning

context/scanner.rs SHALL scan an ordered list of rule directories. For each directory that exists, it SHALL enumerate all .md files, parse each via parse_rule_file(), and collect the successfully parsed rules.

**Acceptance:**
- A directory containing rust-conventions.md and testing.md produces 2 parsed rules
- A directory that does not exist is silently skipped
- Non-.md files in the directory are ignored
- Parse failures for individual files are logged via tracing::warn and skipped

### R7: Rule ID from file stem

The rule ID SHALL be derived from the file stem: rust-conventions.md becomes RuleId("rust-conventions").

**Acceptance:**
- rust-conventions.md produces RuleId("rust-conventions")
- MY-RULE.md produces RuleId("MY-RULE")
- The file extension (.md) is stripped, everything else is preserved

**Files:**

### R8: First-found-wins on ID collision

WHEN the same rule ID is found in multiple directories (e.g. both project and user directories contain rust-conventions.md), the version from the earlier directory in the scan order SHALL be used. The later version SHALL be silently skipped.

**Acceptance:**
- With project and user dirs both containing rust-conventions.md, only the project version is loaded
- The skipped rule is logged at tracing::debug level
- The returned Vec<Rule> contains exactly one rule per unique ID

## Boundaries

- SHALL NOT implement file watchers (notify crate or similar) — mtime stat is sufficient
- SHALL NOT modify the agent loop's iteration logic — only add a callable method on LoopContext
- SHALL NOT implement nested NORN.md detection — that is NX-004
- SHALL NOT implement Claude Code format compatibility — that is NX-003
- SHALL NOT wire into build_runtime — that is NX-005
- SHALL NOT modify the RuleEngine internals — only add rules via the public add_rule() method
