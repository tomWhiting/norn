# NX-003: Claude Code rule format compatibility

> **Cluster:** norn-context
> **Depends on:** NX-002
> **Checklist:**
> - C32 — rules/parser.rs detects Norn format when the triggers: key is present in frontmatter.
> - C33 — rules/parser.rs detects Claude Code format when the globs: or paths: key is present in frontmatter.
> - C34 — Both triggers: and globs:/paths: present in the same file produces a parse error.
> - C35 — Claude Code globs: string maps to a single TriggerCondition::PathGlob.
> - C36 — Claude Code globs: array maps to one TriggerCondition::PathGlob per pattern.
> - C37 — Claude Code description: maps to Rule.name.
> - C38 — Claude Code format rules default to DeliveryMode::SystemContextAppend.
> - C39 — Claude Code format rules default to TriggerTiming::After.
> **Stories:**
> - S8 (Human Operator, Configuring Project Context) — As a human operator, I want to drop Claude Code rule files (with globs: frontmatter) into .norn/rules/ without modification so that I don't maintain two formats.
> - S14 (Rule Author, Writing Conditional Guidance) — As a rule author, I want to write rules in Claude Code's simpler format (with globs: only) so that I can reuse rules across both tools.
> - S15 (Rule Author, Writing Conditional Guidance) — As a rule author, I want a clear error when my rule file has both triggers: and globs: keys so that I know the format is ambiguous.

## Purpose

Users should be able to drop Claude Code rule files (with globs: frontmatter) into .norn/rules/ without modification. This brief adds format detection and auto-mapping to the rules parser so both Norn's native format and Claude Code's simpler format are supported.

## Task

Modify rules/parser.rs to detect the rule format by examining frontmatter keys. When triggers: is present, use the existing Norn parser path. When globs: or paths: is present, auto-map to Norn types: each glob becomes a PathGlob trigger, delivery defaults to SystemContextAppend, timing defaults to After, description maps to Rule.name. When both triggers: and globs:/paths: are present, return a parse error.

## Requirements

### R1: Norn format detection

WHEN the frontmatter YAML contains a triggers: key, the parser SHALL use the existing Norn format parsing path.

**Acceptance:**
- A file with triggers: key is parsed via the existing path
- All existing rules/parser.rs tests continue to pass unchanged
- The detection checks for key presence, not key value

**Files:**
- modify: crates/norn/src/rules/parser.rs

**Checklist:**
- C32 — rules/parser.rs detects Norn format when the triggers: key is present in frontmatter.

**Stories:**
- S13 (Rule Author, Writing Conditional Guidance) — As a rule author, I want to write rules in Norn's native format (with triggers, delivery, and timing) so that I can use all three trigger types and all three delivery modes.

### R2: Claude Code format detection

WHEN the frontmatter YAML contains a globs: or paths: key (and no triggers: key), the parser SHALL use the Claude Code compatibility path.

**Acceptance:**
- A file with globs: key and no triggers: key is detected as Claude Code format
- A file with paths: key and no triggers: key is detected as Claude Code format
- paths: is treated as an alias for globs:

**Files:**
- modify: crates/norn/src/rules/parser.rs

**Checklist:**
- C33 — rules/parser.rs detects Claude Code format when the globs: or paths: key is present in frontmatter.

**Stories:**

### R3: Ambiguous format rejection

IF the frontmatter contains both triggers: and globs: (or paths:), parse_rule_file() SHALL return RulesError::ParseFailed with a message indicating ambiguous format.

**Acceptance:**
- A file with both triggers: and globs: returns RulesError::ParseFailed
- The error message contains 'ambiguous' or similar indication
- A file with triggers: and paths: also returns an error

**Files:**
- modify: crates/norn/src/rules/parser.rs

**Checklist:**

### R4: Single glob string mapping

WHEN globs: is a single string, it SHALL be mapped to one TriggerCondition::PathGlob with that string as the pattern.

**Acceptance:**
- globs: '**/*.rs' produces one PathGlob trigger with pattern '**/*.rs'
- The resulting rule has exactly one trigger in its triggers vec

**Files:**

### R5: Glob array mapping

WHEN globs: is an array of strings, each string SHALL be mapped to a separate TriggerCondition::PathGlob.

**Acceptance:**
- globs: ['**/*.rs', '**/*.toml'] produces two PathGlob triggers
- Each trigger has the correct pattern from the array
- Order is preserved
