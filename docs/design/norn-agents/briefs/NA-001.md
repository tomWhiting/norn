# NA-001: Profile module restructuring and markdown frontmatter parser

> **Cluster:** norn-agents
> **Checklist:**
> - C1 — Profile parser accepts markdown files with YAML frontmatter and extracts structured fields plus markdown body as system prompt.
> - C2 — Profile frontmatter supports name, description, model, tools (list or comma-separated), disallowedTools, reasoning_effort, reasoning_summary, and capabilities fields.
> - C9 — Capability parser accepts markdown files with YAML frontmatter and extracts tools, disallowedTools, and prompt_fragment.
> - C10 — Capability resolution merges capabilities into a profile: tools union, disallowedTools append, prompt_fragment appends to system prompt.
> **Stories:**
> - S19 (Human Developer, Configuring Agent Profiles) — As a human developer, I want to write agent profiles as markdown files with YAML frontmatter so that the format matches our existing Meridian profiles.

## Purpose

Norn profiles currently use TOML/JSON via a flat profile.rs file. The norn-agents design requires markdown profiles with YAML frontmatter, matching the 31 existing Meridian profiles at .meridian/profiles/. This brief converts profile.rs into a profile/ folder module and implements frontmatter parsing and capability resolution as libnorn APIs, establishing the foundation for profile scanning (NA-002) and spawn-with-profile (NA-006).

## Task

Convert profile.rs into a profile/ folder module with types.rs, loader.rs, and resolve.rs. Implement split_frontmatter (own 30-line implementation, not imported from claude-runner) to parse --- delimited YAML frontmatter from markdown files. Implement parse_profile to deserialise YAML frontmatter into a norn-specific RawNornProfileFrontmatter struct and map it to the existing Profile type, with the markdown body becoming the system prompt. Implement parse_capability for markdown capability files. Move capability resolution logic (resolved_tools, resolved_instructions, resolved_disallowed, from_profile) into resolve.rs. Preserve all existing TOML/JSON loading via from_file for backwards compatibility during migration.

### R3: split_frontmatter and markdown profile parser

The loader SHALL parse markdown files with YAML frontmatter delimited by --- lines, extracting structured fields from the YAML and the markdown body as the system prompt.

**Acceptance:**
- split_frontmatter(content) -> Result<(&str, &str), ParseError> splits on --- delimiters, returns (yaml, body)
- Self-contained implementation (~30 lines), does not import from claude-runner
- Handles edge cases: no closing ---, empty body, empty frontmatter, CRLF line endings
- parse_profile(content, path) -> Result<Profile, ParseError> calls split_frontmatter, deserialises YAML via serde_yaml into RawNornProfileFrontmatter, maps to Profile
- RawNornProfileFrontmatter supports fields: name, description, model, tools (list or comma-separated string), disallowedTools (alias disallowed_tools), reasoning_effort, reasoning_summary, capabilities
- Markdown body after frontmatter becomes the first entry in Profile.system_instructions
- Profile::from_file with .md extension delegates to parse_profile
- Test: parse a markdown profile with all fields, verify Profile struct populated correctly
- Test: parse a markdown profile with only name and model, verify defaults for optional fields
- Test: split_frontmatter with missing closing --- returns error

**Files:**

- modify: crates/norn/src/profile/types.rs
