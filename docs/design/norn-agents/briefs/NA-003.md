## Purpose

The task system is currently flat (no parent/child), in-memory only, and missing a Failed status. Multi-agent coordination requires hierarchical tasks where parents decompose work into subtasks, agents claim tasks atomically, and parent status reflects children. This brief restructures task.rs (591 lines, at the limit) into a task/ folder module and implements the full hierarchy with status roll-up, establishing the foundation for DiskTaskStore (NA-004) and extension wiring (NA-005).

## Task

Convert tools/task.rs into a tools/task/ folder module with types.rs (TaskStatus, TaskEntry, TaskStore trait), memory.rs (InMemoryTaskStore), rollup.rs (status roll-up logic), and tool.rs (TaskTool). Add Failed as a fifth TaskStatus variant. Add parent_task_id and assigned_agent fields to TaskEntry. Extend TaskStore trait with create_subtask, children, ancestors, claim, create_group, list_groups. Implement status roll-up with priority ordering Failed > Blocked > InProgress > Pending > Completed. Extend TaskTool with new action handlers for all new operations.

### R5: Status roll-up logic

Parent task effective status SHALL be computed from children's statuses with priority ordering: Failed > Blocked > InProgress > Pending > Completed.

**Acceptance:**
- effective_status(children_statuses) -> TaskStatus function in rollup.rs
- Any child Failed -> parent Failed
- Any child Blocked (none Failed) -> parent Blocked
- Any child InProgress (none Failed/Blocked) -> parent InProgress
- Mix of Pending and Completed (no Failed/Blocked/InProgress) -> parent InProgress
- All children Pending -> parent Pending
- All children Completed -> parent Completed
- Empty children list -> returns the parent's own status unchanged
- Test: each priority case verified individually
- Test: mixed states (one Blocked + one InProgress) -> Blocked

**Files:**
- modify: crates/norn/src/tools/task/rollup.rs

**Checklist:**
