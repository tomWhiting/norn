# NA-007: Reactive wait, DFS shutdown, and inbound channel wiring

> **Cluster:** norn-agents
> **Depends on:** NA-006
> **Checklist:**
> - C32 — Child agent drains InboundChannel at tool boundaries using the existing Steer/FollowUp drain logic.
> - C33 — WaitAgentTool uses tokio::sync::watch channel subscription instead of 50ms polling.
> - C34 — WaitAgentTool supports waiting on multiple agents via tokio::select! for first-to-finish.
> - C36 — close_agent performs DFS shutdown of all descendants before transitioning the target.
> **Stories:**
> - S3 (AI Agent, Delegating Work to Sub-Agents) — As an AI agent, I want to wait for a child agent reactively so that I consume zero CPU while idle instead of polling every 50ms.
> - S4 (AI Agent, Delegating Work to Sub-Agents) — As an AI agent, I want to wait on multiple children and react to whichever finishes first so that I can process results as they arrive.
> - S5 (AI Agent, Delegating Work to Sub-Agents) — As an AI agent, I want to send a steering message to a running child so that I can redirect its approach without killing and respawning it.
> - S6 (AI Agent, Delegating Work to Sub-Agents) — As an AI agent, I want to close a parent agent and have all its descendants shut down recursively so that I do not leave orphaned agents running.
> - S10 (AI Agent, Running as a Sub-Agent) — As a sub-agent, I want to receive steering messages from my parent at tool boundaries so that I can adjust my approach without interruption.

## Purpose

WaitAgentTool polls the registry every 50ms, wasting CPU. CloseAgentTool does single-level transitions, leaving grandchildren orphaned. Sub-agents have no way to receive steering messages from their parent. This brief rewrites wait to use watch channel subscriptions (zero CPU idle), adds multi-agent wait via tokio::select!, implements recursive DFS shutdown, and wires InboundChannel so children drain Steer/FollowUp messages at tool boundaries.

## Task

Rewrite WaitAgentTool to subscribe to the AgentHandle's status watch::Receiver instead of polling. Support waiting on multiple agents via tokio::select! for first-to-finish patterns. Rewrite CloseAgentTool to perform DFS shutdown: find all descendants in the registry, shut them down leaf-to-root. Wire InboundChannel draining for sub-agents so Steer and FollowUp messages from the parent are processed at tool boundaries. Rename signal_agent tool's usage to reflect steering patterns via InboundChannel.

## Requirements

### R1: Reactive wait via watch channel

WaitAgentTool SHALL subscribe to the child's status watch channel instead of polling the registry every 50ms.

**Acceptance:**
- WaitAgentTool retrieves AgentHandle from AgentHandles extension using the target agent_id
- Subscribes to handle.status_rx (watch::Receiver<AgentStatus>)
- Awaits status change to Completed or Failed — zero CPU while idle
- Timeout still applies via tokio::time::timeout wrapping the watch
- Immediate-poll mode (timeout: 0) checks current status without subscribing
- Falls back to registry polling if AgentHandle not found (for agents not spawned by current parent)
- WAIT_POLL_MS constant removed
- Test: spawn child, wait_agent returns when child completes (no 50ms polling)

### R2: Multi-agent wait via tokio::select!

WaitAgentTool SHALL support waiting on multiple agents and returning when the first one finishes.

**Acceptance:**
- WaitAgentArgs.agent_id accepts a single string OR the tool gains an additional parameter for multiple agent IDs
- When multiple IDs provided: subscribe to all watch channels, use tokio::select! (or select_all) to await first completion
- Return payload identifies which agent completed first and its status
- Remaining agents continue running — only the first-to-finish is reported
- Test: spawn 2 children, wait on both, fast child completes first, verify fast child's ID returned

### R3: DFS shutdown for close_agent

CloseAgentTool SHALL perform depth-first shutdown of all descendants before transitioning the target agent.

**Acceptance:**
- close_agent finds all descendants of the target in the AgentRegistry (using parent_id chains)
- Shuts down leaf nodes first, working up to the target (DFS post-order)
- Each agent transitions: mark_completing, then mark_completed (or mark_failed if shutdown fails)
- Child JoinHandles are aborted or signalled via InboundChannel with a shutdown steer message
- Return payload lists all agents shut down with their final status
- Test: root -> child -> grandchild, close root, all three end up Completed
- Test: close on a leaf agent with no children works (degenerate case)

### R4: InboundChannel wiring for sub-agents

Sub-agents SHALL drain their InboundChannel at tool boundaries, processing Steer and FollowUp messages from the parent.

**Acceptance:**
- NA-006 creates InboundChannel pair at spawn time — this brief ensures the child's run_agent_step receives the InboundChannel receiver
- Child agent drains InboundChannel using the existing Steer/FollowUp drain logic in the agent loop
- signal_agent tool can push to a child's InboundChannel via the AgentHandle's InboundSender when the target is a child agent
- Steer messages injected after current tool batch, before next provider request
- FollowUp messages injected when the child would otherwise stop
- Test: spawn child, send steer message via signal_agent, verify child receives it
