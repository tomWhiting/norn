## Purpose

SpawnAgentTool is currently synchronous (blocks parent), creates a bare LoopContext with no profile, and passes an empty tool definitions slice to the child. This brief rewrites spawn to be async via tokio::spawn, accept a profile parameter for child configuration, build per-child ToolContext (solving the shared identity problem where children thought they were root), pass filtered tool definitions so the child model can see its tools, and send completion notifications to the parent via mailbox. This is the core brief that makes multi-agent coordination work.

## Task

Create tools/agent/handle.rs with AgentHandle (agent_id, status watch::Receiver, InboundSender, JoinHandle) and AgentHandles collection type. Rewrite SpawnAgentTool::execute to: (1) accept optional profile parameter, resolve via libnorn scanner, build child LoopContext via from_profile, (2) build per-child ToolContext with child-specific AgentToolInfra (correct agent_id, parent_id), (3) extract filtered ToolDefinitions from the parent's ToolRegistry through the allow-list, (4) launch child via tokio::spawn, return immediately with agent_id, (5) tokio::spawn wrapper handles completion: marks registry status, sends mailbox notification with trigger_turn:true, updates status watch channel. Rewrite SubAgentExecutor to own a child_context: Arc<ToolContext> and dispatch tool calls through the full four-phase lifecycle (pre_validate, execute, post_validate, on_success) with the child's context instead of delegating to registry.execute().

## Requirements

### R1: AgentHandle and AgentHandles types

AgentHandle SHALL encapsulate the runtime resources for a spawned child agent, and AgentHandles SHALL be a ToolContext extension storing all handles keyed by agent_id.

**Acceptance:**
- AgentHandle struct in tools/agent/handle.rs: agent_id (Uuid), status_rx (watch::Receiver<AgentStatus>), inbound_tx (InboundSender), join_handle (JoinHandle<()>)
- AgentHandles struct wrapping Arc<Mutex<HashMap<Uuid, AgentHandle>>>
- AgentHandles::insert(handle), AgentHandles::get(agent_id) -> Option<&AgentHandle>, AgentHandles::remove(agent_id)
- AgentHandles is Send + Sync (for ToolContext extension storage)
- Test: insert and retrieve an AgentHandle by UUID

### R2: SubAgentExecutor with per-child ToolContext and full lifecycle

SubAgentExecutor SHALL own a child-specific ToolContext and dispatch tool calls through the full four-phase lifecycle (pre_validate, execute, post_validate, on_success) with the child's context, NOT the parent's shared context.

**Acceptance:**
- SubAgentExecutor gains field: child_context: Arc<ToolContext>
- SubAgentExecutor::new takes child_context as a parameter
- execute() looks up Tool from parent registry via registry.get(name)
- execute() builds ToolEnvelope from arguments
- execute() calls tool.pre_validate(&envelope, &child_context) — blocks on PreValidateOutcome::Block
- execute() calls tool.execute(&envelope, &child_context)
- execute() calls tool.post_validate(&output, &child_context) — collects errors
- execute() calls tool.on_success(&output, &child_context) when output is not error (on_success takes output and context, not envelope)
- Does NOT call self.registry.execute() (which would use the parent's context)
- Test: child calls spawn_agent, infra_from returns child's AgentToolInfra (not root's)
- Test: child tool call runs through pre_validate with child context

**Files:**
- modify: crates/norn/src/tools/agent/infra.rs

### R3: Per-child ToolContext construction

- SharedToolCatalog shared from parent (tool discovery is global)

### R5: Filtered tool definitions for child

The child agent SHALL receive filtered tool definitions from the parent's ToolRegistry through the allow-list, not an empty slice.

**Acceptance:**
- Spawn code iterates parent's ToolRegistry, calls tool.name(), tool.description(), tool.input_schema() for each available tool
- Filters through the allow-list (same one used by SubAgentExecutor for execution gating)
- Builds Vec<ToolDefinition> passed as the tools parameter to run_agent_step
- When no allow-list specified: all parent tools included (minus spawn_agent itself to prevent infinite recursion — optional safety)
- Child model sees tool names and schemas in its context

### R6: Async spawn via tokio::spawn

SpawnAgentTool SHALL launch the child via tokio::spawn and return immediately with the agent_id and registry path.

**Acceptance:**
- Child's run_agent_step runs inside tokio::spawn, not blocking the parent
- SpawnAgentTool::execute returns immediately with JSON: agent_id, path, status: 'active'

- AgentHandle stored in AgentHandles extension on the spawning agent's ToolContext

### R7: Completion notification via tokio::spawn wrapper

When a child reaches terminal status, the spawning infrastructure SHALL send a mailbox notification to the parent with trigger_turn: true.

**Acceptance:**
- tokio::spawn wrapper awaits run_agent_step, then:
- On success: marks child Completing then Completed in AgentRegistry
- On error: marks child Failed in AgentRegistry
- Sends MailboxMessage to parent's inbox: from = child_id, to = parent_id, content = {agent_id, status, result_summary}, trigger_turn = true
- Updates status watch channel with terminal status (wakes WaitAgentTool subscribers)
