# NA-010: Fork tool async redesign

> **Cluster:** norn-agents
> **Depends on:** NA-006 (AgentHandle, per-child ToolContext)
> **Checklist:**
> - C50 — Fork tool forks the current agent onto a different model and returns structured audit result
> - C70 — Fork inherits filtered parent context and returns structured audit result
> - C27 — SpawnAgentTool launches the child via tokio::spawn (template for fork)
> - C30 — AgentHandle stored as ToolContext extension (fork needs same)
> - C31 — InboundChannel created for each child agent (fork needs same)
> - C35 — Child sends completion notification to parent's mailbox with trigger_turn: true
> **Stories:**
> - S2 (AI Agent) — As an AI agent, I want spawn to return immediately so that I can continue working while the child executes in parallel (fork should have the same property).
> - S7 (AI Agent) — As an AI agent, I want to be notified via mailbox when a child completes so that I do not have to poll for completion.

## Purpose

Fork is currently broken in six ways: synchronous (blocks parent), no SessionTree branching, orphaned tool_call causes HTTP 400, uses parent's ToolContext identity, hardcoded system instruction, and no inbound channel. This brief fixes all six issues, converting fork from a synchronous blocking call into an async concurrent agent that inherits parent context and runs on a (potentially different) model. Additionally, fork gains task-structured input and output for reliable, predictable results.

Fork is semantically distinct from spawn: **fork = same identity, different model** (tactical "more hands"); **spawn = fresh identity, configured through profile** (team coordination).

## Task

Convert the fork tool from synchronous execution to async execution using the same `tokio::spawn` pattern as spawn. Fix context construction to eliminate orphaned tool_calls via synthetic result injection. Wire SessionTree branching, per-child ToolContext with hierarchical IDs, inbound channel, and parent system instruction inheritance. Add task-structured input/output for predictable fork results. Ensure the fork child has clear identity reinforcement across system instruction, developer message, and task framing.

## Requirements

### R1: Async fork execution via tokio::spawn

ForkTool::execute() SHALL launch the child via `tokio::spawn` and return immediately with `{ fork_id, status: "active" }`, matching spawn's async pattern.

**Acceptance:**
- ForkTool::execute() creates child context, launches via tokio::spawn, returns immediately
- Return value: `{ fork_id: uuid, status: "active" }`
- Creates `AgentHandle` with status watch channel, inbound sender, join handle, and child EventStore
- AgentHandle stored on parent's `AgentHandles` keyed by fork agent_id
- On completion: `finish_child()` extracts result, sends mailbox notification with `trigger_turn: true`
- On failure: marks registry entry as Failed, sends failure notification
- Spawn pattern used as template: `spawn.rs:351-410` (`launch_child`)
- Test: fork returns immediately, parent continues, completion notification received

**Files:**
- modify: crates/norn/src/tools/agent/fork_tool.rs
- modify: crates/norn/src/agent/fork.rs

**Checklist:** C50, C27, C35

### R2: Synthetic tool result injection for orphan-free context

Fork context construction SHALL include the full current turn and inject a synthetic ToolResult for the fork call itself, eliminating the orphaned tool_call that causes HTTP 400.

**Acceptance:**
- Child context includes ALL parent events including the current turn
- All tool_calls in the current AssistantMessage have matching ToolResults in the child context
- Tool results for tools that completed before fork: taken from parent EventStore (already there)
- Fork's own tool result: synthetic `{ fork_id, status: "active", message: "fork created successfully" }`
- Defensive check: verify every tool_call in the most recent AssistantMessage has a matching ToolResult before building child context; log warning if any is missing
- The ContextFilter (exclude_tool_calls, include_recent_n, include_system) still applies on top
- Child receives its task as a fresh UserMessage after the complete context
- Test: fork in a multi-tool-call turn (e.g., [read, search, fork]) — child context has all three tool results, no 400 error

**Files:**
- modify: crates/norn/src/agent/fork.rs

**Checklist:** C70

### R3: Per-child ToolContext with hierarchical fork IDs

Fork SHALL create a per-child ToolContext with its own AgentToolInfra, matching spawn's `build_child_context()` pattern. Fork IDs SHALL be hierarchical.

**Acceptance:**
- Per-child ToolContext created with fresh AgentToolInfra
- Child AgentToolInfra has: own agent_id, parent_id = parent's agent_id, own AgentHandles (for grandchildren)
- Agent ID format: hierarchical registry path `{parent_path}/fork/{uuid}`
- Shared infrastructure forwarded from parent: TaskStore, ToolCatalog, DiagnosticCollector, SessionTree
- Tools dispatched through SubAgentExecutor with child's ToolContext (not parent's)
- Test: tool inside fork sees child agent_id, not parent's

**Files:**
- modify: crates/norn/src/tools/agent/fork_tool.rs

**Checklist:** C30

### R4: SessionTree branching with ForkComplete event

ForkTool SHALL read SharedSessionTree from ToolContext and pass it to fork(). A ForkComplete event SHALL be appended to the parent's timeline when the fork finishes.

**Acceptance:**
- ForkTool reads SharedSessionTree from ToolContext extension (same as spawn)
- Passes (tree, parent_session_id) to fork() instead of None
- Fork event recorded on parent's timeline at branch point
- On fork completion: ForkComplete SessionEvent appended to parent timeline
- ForkComplete carries: child session_id, result_summary, usage, duration_ms
- ForkComplete is a completion reference (pointer), not a content merge
- Test: fork with SessionTree creates a branch visible in tree.list_children()

### R5: Parent system instruction inheritance with fork identity reinforcement

Fork child SHALL inherit the parent's base system instruction and receive multi-layered identity reinforcement.

**Acceptance:**
- Fork inherits parent's system_sections[0] (base system instruction) from LoopContext
- Fork preamble prepended (not replacing): "You are a forked agent executing a bounded task. Complete the requirements and return your structured output. Your results will be returned to the parent agent's timeline."
- Developer message (Responses API): reinforces fork identity and provides the structured output schema
- Task framing (UserMessage): presents the request and task requirements clearly
- Three layers of reinforcement: system instruction, developer message, task UserMessage
- Test: fork child's LoopContext contains parent system instruction + fork preamble

### R7: Task-structured input and output

Fork tool input SHALL accept an optional tasks array. When provided, the fork's structured output schema SHALL mirror the tasks for reliable, predictable results.

**Acceptance:**
- ForkArgs gains optional `tasks: Option<Vec<ForkTask>>` field
- ForkTask has: name (string), description (string)
- When tasks provided: structured output schema dynamically constructed with `response` (string) + `tasks` array (each: name, completed bool, notes string)
- When tasks not provided: default schema `{ response: string }` — free text only
- Schema installed on child LoopContext via the existing structured output mechanism (D3)
- Completion notification carries the validated structured output
- ForkComplete event carries the structured output as result_summary
- Fork tool schema updated with tasks parameter and documentation
- Test: fork with tasks array returns structured output matching the schema

### R8: Tool allow-list verification through async path

- Child model sees only the allowed tools in its tool definitions
- When tools is None: child gets full parent registry (existing behaviour)
- Test: fork with tools: ["bash", "read"] — child can only call bash and read
