# NA-008: Sub-agent observability

> **Cluster:** norn-agents
> **Depends on:** NA-006
> **Checklist:**
> - C37 — Sub-agents can be configured with EventType::Progress schema via EventSchemaSet for structured status updates.
> - C38 — tool_use_description on sub-agent tool calls is accessible to the parent for progress visibility.
> - C39 — Sub-agent's EventStore is linked to the parent's session for audit trail purposes.
> **Stories:**
> - S11 (AI Agent, Running as a Sub-Agent) — As a sub-agent, I want to emit structured progress updates via per-event schemas so that my parent can observe my work without me explicitly reporting.
> - S25 (Human Developer, Observing Agent Activity) — As a human developer, I want sub-agent progress visible through tool_use_descriptions so that I can see what a child agent is doing without structured reporting overhead.

## Purpose

Parents have no visibility into what sub-agents are doing mid-execution. This brief configures sub-agents with EventType::Progress schemas for structured status updates, makes tool_use_description on child tool calls accessible to the parent, and links child EventStores to the parent session for audit trail purposes.

## Task

When spawning a sub-agent with a profile, configure the child's EventSchemaSet with an EventType::Progress schema if the profile specifies one. Ensure tool_use_description on child tool calls flows into the child's EventStore where the parent can read it. Link the child's EventStore to the parent's session tree so all events are part of a single audit trail.

## Requirements

### R1: Progress schema configuration for sub-agents

Sub-agents SHALL be configurable with an EventType::Progress schema via the existing EventSchemaSet mechanism.

**Acceptance:**
- SpawnAgentTool checks profile for a progress_schema in settings (or a dedicated profile field)
- If present, builds an EventSchemaSet with EventType::Progress mapped to the schema
- EventSchemaSet installed on child's LoopContext.event_schemas
- Child model can emit structured progress updates validated against the schema
- When no progress schema configured: child runs without event schema validation (existing behaviour)
- Test: spawn with progress schema, child emits valid progress event, validate passes

**Files:**
- modify: crates/norn/src/tools/agent/spawn.rs

**Checklist:**
- C37 — Sub-agents can be configured with EventType::Progress schema via EventSchemaSet for structured status updates.

**Stories:**
- S11 (AI Agent, Running as a Sub-Agent) — As a sub-agent, I want to emit structured progress updates via per-event schemas so that my parent can observe my work without me explicitly reporting.

### R2: tool_use_description visibility

tool_use_description on sub-agent tool calls SHALL be accessible to the parent for progress visibility.

**Acceptance:**
- Child's ToolEnvelope metadata includes tool_use_description when present
- tool_use_description is recorded in the child's EventStore as part of the tool call event
- Parent can read child's EventStore to see tool_use_descriptions (EventStore is accessible via the child's agent_id in the registry or via the AgentHandle)
- Test: child makes a tool call with description metadata, parent reads it from child EventStore

**Files:**
- modify: crates/norn/src/tools/agent/spawn.rs

**Checklist:**
- C38 — tool_use_description on sub-agent tool calls is accessible to the parent for progress visibility.

**Stories:**
- S25 (Human Developer, Observing Agent Activity) — As a human developer, I want sub-agent progress visible through tool_use_descriptions so that I can see what a child agent is doing without structured reporting overhead.

### R3: EventStore linking for audit trail

The child's EventStore SHALL be linked to the parent's session for audit trail purposes.

**Acceptance:**
- Child EventStore created as a named branch in the parent's SessionTree (if SessionTree is in use)
- Branch metadata includes: child agent_id, parent agent_id, profile name (if any), spawn timestamp
- If SessionTree is not in use (standalone mode): child EventStore is standalone but stored on the AgentHandle for parent access
- Parent can enumerate child EventStores via AgentHandle
- Test: spawn child, child produces events, parent can read child events through the handle

**Files:**
- modify: crates/norn/src/tools/agent/spawn.rs
- modify: crates/norn/src/tools/agent/handle.rs

**Checklist:**
- C39 — Sub-agent's EventStore is linked to the parent's session for audit trail purposes.

## Boundaries

- SHALL NOT modify the EventSchemaSet validation logic — use it as-is
- SHALL NOT implement a Progress-specific tool (progress events flow through the existing event mechanism)
- SHALL NOT add real-time streaming of child events to the parent — parent reads on demand
- SHALL NOT add todo!() or unimplemented!() macros anywhere
