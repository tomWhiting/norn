# M-002: Move storage traits and models

> **Cluster:** messaging
> **Depends on:** M-001
> **Checklist:**
> - C4 — StorageError defined in libmessage::storage::error
> - C5 — MessagingStorage trait defined in libmessage::storage::traits
> - C6 — All model types moved to libmessage::storage::models
> - C7 — Notification bug fix: read-status update clears notification entry
> **Stories:**
> - S7 (Human Developer, Debugging) — As a developer, I want the storage trait surface to live in the messaging domain crate so that I can understand the contract without navigating the storage crate.

## Purpose

Establish the storage contract in libmessage so that PG and Redis implementations can target a stable trait surface owned by the messaging domain, not the storage crate.

## Task

Move the StorageError enum, the MessagingStorage trait, and all model types (inbox items, notification records, read status) from meridian-services into libmessage/src/storage/. Update all downstream crates to import from the new location. Fix the notification bug where read-status updates don't clear the notification entry (C7 — one-line fix in the file being moved).

## Requirements

### R1: Move StorageError to libmessage

WHEN any libmessage operation encounters a storage failure THE SYSTEM SHALL return a StorageError from libmessage::storage::error AND it SHALL NOT reference meridian_storage error types directly.

**Acceptance:**
- StorageError defined in libmessage::storage::error
- All downstream crates compile against the new path

**Files:**
- create: crates/libmessage/src/storage/error.rs
- modify: crates/meridian-storage-pg/src/messaging.rs

**Checklist:**
- C4 — StorageError defined in libmessage::storage::error

**Stories:**
- S7 (Human Developer, Debugging) — As a developer, I want the storage trait surface to live in the messaging domain crate so that I can understand the contract without navigating the storage crate.

### R2: Move MessagingStorage trait

WHEN a service needs to persist or retrieve messaging data THE SYSTEM SHALL use the MessagingStorage trait from libmessage::storage::traits AND it SHALL NOT import storage traits from meridian-services.

**Acceptance:**
- MessagingStorage trait defined in libmessage::storage::traits
- PG implementation compiles against the new trait location

**Files:**
- create: crates/libmessage/src/storage/traits.rs
- modify: crates/meridian-storage-pg/src/messaging.rs
- modify: crates/meridian-services/src/messaging/mod.rs

**Checklist:**
- C5 — MessagingStorage trait defined in libmessage::storage::traits

### R3: Move model types

WHEN any crate needs messaging model types (inbox items, notification records, read status) THE SYSTEM SHALL import them from libmessage::storage::models AND it SHALL NOT import them from meridian-services::messaging.

**Acceptance:**
- All model types defined in libmessage::storage::models
- No downstream crate imports model types from the old location

**Files:**
- create: crates/libmessage/src/storage/models.rs
- modify: crates/meridian-services/src/messaging/mod.rs
- modify: crates/meridian-storage-pg/src/messaging.rs
- modify: crates/meridian-storage-redis/src/messaging.rs

**Checklist:**
- C6 — All model types moved to libmessage::storage::models

### R4: Fix notification read-status bug

WHEN a message's read status is updated THE SYSTEM SHALL also delete the corresponding notification entry AND it SHALL NOT leave stale notifications after read-status changes.

**Acceptance:**
- read_status update clears the matching notification row
- No stale notifications remain after marking a message as read

**Files:**
- modify: crates/libmessage/src/storage/traits.rs

**Checklist:**
- C7 — Notification bug fix: read-status update clears notification entry

## Boundaries

- SHALL NOT change the StorageError variant set
- SHALL NOT modify the PG migration files
